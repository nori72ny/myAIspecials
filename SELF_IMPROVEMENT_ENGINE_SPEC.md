# AI Workspace 「自律自己進化・最適化エンジン（Self Improvement Engine）」機能設計仕様書

本書は、AI Workspace全体の稼働状況、エージェント利用率、ワークフロー実行頻度、ナレッジの参照頻度、およびプロジェクト全体のボトルネックをシステム自身が常時観測・自律分析し、「重複機能の検出」「不要アセットの整理」「高コストプロセスの統合・最適化」および「具体的なシステム改善プラン（自動リファクタリング、プロンプト調整、エージェント再アサイン）」をオーナーに提案・自律実行する**「Self Improvement Engine（自己改善エンジン）」**のシステム構成、データスキーマ、分析・改善ロジック、および画面設計を定義した製品設計仕様書です。

本システムは、人間から指示されるのを待つ受動的なツールではなく、AI Workspace全体が「自己の無駄や非効率を最も嫌う」という自己最適化本能を備えた**「AI Workspaceの最高技術責任者（Autonomous CTO Engine）」**の役割を担います。

---

## 1. システム概要 ＆ 自己改善コンセプト (Overview & Concept)

Self Improvement Engineは、Workspace内の全サブシステムから利用統計テレメトリを常時取得し、メタレベル（システム自身を俯瞰する視点）から無駄、非効率、重複、陳腐化している領域を自動検出します。

```
       ===================================================================
                       [ システム全域からのテレメトリ収集 ]
       ===================================================================
       ├─ [Agents]    ➔ エージェント毎のAPIコスト、成功率、最終アクティブ日時
       ├─ [Workflows] ➔ ワークフローの実行頻度、総実行時間、エラー発生率
       ├─ [Projects]  ➔ タスク進行の停滞期間、手戻り回数、アサインの競合状態
       ├─ [Knowledge] ➔ ナレッジの最終更新日、参照頻度、孤立した不要ファイル
       └─ [LLM Usage] ➔ トークン消費比率、モデル毎のコストパフォーマンス効率
                                       │
                                       ▼
       ===================================================================
                       [ Self Improvement Engine Core ]
       ===================================================================
       ├─ ① 自律システムアナライザー (Analysis)
       │    - リソース利用状況の把握、エージェント/ナレッジ/プロンプトの死活監視
       ├─ ② クリティカル最適化検出器 (Optimization)
       │    - 重複エージェント、不要ワークフロー、高コストLLM、陳腐化ナレッジの検出
       ├─ ③ インテリジェント改善提案エンジン (Recommendation)
       │    - 改善策の自動立案、優先度判定、APIコスト削減効果 ＆ 影響リスク予測
       └─ ④ 自己改善ダッシュボード (Self-Optimizing Cockpit)
                                       │
                     ┌─────────────────┴─────────────────┐
                     ▼ (承認後の自律アクション)               ▼ (人間によるアライメント)
       【🤖 AI Workspaceの自動改善】                【👥 人間（オーナー/管理者）】
        - 重複プロンプトの自律統合・削除              - コスト削減プランの最終承認
        - 定型ワークフローの実行間隔・条件の最適化    - 提案されたリプラン・アサインの実行許可
        - 安価なLLM（Gemini Flash等）への自動切替   - ナレッジベースの自律アーカイブ整理
```

### テーブル定義詳細

#### ① `self_improvement_summary` (自己改善サマリーマスタ)
システム全体の最新の最適化ステータスを高速描画するための集計キャッシュマスタ。
*   `id`: UUID (Primary Key)
*   `company_id`: UUID (Foreign Key - 組織スペースのID)
*   `system_efficiency_score`: DECIMAL(5, 2) (システム総合効率スコア。0.00〜100.00)
*   `total_stale_assets_count`: INT (現在検出されている不要・重複アセットの総数)
*   `projected_savings_usd`: DECIMAL(10, 4) (提案をすべて適用した場合に期待される月間APIコスト削減総額)
*   `last_fully_analyzed_at`: TIMESTAMP (システム全体をディープスキャンした最終日時)
*   `created_at` / `updated_at`: TIMESTAMP

#### ② `resource_metrics` (アセット稼働・コストメトリクス)
エージェント、ワークフロー、ナレッジの利用頻度や、個別にかかっているAPIコストの追跡履歴。
*   `id`: UUID (Primary Key)
*   `summary_id`: UUID (Foreign Key)
*   `asset_type`: VARCHAR(50) (アセット種別。`AGENT`, `WORKFLOW`, `PROMPT`, `KNOWLEDGE`)
*   `asset_id`: UUID (各アセットテーブルのPrimary KeyへのポリモーフィックなID参照)
*   `call_count`: INT (累計利用/起動回数)
*   `cost_usd`: DECIMAL(12, 8) (発生した累計APIコスト)
*   `success_rate`: DECIMAL(5, 2) (タスクの成功率。100%満点)
*   `last_used_at`: TIMESTAMP (最終利用日時。死活判定に使用)
*   `updated_at`: TIMESTAMP

#### ③ `detected_anomalies` (検出されたシステム異常・非効率)
重複や未使用アセット、高コストプロセスなど、改善が必要な「非効率ノード」の検出記録。
*   `id`: UUID (Primary Key)
*   `summary_id`: UUID (Foreign Key)
*   `anomaly_type`: VARCHAR(100) (異常の分類。`DUPLICATE` (重複機能), `STALE` (長期未使用), `OVERCOST` (想定外の急激なコスト高騰), `ERROR_PRONE` (エラー多発・処理停滞))
*   `source_asset_id` / `target_asset_id`: UUID (比較元、および重複/競合先のアセットID。死活アセットの場合は片方のみ)
*   `similarity_score`: DECIMAL(4, 3) (セマンティック類似度。0.000〜1.000)
*   `detected_at`: TIMESTAMP

#### ④ `improvement_proposals` (改善・最適化提案)
システム異常を解消するための、AIが算出した具体的なアクションプラン。
*   `id`: UUID (Primary Key)
*   `summary_id`: UUID (Foreign Key)
*   `title`: VARCHAR(255) (提案タイトル。例: `MEOエージェントをV2へマージ ＆ 冗長コスト削減`)
*   `description`: TEXT (提案の理由、具体的な実行手順、およびコード/プロンプトの修正案)
*   `priority`: VARCHAR(50) (優先順位。`URGENT` (最優先・即時実行推奨), `MEDIUM`, `LOW`)
*   `impact_score`: DECIMAL(5, 2) (削減効果期待値。例: 月々のAPIコスト削減、処理スピード向上率)
*   `risk_score`: DECIMAL(5, 2) (この改善を適用する際のリスク・影響度。100点中、数値が低いほど安全。依存関係が多いほど高くなる)
*   `status`: VARCHAR(50) (ステータス。`PENDING` (承認待ち), `APPLIED` (実行・適用完了), `IGNORED` (人間による却下・無視))
*   `created_at`: TIMESTAMP

## 4. 分析・改善・最適化ロジック (Analysis & Optimization Engineering)

自己改善エンジンが、システム全体から「無駄」を数学的かつ構造的に見つけ出し、安全に自己治癒（リファクタリング）するためのコアアルゴリズム。

```
              [ 全システム利用テレメトリのスキャン ]
                                │
         ┌──────────────────────┼──────────────────────┐
         ▼ (分析 1)             ▼ (分析 2)             ▼ (分析 3)
  [ セマンティック類似度 ]      [ ライフサイクル死活 ]     [ コスト ✕ エラー効率 ]
  - プロンプト/コードの       - 30日間未使用アセット     - 成功率が低く、
    Embedding ベクトル比較      の自動スキャン             高価なLLMの使用を特定
         │                             │                       │
         └──────────────────────┬──────┴───────────────────────┘
                                ▼
                   [ システム重複・非効率の特定 ]
                                │
                                ▼
                 [ 改善アクションプランの自動生成 ]
      ├─ 1. 依存関係グラフ（Knowledge Graph）の安全スキャン
      ├─ 2. 変更によるリスクスコア（Risk Score）の計算
      └─ 3. プラン（プロンプト自動マージ、LLM自動切替等）の提示
```

### ① 重複・類似アセットの検出 (Semantic Similarity Audit)
*   **プロンプト重複の検出**:
    Workspace内の全AIエージェントのシステムプロンプト、および登録されたプロンプトテンプレートを `text-embedding-004` (Gemini API) でベクトル化。
    全ペア間のコサイン類似度 $S$ を測定。
    $$S = \frac{\mathbf{A} \cdot \mathbf{B}}{\|\mathbf{A}\| \|\mathbf{B}\|}$$
    類似度 $S \ge 0.90$ のアセットを「重複候補（DUPLICATE）」として自動マーキング。
*   **不要アセット（Stale Assets）の検出**:
    `resource_metrics` から最終利用日時が「30日以上前」のエージェント、ワークフロー、ナレッジファイルを自動的に「不要アセット（STALE）」としてマーキング。

### ② 改善アクションの安全度（Risk Assessment）の算定
システムを自動改善する際、他の稼働中タスクを破壊しないための「影響・依存関係分析（Critical Path Safety Check）」。
*   **計算ロジック**:
    改善対象アセット $A$ のナレッジグラフ（Knowledge Graph）上のイン・アウト度数をスキャン。
    $$Risk = w_1 \times N_{dependent\_tasks} + w_2 \times N_{parent\_workflows}$$
    *   **リスク低 (Risk < 30)**: 他のタスクやワークフローに依存していない、または極めて限定的。AIによる**全自動クリーンアップ・アーカイブ**を許可。
    *   **リスク高 (Risk $\ge$ 30)**: 稼働中の重要プロジェクトや自動ワークフローがこのアセットに強く依存。AIは勝手に削除せず、「Mission Control Center」へアサイン書き換え案を含めた「承認要請」を発行。

### ③ コスト最適化（Cost-Performance Routing）
*   **自動切り替えルーティング**:
    過去のタスク完了ログから、「エラーがなく、比較的シンプルなデータ抽出タスク」であるにもかかわらず、高価な `anthropic/claude-3.5-sonnet` が指定されている箇所を特定。
    「このワークフローのステップ3は、Gemini 2.5 Flash に切り替えることで、**品質を落とさずにコストを 95% 削減**できます」というルーティング最適化ポリシーを自動生成します。

## 5. 他システム（OS、ツイン、ブレイン、ハブ、グラフ、目標、PM、ストア）との統合

Self Improvement Engineは、AI Workspace全体の「最高技術責任者（Autonomous CTO）」として、全システムの骨格と神経を美しくリファクタリングし、整合性を保つ統治（ガバナンス）を行います。

1.  **AI Company OS / Personal AI OS 連携 (インフラの自己修復)**
    *   OSの稼働メモリ、API接続、トークンリミットを最適化。
    *   サーバー全体の負荷を最小化し、クリーンで高速なWorkspaceインフラを維持。
2.  **ナレッジグラフ・エンジン（Knowledge Graph Engine）連携 (陳腐化ナレッジのアーカイブ)**
    *   長期間参照されておらず、かつ最新のナレッジと矛盾（セマンティックな不整合）が生じている古いナレッジをグラフ上で特定。
    *   「古い規約（2024年度）は、最新の規約（2026年度）と衝突しています。古いファイルをアーカイブに退避させ、グラフの結線を最新に統一しました」とグラフを自律修復（Knowledge Refactoring）。
3.  **Digital Twin Core 連携 (オーナーの美学の共有)**
    *   「何を以て『システムが美しく整理されている』とするか」という美観基準や、予算管理のスタンスをツインコアの「価値観（VALUES）」から同期。
    *   オーナーが「多少の重複があっても、過去の歴史ログは残したい」という価値観を持っている場合、削除ではなく「非表示・アーカイブ」を優先するよう最適化の行動バイアスを自動調整。
4.  **AI Project Manager / Goal Engine 連携 (プロジェクト最適化)**
    *   進行中のプロジェクトのタスク内で、手戻り（Rejections）が頻発しているタスクを検知。
    *   「NextJSコーダーのコンパイルエラーが多発しています。タスクの要件定義プロンプトを、より構造化された形式へ自動リファクタリングしてタスクに再適用します」と、プロジェクト進行のクリティカルパスを技術的に支援。
