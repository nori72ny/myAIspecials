# AI Workspace 「AIプロジェクトマネージャー（AI Project Manager）」機能設計仕様書

本書は、AI Workspaceにおけるあらゆるプロジェクト（新規事業立ち上げ、開発、マーケティング、デザイン、資料作成等）のタスク分解、エージェントへの自動配備、ガントチャートやカンバンの自動更新、成果物管理、進捗予測、リスク検知、および自律改善提案を一手に担うAI駆動型プロジェクト管理システム**「AI Project Manager（AIプロジェクトマネージャー）」**のシステム構成、データスキーマ、自律管理設計、および画面レイアウトを定義した最上位の製品設計仕様書です。

プロジェクトの目的（ゴール）や大まかな要望を人間が入力した瞬間、AIが「タスクの自動分解」「優先順位と依存関係（クリティカルパス）の定義」「最適な担当エージェント/チーム/ワークフローへの全自動配備（アサイン）」を行い、日々の自律業務実績から「遅延リスク」や「バグによる停滞」をリアルタイムで検知・解決。常に人間のプロジェクトマネージャー以上の精度でプロジェクトをゴールまで安全に自律運行させる**「AI統括ディレクター（Autonomous PM Engine）」**の役割を担います。

## 1. 業務フロー ＆ AIプロジェクト自律管理コンセプト (Overview & Concept)

AI Project Managerは、人間が設定したプロジェクトの要件や目標から「逆算」して、詳細なタスクリスト（WBS: Work Breakdown Structure）を自動構築します。さらに、依存関係を持つ各タスクを専門のAIエージェント、チーム、または自動ワークフローへと自動的にアサイン。実務の進捗テレメトリ（コンパイル結果、API送信、成果物の品質評価）を監視し、計画の自律修正（リプラン）を実行します。

```
                         [ 1. プロジェクト創出 / 定義 ]
        (目的、期限、予算、期待成果物の入力、またはプリセットテンプレート適用)
                                       │
                                       ▼
                     [ 2. AI自律タスク分解 ＆ パス定義 ]
      ├─ LLMによる「詳細タスク（WBS）の自動分解」
      ├─ タスク間の依存関係（例: Aが完了しないとBは開始不可）の定義
      └─ クリティカルパス（最も遅延リスクの高い主要経路）の動的算出
                                       │
                                       ▼
                   [ 3. 担当アサイン・自動配備 (Resource Map) ]
      ├─ 各タスクに最適なエージェント、チーム（AI Team Builder）、ワークフローを割り当て
      ├─ 必要ツールの配線確認（Tool Connector Hub連携）
      └─ ナレッジグラフ（Knowledge Graph）から初期要件ドキュメントを同期アタッチ
                                       │
                                       ▼
                     [ 4. 実行 ＆ テレメトリ監視 (Tracking) ]
      ├─ 各タスクの実行ログ、成果物（資料・アプリ・サイト）の出力状況の監視
      ├─ ガントチャート ＆ カンバンボードの自動更新
      └─ 成果物管理（Deliverables Vault）への自動整理とインデックス登録
                                       │
                                       ▼
                  [ 5. 進捗予測 ＆ リスク自律検知 (Forecasting) ]
      ├─ 実績進捗ベースでの「完了予測日（Estimated Completion Date）」の自動算出
      ├─ リスク検知（バグによる停滞、APIエラー、ボトルネック、デッドライン逼迫）
      └─ 改善提案（Recommendations）とプラン自動アップデート（リプラン）
                                       │ (フィードバックループ)
                                       └──────────────────────┘
```

### テーブル定義詳細

#### ① `projects` (プロジェクト定義マスタ)
*   `id`: UUID (Primary Key)
*   `company_id`: UUID (Foreign Key - 組織スペースのID)
*   `name`: VARCHAR(255) (プロジェクト表示名。例: `新宿個室カフェ La Lounge オープン ＆ 初期集客`)
*   `description`: TEXT (プロジェクトの目的、期待される最終成果、成功基準の解説)
*   `status`: VARCHAR(50) (状況。`INIT` (起案・タスク生成中), `ACTIVE` (進行中), `PAUSED` (一時停止), `COMPLETED` (完了))
*   `start_date` / `end_date`: TIMESTAMP (目標進行期間)
*   `current_progress_rate`: DECIMAL(5, 2) (完了タスクのウェイト比から自動計算される進捗率。0.00〜100.00 %)
*   `template_origin_id`: UUID (Foreign Key, Nullable - 複製元となったテンプレートプロジェクトのID。テンプレート機能で使用)
*   `created_at` / `updated_at`: TIMESTAMP

#### ② `tasks` (タスク定義 ＆ AIアサインテーブル)
プロジェクトを構成する詳細な作業タスクと、アサインされる自律リソース（エージェント/チーム/ワークフロー）のマッピング。
*   `id`: UUID (Primary Key)
*   `project_id`: UUID (Foreign Key)
*   `title`: VARCHAR(255) (タスク名。例: `Next.js SPA LPのコーディングとNetlifyデプロイ`)
*   `description`: TEXT (タスクの詳細仕様、AIに対する実行プロンプト・ゴール要件)
*   `status`: VARCHAR(50) (ステータス。`BACKLOG`, `TODO`, `IN_PROGRESS`, `REVIEW`, `DONE`)
*   `priority`: VARCHAR(50) (優先順位。`LOW`, `MEDIUM`, `HIGH`)
*   `assigned_type`: VARCHAR(50) (割り当て先の種類。`AGENT` (個別AIエージェント), `TEAM` (AIチーム), `WORKFLOW` (自動化フロー))
*   `assigned_id`: UUID (Foreign Key - 割り当てられた `marketplace_agents` または `ai_teams`, `workflow_steps` のプライマリキー)
*   `start_date` / `due_date`: TIMESTAMP (タスク期日)
*   `progress`: DECIMAL(5, 2) (タスク個別の進捗率。0.00〜100.00 %)
*   `created_at`: TIMESTAMP

#### ③ `task_dependencies` (タスク間依存関係定義テーブル)
WBSにおけるタスクの前後関係（クリティカルパス算出のための配線情報）。
*   `id`: UUID (Primary Key)
*   `task_id`: UUID (Foreign Key - 後続タスク)
*   `depends_on_task_id`: UUID (Foreign Key - 先行タスク)
*   `dependency_type`: VARCHAR(50) (依存関係の定義。`FS` (Finish-to-Start: 先行タスクが完了したら開始可能), `SS` (Start-to-Start: 先行タスクが開始されたら並行開始可能))
*   `created_at`: TIMESTAMP

#### ④ `deliverables` (プロジェクト成果物登録マスタ)
プロジェクト活動を通じて生み出されたすべてのデータ、資料、ソースコード、公開URL。
*   `id`: UUID (Primary Key)
*   `project_id`: UUID (Foreign Key)
*   `task_id`: UUID (Foreign Key, Nullable - どのタスクから出力された成果物か)
*   `name`: VARCHAR(255) (成果物のファイル名、または表示名。例: `index.tsx`, `競合分析レポート_v2.md`)
*   `type`: VARCHAR(50) (種別。`CODE`, `DOCUMENT`, `IMAGE`, `PRESENTATION`, `WEBSITE_URL`, `DATA_SHEET`)
*   `file_path`: VARCHAR(1024) (格納先のパス。Knowledge Hub、プロジェクト共有Files、または外部URL)
*   `metadata_json`: JSON (成果物特有のメタデータ。例: `{"file_size_bytes": 10450, "test_coverage_percent": 98.2, "deployment_url": "https://..."}`)
*   `created_at`: TIMESTAMP

## 4. プロジェクト管理・自動化設計 (Project Management & Execution Design)

AIがプロジェクト管理を「自律運行」するための主要エンジンの設計。

### ① WBS自動分解 ＆ 依存関係定義アルゴリズム (Auto WBS Engine)
*   **作動プロセス**:
    1.  人間が「新宿カフェの集客用LPを制作して、Netlifyで公開、口コミ返信フローまで構築して」と目的を入力。
    2.  AI PMは目的と「ターゲット完了日（end_date）」、および利用可能なエージェントとツールのポートフォリオ（Tool Hub, Marketplace）を取得。
    3.  大規模言語モデル（LLM）を用いて、目的を達成するための「WBSタスクツリー」を自動生成（タスク分解）。
    4.  各タスクに対し、「先行タスクがなければ着手できない」論理的関係（例：「デザインが完了しないとコーディングは不可能」＝ `FS` 依存）を自動的にマッピング（依存関係の定義）。
    5.  マッピングした依存タスク網をトポロジカルソートし、最も遅延影響度の大きい**「クリティカルパス（Critical Path）」**を自動特定してダッシュボードにハイライトします。

### ② 進捗・遅延・完了予測ロジック (Predictive Forecasting Engine)
*   **計算モデル**:
    毎日、各タスクの `progress` の変位量（前日差分）と、アサインされたAIエージェントの処理速度（過去のタスク完了時間の中央値）をトラッキング。
    各クリティカルパス上の未完了タスク $T_i$ に対する「残工期（日数）」をシミュレーションします。
    $$\text{Remaining Days for } T_i = \frac{100 - \text{Current Progress } P_i}{\text{日次平均進捗率 } \Delta P}$$
    この累積残工期から、プロジェクト全体の**「自律完了予測日（Estimated Completion Date）」**を算出。
    目標完了日（`end_date`）を予測日が超過（$T_{forecast} > T_{due}$）した場合、即時システムが「遅延警告（イエロー/レッドアラート）」を発行します。

### ③ リスク検知 ＆ 改善提案（Self-Healing PM Pipeline）
*   **リスク検知パターン**:
    *   **進捗膠着リスク**: タスクが `IN_PROGRESS` のまま3日以上進捗率が変動していない（AIがコンパイルエラーや例外処理のループでハマっている状態を自動検知）。
    *   **リソース不足リスク**: 依存関係のあるタスク同士が同時並行で動き始めた際、割り当てエージェントが重複しており、APIレートリミット超過や競合が発生している。
    *   **デッドライン逼迫リスク**: 予測値と実績値の解離が広がっている。
*   **改善提案（リプラン）プロセス**:
    リスクを検知したAI PMは、直ちに代替の「リプラン計画」を生成。
    *   *   **対策 A**: 膠着しているエージェントへ「エラーログを詳細分析して自動修正を指示する」というデバッグ用サポートエージェント（Tester）を追加アサイン。
    *   *   **対策 B**: 難解なバグが発生しているタスクの割り当てLLMを、安価なモデルから最上位モデル（Claude 3.5 Sonnet）へ動的にアップグレードし、一瞬でコンパイルを通す。
    *   *   **対策 C**: 手動タスクの介入要請。人間に「ここの仕様があいまいなため、AIが迷っています。テキストで仕様を追加してください」というメッセージとともに承認トレイへプッシュ。

## 5. 他システム（OS、ツイン、ブレイン、ハブ、グラフ、マーケット、チーム、目標）との統合

AI Project Managerは、AI Workspace全体の「実務のオペレーション司令塔（COO）」として、全機能がこのプロジェクトを推進するためにシンクロ連動します。

1.  **AI Company OS / Personal AI OS 連携**
    *   AI Company OSの組織図内の「各部署の進捗、プロジェクト稼働率」と完全同期。
    *   経営層（人間およびデジタルツイン）に対し、組織全体のプロジェクトヘルス、遅延状況、成果物の出力をリアルタイムにレポーティングします。
2.  **ナレッジグラフ・エンジン（Knowledge Graph Engine）連携 (知財の紐付け)**
    *   プロジェクトの作成、タスクの分解、成果物の創出、リスクの発生は、すべてナレッジグラフ上に動的にノードおよび関係（エッジ）として描画。
    *   「LP用の画像成果物」は、グラフ上で「新宿LPプロジェクト ➔ PRODUCES ➔ 画像アセット ➔ GENERATED_BY ➔ Designer_Agent」として有機的に結線。
    *   次の類似プロジェクトを複製（コピー）する際、AIはこのグラフ構造をロードして「過去の成功した進捗パス、最適なエージェントの組み合わせ、使用した資料」を100%踏襲してWBSをプランニングできます。
3.  **AI Team Builder 連携 (チームの動的アサイン)**
    *   タスク分解によって「MEO集客」と「SEO記事制作」が同時に必要になった場合。
    *   AI PMは、Team Builderから事前に定義された「Marketing Team Alpha（3エージェント混成）」をチームごと該当タスクに配備し、チーム間対話と投票レビューをプロジェクト管理ガバナンス内にカプセル化して稼働させます。
4.  **Goal Management Engine 連携 (目標とタスクの整合)**
    *   Goal Engineに掲げられた「売上目標」「利益目標」「集客目標」を達成するための、具体的な「実行戦術」がこのAI Project Manager内にプロジェクトとして配置されます。
    *   プロジェクトが完了（成果物がデプロイ）すると、Tool Connector Hub経由で売上や順位データがGoal Engineに自動フィードバックされ、目標の達成率がリアルタイムで更新されます。

## 6. 将来拡張設計 (Future Extensibility Plan)

### ① Cross-Workspace Project Orchestration (企業間AI共同プロジェクト自律推進)
*   **提携企業、下請け会社、または取引先のAI Workspace（複数のOS環境）のプロジェクト進捗を安全に接続し、自律的なJV（ジョイントベンチャー）を推進する仕組み**:
    *   自社が「新店舗オープン」というプロジェクトを進行させ、店舗内装の設計・施工を外部提携会社のAI Workspaceが請け負っている場合。
    *   お互いのプロエジェクトデータベース（`tasks`, `projects`）を、P2Pのセキュアな暗号化プロトコルを介してマッピング。
    *   お互いの社内コストや機密コードは厳重に保護（ゼロ知識証明）しつつ、「クリティカルパス上の依存関係（例: 提携会社の店舗設計書が完了 ➔ 自社LPに間取り図を掲載するタスクが開始可能）」のみを同期。
    *   提携会社の設計書の進捗が遅れた際、自社のAI PMが遅延を自動的に検知してLP公開日を調整したり、相手のAI PMに「進捗の状況を教えてください」と自動対話（AI-to-AI PM Negotiation）を行う、企業間分散プロジェクト管理ガバナンスへと拡張します。

### ② Project Simulation & Swarm Dry-Run (超高速仮想プロジェクトドライラン)
*   **実際のプロジェクトを開始、またはアサインする前に、バーチャルサンドボックス内でAIエージェントチームに超高速（数万倍速）でプロジェクトを一回転プレイ（ドライラン）させ、最適なWBSとリスクを予知する仕組み**:
    *   プロジェクトを正式キックオフする前に、AI PMが「仮想シミュレーター」を起動。
    *   プロジェクト要件とWBSタスク網を、シミュレーション用エージェント群（Coder, Writer, Analyst）に渡し、バックグラウンドのLLMスレッド上で「仮想対話、コード生成、品質レビュー」を1万回繰り返し実行。
    *   「このままのデザイン規約だと、NextJS_Coderが後半のフェーズでTailwindのカラーコードのエラーでハマる確率が **72%** あります」「このアサインだと、3日目にAPIレート制限に引っかかります」といった、未来のボトルネックをキックオフ前に超高確率で予知（Strategic Predictive Sandbox）。
    *   シミュレーション結果から、あらかじめ最適化・バグ対策が施された「完璧な初期WBSと担当アサイン設計」のみを現実のWorkspaceへ展開し、手戻りゼロ、バグゼロ、遅延ゼロの圧倒的業務品質を実現する、究極の経営シミュレーターへと進化させます。
