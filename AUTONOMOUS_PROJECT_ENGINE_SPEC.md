# AI Workspace 「自律ライフサイクル推進・オートノマスプロジェクトエンジン（Autonomous Project Engine）」機能設計仕様書

本書は、AI Workspaceにおけるあらゆる新規事業、アプリケーション開発、マーケティングキャンペーン、データ分析業務といった「プロジェクト」全体の受託から完成、公開、保守、継続改善までの全フェーズを、AIが自律的に統治・管理・駆動する自律型プロジェクトマネージャー**「Autonomous Project Engine（オートノマス・プロジェクト・エンジン）」**のシステム構成、データスキーマ、動的リソーススケジューリング、障害自動復旧（Failure Recovery）、生産パイプライン、および画面レイアウトを定義した製品設計仕様書です。

本エンジンは、人間が手動でタスクをアサインし、期限を管理し、成果物をレビューするという従来の受動的プロジェクト管理ツールの限界を完全に破壊。プロジェクトの目的（Goal）と要件（Project Analysis）が投入された瞬間、AIが「タスク工程（Milestones）」「優先順位」「期限」を自律生成（Autonomous Planning）し、社内の最適なAIモデル、エージェント、および外部コネクタを適材適所で配備実行（Autonomous Execution）。途中でAPI障害やAIのサービス停止が発生した場合も、ミリ秒レベルで「代替AI」や「フォールバック代替ルート」を選択して自動復旧（Failure Recovery）を実行し、100%自律的に成果物を完成・公開まで導く、AI Workspace全体の**「自律最高プロジェクト執行・運営責任者（Autonomous Chief Operating Officer Engine）」**の役割を果たします。

## 1. システム概要 ＆ 自律プロジェクト運営コンセプト (Overview & Concept)

Autonomous Project Engineは、下図に示す「自律クローズドループ・パイプライン（Autonomous Closed-Loop Pipeline）」に基づき、プロジェクトの受託から保守運用までのライフサイクル全般を統制します。
人間やデジタルツインが提示した抽象的なゴールを「詳細要件仕様」へセマンティック分解し、動的リソース選択（API・コスト・速度・品質の最適バランス）を行いながら、並列自律エージェントを物理駆動。さらに、全自動のテストおよび本番リリースゲートを設け、現実世界の成果（売上、CVR）をフィードバックし続ける「不滅のプロジェクト駆動インフラ」を構築します。

```
     [ 👥 ユーザーの要求 / 経営ビジョン ] (例: 「新宿店のMEOと Stipe 売上を改善するプロジェクトを立ち上げて」)
                         │
                         ▼
     ===================================================================================
                         [ Autonomous Project Engine ]
     ===================================================================================
     ├─ ① 自律要件分析・市場監査ゲート (Project Analysis Gate)
     │    - 目的分析、詳細要件定義の自律生成、市場・競合脅威・技術的実現可能性の事前同時分析
     ├─ ② 動的工程計画・ワークフロー生成器 (Autonomous Planning Engine)
     │    - ガントマイルストーン、タスク分解、優先度・期限、担当AI、必要自動ワークフローの動的アサイン
     ├─ ③ 賢守リソース・APIオプティマイザ (Resource Optimizer)
     │    - APIコスト、応答速度、回答品質から最適LLMモデルをスケジューリング（最適化問題を解決）
     ├─ ④ 自律物理実行・調停コア (Autonomous Execution Core)
     │    - 召喚された並列Agent、自動Workflow、Connector Hub（Stripe、GBP）を非同期並列ドライブ
     ├─ ⑤ 自己治癒・API/AI障害自己復旧回路 (Failure Recovery System)
     │    - AI障害、APIレート制限（429）、ワークフロータイムアウトをリアルタイム自己検知
     │    - ミリ秒レベルでの「代替AI（Gemini ➔ DeepSeek）」自動切り替え・代替タスク復旧
     └─ ⑥ 統合生産パイプライン (Production Pipeline Manager)
          - 企画 ➔ 設計 ➔ 制作 ➔ テスト（QA） ➔ 公開（Reality Engine） ➔ 保守 ➔ 自動Version管理の統合駆動
                         │
                     ┌───┴──────────────────────┐
                     ▼ (完全完成したプロジェクト成果)     ▼ (プロジェクト運用の健全性逆流)
        【💾 Reality Engine (本番自動公開)】      【📊 AI Project Manager Dashboard】
         - 成果物（LP、コード、資料）の自動デプロイ  - 進捗率、完成予測日、コスト、リスクのリアルタイム表示
         - 認証キー・環境変数のセキュアマウント       - 障害復旧の履歴、 蓄積成果（KPI）の時系列追跡
```

### テーブル定義詳細

#### ① `auto_projects` (自律プロジェクト統合マスタ)
プロジェクト全体の目的、ステータス、API予算、現在までの累積消費コスト、AI予測される完成日時、および全天候リスクスコアを管理。
*   `id`: UUID (Primary Key)
*   `company_id`: UUID (Foreign Key - 組織スペースのID)
*   `title`: VARCHAR(255) (プロジェクト名。例: `「La Lounge 新宿店 集客・売上改善」`)
*   `goal_id`: UUID (Foreign Key, Nullable - 同期するビジネスゴール（Goal Engine）のID)
*   `status`: VARCHAR(50) (ステータス。`ANALYSIS` (要件分析中), `PLANNING` (工程生成中), `EXECUTING` (工程自律実行中), `QA_TEST` (品質・卓越性テスト中), `DONE` (完了・リリース済み))
*   `total_budget_limit`: DECIMAL(10, 4) (プロジェクトに割り当てられたAPIコストの最大上限（USD）。例: `10.0000` (10ドル))
*   `current_costs_accumulated`: DECIMAL(10, 4) (現在までの累積消費APIコスト。例: `2.4200`)
*   `completion_forecast_at`: TIMESTAMP (AI時系列 Prophet モデルが予測した、プロジェクトが100%完了するタイムスタンプ)
*   `risk_score`: DECIMAL(5, 2) (マイルストーン遅延、API予算、外部競合、および品質スコアからAIが毎時算出する「プロジェクト崩壊リスク率」。0.00〜100.00 %)
*   `created_at` / `updated_at`: TIMESTAMP

#### ② `auto_milestones` (自律マイルストーン工程テーブル)
プロジェクト要件から自律生成された、大工程（フェーズ）の定義、優先順位、および期限。
*   `id`: UUID (Primary Key)
*   `project_id`: UUID (Foreign Key)
*   `title`: VARCHAR(255) (マイルストーン名。例: `「[M3] コード実装 ✕ データベースマウント」`)
*   `priority`: INT (工程の優先順。低いほど先。例: `3`)
*   `due_at`: TIMESTAMP (このマイルストーンをクリアすべき期限日時)
*   `status`: VARCHAR(50) (状態。`PENDING`, `RUNNING`, `COMPLETED`, `OVERDUE` (期限超過))
*   `created_at`: TIMESTAMP

#### ③ `auto_tasks` (自律アサイン・タスクテーブル)
マイルストーンを細分化した個別のアクション、担当するAI、見積もり時間、期限、および実行状態。
*   `id`: UUID (Primary Key)
*   `milestone_id`: UUID (Foreign Key)
*   `title`: VARCHAR(255) (タスク名。例: `「予約獲得LPのHTML/Tailwindコード生成」`)
*   `priority`: INT (タスク優先度。例: `1`)
*   `assigned_ai`: VARCHAR(100) (タスクを割り当てられたAIモデル。例: `CLAUDE_3_5_SONNET`)
*   `duration_est`: INT (見積もり処理時間（秒数）)
*   `due_at`: TIMESTAMP (タスク期限)
*   `status`: VARCHAR(50) (状態。`BACKLOG`, `IN_PROGRESS`, `DONE`, `FAILED`)
*   `created_at`: TIMESTAMP

#### ④ `auto_executions` (タスク物理実行・APIコストログ)
個別のタスクに対して実行された具体的なAI API呼び出し、ワークフロー起動、またはエージェント物理動作の、コストおよび処理時間の実績監査。
*   `id`: UUID (Primary Key)
*   `task_id`: UUID (Foreign Key)
*   `executor_type`: VARCHAR(50) (実行主体。`AI_RUN` (単一LLM実行), `AGENT` (自律エージェント起動), `WORKFLOW` (自動ワークフロー実行), `CONNECTOR` (外部API接続))
*   `executor_id`: UUID (Nullable - 起動したエージェントやワークフローのID)
*   `cost`: DECIMAL(10, 6) (この実行で消費した正確なトークンAPIコスト（USD）)
*   `duration_ms`: INT (処理にかかった時間（ミリ秒）)
*   `status`: VARCHAR(50) (状態。`SUCCESS`, `FAILED`, `TIMEOUT`)
*   `created_at`: TIMESTAMP

#### ⑤ `auto_recoveries` (自己治癒・障害自動復旧履歴テーブル)
タスク実行に障害（APIダウン、レート制限、タイムアウト）が発生した際、AIがミリ秒で起動した「自動復旧（Failure Recovery）」の実行履歴。
*   `id`: UUID (Primary Key)
*   `execution_id`: UUID (Foreign Key - 障害が発生した元の実行ログID)
*   `failure_type`: VARCHAR(100) (障害区分。`RATE_LIMIT` (429エラー), `TIMEOUT` (504等), `API_DOWN` (500等), `QA_REJECT` (テスト不合格による差し戻し))
*   `alternative_ai`: VARCHAR(100) (代替として緊急召喚されたAIモデルまたはルート。例: `GEMINI_1_5_PRO`)
*   `recovery_status`: VARCHAR(50) (復旧ステータス。`IN_RECOVERY`, `SUCCESS`, `UNRECOVERABLE` (自律復旧不可能、人間にエスカレーション))
*   `critique_text`: TEXT (障害の詳細、および自律復旧で適用されたフォールバックポリシー（バックオフ等）の解説)
*   `created_at`: TIMESTAMP

#### ⑥ `auto_versions` (継続パブリッシュ・不変バージョン管理)
プロジェクトの完了アセット（LP、資料、コード、DBスキーマ）を Reality Engine で本番公開した際の、Gitに準拠したバージョン、変更ログ、コミットハッシュ。
*   `id`: UUID (Primary Key)
*   `project_id`: UUID (Foreign Key)
*   `version_tag`: VARCHAR(50) (バージョン名。例: `v2.4.0`)
*   `change_log`: TEXT (不変の更新履歴文章)
*   `commit_hash`: VARCHAR(100) (ソースコードがGit管理されている場合のSHA-1コミットハッシュ)
*   `payload_json`: JSON (公開アセットへの物理URL、デプロイ構成、環境変数定義のスナップショット)
*   `created_at`: TIMESTAMP

## 4. 自律プランニング ＆ リソース最適化・障害自動復旧ロジック (Engine Logic)

Autonomous Project Engineが、人の管理や指示を一切受けることなく、100%自律推進（SLA保証）を達成するための4段階の自律推進・自己治癒パイプライン。

```
     [ 1. 目的のインジェスト ➔ 工程・ガントの自律生成 (Autonomous Planning) ]
       - ユーザーの目的（例: 新宿店売上改善）から、必要マイルストーン工程（ auto_milestones ）を自動定義。
       - 各マイルストーンを具体的な自律タスク（ auto_tasks ）に分解。
       - 過去の実績（Memory）から「タスクの見積もり時間（ duration_est ）」を算出し、期限（ due_at ）を設定。
                            │
                            ▼
     [ 2. 賢守リソーススケジューリング最適化 (Resource Optimization) ]
       - 「コストを最小にし、かつ期限に100%間に合わせ、品質を担保する」という効用最大化問題を毎時解決。
       - 分析・テストなど高度な論理推論タスク ➔ Gemini 1.5 Pro ✕ DeepSeek R1 を配備。
       - 定型的なHTML/CSS生成、ドキュメント要約 ➔ Claude 3.5 もしくは GPT-4o を配備。
       - API利用における「レート制限（Rate Limit）回避」のため、リクエスト間隔を自律制御（最適パッシング）。
                            │
                            ▼
     [ 3. 物理エージェント ✕ ワークフロー駆動 (Autonomous Execution) ]
       - スケジュールに従い、Master Orchestrator（共同制作Swarm）や Reality Engine を自律起動。
       - 外部連携（GBP、Stripe等）は Connector Hub 経由で実行。
       - 成果物ができあがると、自動的に「テスト（QA）」工程に送り、静的・動的テストをパスするまでループ。
                            │
                            ▼
     [ 4. 自己治癒・API障害自動復旧 (Failure Recovery State Machine) ]
       - AI APIのダウンやレート制限（429）、タイムアウト（504）を、 auto_recoveries システムが常時監視。
       - エラー検知から **5ミリ秒以内** にフォールバック判断を執行：
         - 429（Rate Limit） ➔ **指数バックオフ自動リトライアルゴリズム**を即時実行。
         - 5xx（サーバーダウン） ➔ 担当タスクを**「代替AI（Gemini 1.5 Pro等）」へ自動振替マウント**して再起動。
       - 復旧に成功した場合、スケジュールを1ミリ秒も遅延させることなく工程を継続。
         自律復旧不可能（3回リトライ失敗）な場合のみ、人間のデジタルツインへセキュアエスカレーション。
```

### ① リソース選択の動的効用スケジュールモデル (SLA Optimization)
タスク $T$ に対するリソース（AIモデル・API） $R$ の割り当ては、マイルストーンの残り時間 $Time_{remain}$、残予算 $Budget_{remain}$、および要求品質 $Quality_{req}$ に基づく動的計画法（Dynamic Programming）を用いて、毎時最適化更新されます。
*   **残り時間に余裕がある場合 ($Time_{remain} \gg 0$):**
    *   システムはコスト効率を最優先し、APIコスト $C_R$ が最も安い「DeepSeek R1」や「Gemini Flash」を優先選択。
*   **マイルストーン期限が逼迫している場合 ($Time_{remain} \to 0$):**
    *   システムは「スピード（最速応答） $S_R$」を最優先し、並列同時処理を最大化。多少高価でも応答速度が最速の「Gemini Pro / GPT-4o」を同時召喚し、並列実行させます。

### ② 指数バックオフ自己治癒リトライアルゴリズム (Exponential Backoff Recovery)
APIレート制限（429）を検知した際の、自己治癒リバース・タイムリミット算定式。
$$Delay_{retry}(n) = Base \times 2^n \pm \text{Jitter}$$
*   $n$: リトライ回数（最大 3 回）。
*   $Base = 1000$ ミリ秒。
*   $Jitter$: 複数エージェントによる同時リクエストの再衝突（Thundering Herd Problem）を回避するための、微小なランダム揺らぎ（±100ms）。
*   この自律バックオフにより、ネットワークと外部APIに不要な負荷をかけることなく、最短かつ最も安全なルートでタスクを自己治癒・正常完了まで回復させます。
