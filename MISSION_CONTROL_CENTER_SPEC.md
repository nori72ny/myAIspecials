# AI Workspace 「ミッション・コントロール・センター（Mission Control Center）」機能設計仕様書

本書は、AI Workspace全体の稼働状況、進行中プロジェクト、所属AIエージェント、自動化ワークフロー、マルチLLM（AI Usage）コスト、ナレッジの鮮度、および各種システムアラートや承認依頼をリアルタイムで一元監視・統治するための最上位司令塔システム**「Mission Control Center（ミッション・コントロール・センター）」**のシステム構成、ダッシュボード設計、データスキーマ、および外部連携アーキテクチャを定義した製品設計仕様書です。

Workspace内に点在する「個別プロジェクト（Projects）」「エージェント（Agents）」「自動化（Automations）」「ツール接続（Tool Hub）」「知識構造（Knowledge Graph）」のすべてからライブテレメトリを収集し、組織のオーナーやデジタルツインが「Workspace全体で今何が起きているか」「AIの稼働率とコスト効率は最適か」「人間による承認待ちタスクはあるか」を瞬時に把握・指揮できる、いわば**「AI Workspaceのコックピット（Central Terminal Board）」**の役割を担います。

---

## 1. システム概要 ＆ 司令塔コンセプト (Overview & Concept)

Mission Control Centerは、Workspace内の全サブシステムからデータをリアルタイムに吸い上げ、統合ダッシュボードへと反映します。さらに、エラーの検知時や「人間による最終確認（Human-in-the-Loop）」が必要な場合に、管理者へインタラクティブな通知を配信し、適切な介入手段を提供します。

```
       ===================================================================
                         [ 外部イベント / テレメトリ収集 ]
       ===================================================================
       ├─ [Projects]  ➔ 進行ステータス、デッドライン、期限通知
       ├─ [Agents]    ➔ 稼働状態、タスクの成否、利用履歴統計
       ├─ [Workflows] ➔ 実行中、予約スケジュール、完了、失敗ログ
       ├─ [Brain/LLM] ➔ 各モデル（Gemini, Claude, DeepSeek等）のAPIトークン・コスト
       ├─ [Knowledge] ➔ ナレッジグラフの更新履歴、利用度、最重要インサイト
       └─ [ActionHub] ➔ コネクタの認証ステータス、APIコール数、失敗ログ
                                       │
                                       ▼
       ===================================================================
                      [ Mission Control Center Core Engine ]
       ===================================================================
       ├─ 1. ライブテレメトリ集計エンジン (Real-time Telemetry Collector)
       ├─ 2. コスト ＆ 効率最適化アナライザー (LLM Cost Optimizer)
       ├─ 3. アラート ＆ 承認ディスパッチャー (Alert & Consent Dispatcher)
       └─ 4. インタラクティブ統合ダッシュボード (Interactive Control Panel)
                                       │
                                       ├──────────────────────────────────┐ (アラート・承認要請)
                                       ▼                                  ▼
                     [ 👥 人間（管理者・オーナー） ]         [ 🧠 個人・企業デジタルツイン ]
                     ├─ 今日の状況・重要事項の把握           ├─ バックグラウンドでの自律監視
                     ├─ 優先タスクの執行・アサイン確認       ├─ エラー検知時の自己修復・代替案模索
                     └─ 成果物の最終承認・公開実行           └─ 自律的なリソース・コスト最適化
```

### テーブル定義詳細

#### ① `mission_control_summary` (統合テレメトリアグリゲーション)
Workspace全体の現在のリアルタイム健全性と稼働総サマリーを最速で描画するための高頻度更新キャッシュテーブル。
*   `id`: UUID (Primary Key)
*   `company_id`: UUID (Foreign Key - 組織スペースのID)
*   `total_active_projects`: INT (進行中プロジェクトの現在総数)
*   `total_active_agents`: INT (現在稼働中のAI社員数)
*   `total_active_automations`: INT (登録・アクティブな定期実行スケジュール数)
*   `current_monthly_cost_usd`: DECIMAL(10, 4) (当月のAPI消費コスト累計)
*   `budget_limit_usd`: DECIMAL(10, 4) (システム管理者が設定した月間API予算上限)
*   `health_status`: VARCHAR(50) (現在のWorkspace健全性。`EXCELLENT` (オールクリア), `WARNING` (要確認エラーあり), `CRITICAL` (深刻なバグ・予算オーバー等による緊急停止))
*   `updated_at`: TIMESTAMP

#### ② `telemetry_alerts` (システムアラート ＆ エラーログ)
Workspace内の各モジュールから吸い上げられた、管理者の即時確認や対応が必要なアラート。
*   `id`: UUID (Primary Key)
*   `summary_id`: UUID (Foreign Key)
*   `alert_level`: VARCHAR(50) (重要度。`INFO` (通知), `WARNING` (警告), `ERROR` (エラー・処理中断))
*   `message`: TEXT (アラートの具体的な解説。例: `エージェント 'MEO_Bot' のGoogle APIキーが無効化されました`)
*   `is_resolved`: BOOLEAN (対応・解決が完了したか)
*   `resolved_at`: TIMESTAMP (解決された日時)
*   `created_at`: TIMESTAMP

#### ③ `llm_usage_finance_logs` (AI Usage ＆ APIコストログ)
マルチLLM（Gemini, OpenAI, Claude, DeepSeek等）をエージェントやワークフローが呼び出すたびに記録されるコストトラフィックログ。
*   `id`: UUID (Primary Key)
*   `summary_id`: UUID (Foreign Key)
*   `model_name`: VARCHAR(100) (呼び出されたLLMモデル名。例: `google/gemini-2.5-pro`, `anthropic/claude-3.5-sonnet`)
*   `input_tokens` / `output_tokens`: INT (消費されたトークン数)
*   `cost_usd`: DECIMAL(12, 8) (実際に発生したAPIコスト)
*   `is_success`: BOOLEAN (API呼び出しが正常に成功したか。失敗時はコスト0、または失敗ログの追跡用)
*   `created_at`: TIMESTAMP

#### ④ `system_approvals` (人間による承認依頼 (Human-in-the-Loop))
AIが自律行動する過程で、「最終確認」や「高額コスト支出の合意」「外部への本番投稿」の前に処理を一時保留し、人間に最終クリックを求める案件のログ。
*   `id`: UUID (Primary Key)
*   `summary_id`: UUID (Foreign Key)
*   `requester_agent_id`: UUID (Foreign Key - 承認を要求したAIエージェントのID)
*   `target_project_id`: UUID (Foreign Key - 紐づくプロジェクトのID)
*   `payload_json`: JSON (承認された際に実行するアクションのメタデータ。例: `{"action_type": "NETLIFY_DEPLOY", "site_id": "shinjuku-lounge-lp"}`)
*   `approval_status`: VARCHAR(50) (ステータス。`PENDING` (承認待ち), `APPROVED` (承認済), `REJECTED` (拒否・差し戻し))
*   `created_at`: TIMESTAMP

## 4. ダッシュボード・シグナル設計 (Dashboard Analytics Logic)

ダッシュボード上に高解像度かつ信頼性の高いデータを表示するための、主要KPIの算定・更新ロジック。

### ① 全体AI稼働率 (AI Utilization Rate)
*   **定義**: Workspace内の全タスク処理時間、またはアサイン中の全プロジェクトにおける「自律タスクの稼働比率」。
*   **計算式**:
    $$\text{AI Utilization Rate} = \frac{\text{AIエージェントがアクティブに稼働した総ジョブ秒数}}{\text{全システム監視時間（24h $\times$ チームメンバー数）}}$$
*   **更新間隔**: 5分ごとにバックグラウンドで集計・キャッシュ。

### ② コスト最適化エンジン (Cost Optimization Policy Engine)
*   **概要**: 予算超過を防ぎ、最も安価かつ高度なモデルを効率的に使い分けるためのポリシー設定。
*   *   **ポリシー A (High Quality Preference)**:
        プログラムの実装、複雑な文章要約、データベース設計などには、コストが高くとも `anthropic/claude-3.5-sonnet` をデフォルトアサイン。
    *   **ポリシー B (Cost & Speed Optimization - 推奨)**:
        通常の競合MEO調査、日々の売上スプレッドシート集計、ログの解析、一次応答などの定型タスクは、圧倒的に高速かつトークンコストが非常に安価な `google/gemini-2.5-flash` を自動選択。
    *   **ポリシー C (Autonomous Router)**:
        タスクが発生した際、Unified AI Brainが「タスクの難易度」を1〜5で事前査定。難易度4以上のみSonnetにルーティングし、難易度3以下は自動的にGeminiに転送することで、品質を100%維持したまま**APIコストを最大70%削減**します。

## 5. 外部連携 ＆ 統括ガバナンス (Unified Governance & Coordination)

Mission Control Centerは、AI Workspace全体の「最高経営責任（CEO）」であり、すべてのモジュールの神経をつなぎ合わせるマスターボードです。

```
              +-------------------------------------------+
              |           Mission Control Center          |
              +---------------------+---------------------+
                                    |
          ┌─────────────────────────┼─────────────────────────┐
          ▼                         ▼                         ▼
  【AI Company OS】         【Action Center】         【Knowledge Graph】
   - 組織図の稼働率同期       - セキュアAPIエラー検知     - 情報の更新履歴とハブ度
   - プロジェクト期限追跡     - 承認待ちタスクの一時保留   - 孤立ナレッジのアラート
```

1.  **AI Company OS / Personal AI OS との同期**:
    *   AI Company OSのダッシュボードは、このMission Control Centerのデータを主要なインジケーター（健康診断情報）としてインポートします。
    *   「チームビルディング」や「デジタルツインの稼働」による負荷（メモリ、API上限）を監視し、サーバーのパンクやレートリミット超過を未然に警告します。
2.  **Action Center ＆ Tool Connector Hub との同期 (安全弁)**:
    *   Action Centerを通じて「WordPressへの自動記事投稿」や「NetlifyへのLP公開」を実行する際、Mission Control Centerの `system_approvals`（承認待ちテーブル）に自動的にフラグをインサートして処理をサスペンド。
    *   管理者が画面上の「🟢 承認」をクリックした瞬間にWebhookがキックされ、Action Centerから本番のパブリッシュが安全に執行されます。
3.  **ナレッジグラフ・エンジン（Knowledge Graph Engine）との同期**:
    *   新しく作成された成果物や、読み込まれた外部資料が、自動的にナレッジグラフ上のどのノードと結びついたかの「同期履歴」を監視。
    *   グラフ上のノードが「孤立（孤立ノード数が増大）」している場合、Mission Control Centerは「プロジェクトフォルダ外に接続されていない資料が2件あります。タグを再付与してグラフに統合してください」とダッシュボード上に推薦（Recommender System）を表示します。

## 6. 将来拡張設計 (Future Extensibility Plan)

### ① Cross-Workspace Executive Portal (企業間AI組織・アライアンス監視ポータル)
*   **自社だけでなく、提携企業や子会社のAI Workspace（複数のOS環境）の稼働状況を、1つの超最上位コントロールセンターから串刺しで監視・指揮する仕組み**:
    *   「新宿カフェプロジェクト」で、自社の開発チームと、外部提携不動産会社の仕入れチームが同時に並行稼働している場合。
    *   提携先が公開を許可した特定のプロジェクト・エージェント稼働指標を、セキュアなフェデレーション（API連携）経由で本コントロールセンターに統合。
    *   お互いの企業機密（開発原価や顧客データ等）はマスポートで隠蔽しつつ、「マイルストーンの進捗」や「APIエラーによるボトルネックの有無」だけを1つのコックピットから相互監視し、JV（ジョイントベンチャー）の進行を最大加速させるマルチテナント統括ポータルへと拡張します。

### ② Self-Healing Infrastructure (完全自律型自己修復 ＆ 冗長化システム)
*   **APIエラーやシステム障害が発生した際、AIエージェントが人間（システム管理者）の指示を待たずに、代替モデルの配備やAPIキーの再発行、代替コネクタの接続を自律的に判断・完遂して処理を継続する仕組み**:
    *   「MEO_Local_Marketer」が稼働中、メインで利用していた Perplexity API で一時的なサーバー障害（500エラー）が発生したと捕捉。
    *   コントロールセンターの「自律復旧AI（COO Agent）」がこのアラートを瞬時に検知。
    *   即座に実行を一時停止（ポーズ）し、代替の「Gemini 2.5 Pro (Search Groundingモード)」を予備システムとして動的にルーティング。
    *   同じタスクを再実行させ、人間の業務時間を1秒もロスすることなくタスクを成功ステータスへ回復（Self-Healing Pipelines）させ、事後報告として「APIエラーのため予備モデルで自動復旧・処理を完了しました」というメッセージのみをダッシュボードの通知トレイに残す、究極の自律統治インフラへと拡張します。
