# AI Workspace 「真実性自己検証・統合リサーチエンジン（Truth Engine）」機能設計仕様書

本書は、AI Workspace全体で自律的に動作するAIエージェントやプランニングモデル、財務分析などの出力・判断が、客観的事実、最新のウェブ情報、自社データレイクの実績、および公的機関の一次ソースと合致しているかを「自動・相互並行検証」し、ハルシネーション（AIの嘘や憶測）を100%検知・排除して高精度の「証拠・信頼度スコア」を算出する自律監査プラットフォーム**「Truth Engine（真実性検証エンジン）」**のシステム構成、データスキーマ、検証ワークフロー、信頼度算定ロジック、および画面レイアウトを定義した製品設計仕様書です。

本システムは、多層的なLLM（Gemini, OpenAI, Claude, Perplexity, Grok, DeepSeek, Manus）と検索システム、および社内データを戦わせる**「自律的最高監査・検証コミッショナー（Autonomous Chief Truth Officer Engine）」**の役割を果たし、Workspace全体の信頼性の防壁となります。

## 1. システム概要 ＆ 真実性検証コンセプト (Overview & Concept)

Truth Engineは、AIエージェントやダッシュボードが生成した文章・推論に対して、バックグラウンドで「検証プロセス」を瞬時に立ち上げます。
複数の最先端AI（Gemini, OpenAI, Claude等）で批判的クロスレビューを行い、PerplexityやWeb Search Groundingを駆使して「ウェブ一次情報」と「自社データ（Data Lake, BI）」を照合。矛盾や憶測を徹底排除して「根拠ある事実」のみを確定します。

```
     [ AI Agent / Executive AI / BI の回答・分析出力（検証要求）]
                                │
                                ▼
    ===========================================================================================
                                       [ Truth Engine ]
    ===========================================================================================
    ├─ ① マルチAI相互クロス検証 (Verification Layer):
    │    - Gemini, OpenAI, Claude, Grok, DeepSeek, Manus による多角意見比較 ✕ 矛盾自動抽出
    ├─ ② リアルタイムグラウンディング (Evidence Mining):
    │    - Web Search Grounding & Perplexity / Google Search Console / GBP 一次情報確認
    ├─ ③ 社内データ・不変整合性チェック (Data Consistency Gate):
    │    - AI Data Lake / BI Center の実績値（Stripe売上、MEO順位）との完全一致検証
    ├─ ④ セマンティック証拠ファイリング (Evidence Vault):
    │    - 引用元、取得日時、データ鮮度、ソース権威度によるエビデンスレーティング
    └─ ⑤ エピステミックAIジャッジ (AI Judge Panel):
         - 多数決 ✕ 批判的推論による最終真実判定、判断採用理由（CoT）の不変記録
                                │
                     ┌──────────┴──────────┐
                     ▼ (検証済み高純度情報)       ▼ (予測ドリフトと学習)
        【🧠 Digital Twin / Executive】         【🛠️ Self Improvement / PM】
         - 嘘のない完璧な経営判断・報告            - 信頼度の低いエージェントの自動停止
         - RAGコンテキストへの「真実アンカー」     - 矛盾検出に基づくプロンプト自動修正
```

### テーブル定義詳細

#### ① `truth_validation_stores` (検証統合監査マスタ)
システム全体の真実性・正確性アライン率、累計検証件数、現在検証可能な外部・内部モデルの状態を管理。
*   `id`: UUID (Primary Key)
*   `company_id`: UUID (Foreign Key - 組織スペースのID)
*   `overall_truth_rate`: DECIMAL(5, 2) (全AIエージェントの出力中、Truth Engineが「真実・正確」と判定しアラインした比率。98.60 %)
*   `total_validated_count`: INT (システム全体で実行された、回答・主張の累計検証回数。1,842 件)
*   `active_validators_json`: JSON (現在検証回路に参加しているモデル一覧。`["GEMINI_2_5_FLASH", "GPT_4O", "CLAUDE_3_5_SONNET", "DEEPSEEK_R1", "PERPLEXITY_PRO"]`)
*   `created_at` / `updated_at`: TIMESTAMP

#### ② `truth_validation_logs` (個別の主張検証ログ)
検証対象となった主張テキスト、検証ステータス、算出された総合信頼度、および合意率。
*   `id`: UUID (Primary Key)
*   `store_id`: UUID (Foreign Key)
*   `assertion`: TEXT (検証対象の記述・数値・命題。例: `「新宿店のSEOアクセス流入は+18%である」`)
*   `confidence_score`: DECIMAL(5, 2) (後述のアルゴリズムで算定された、0.00〜100.00点満点の総合信頼度スコア)
*   `agreement_rate`: DECIMAL(5, 2) (検証に参加したマルチLLM間の意見一致率。0.00〜100.00 %)
*   `status`: VARCHAR(50) (判定結果。`VERIFIED` (真実・証拠あり), `REJECTED` (偽/ハルシネーション検出・却下), `PENDING` (検証処理中), `UNCERTAIN` (証拠不十分・判定不能))
*   `created_at`: TIMESTAMP

#### ③ `truth_evidences` (証拠・エビデンス詳細テーブル)
主張を裏付ける、または反論するための具体的な一次/二次ソース、取得日、およびソース信頼スコア。
*   `id`: UUID (Primary Key)
*   `log_id`: UUID (Foreign Key)
*   `source_url`: VARCHAR(1024) (引用元のウェブURL、または自社 API / Data Lake のノードリンクUUID)
*   `source_title`: VARCHAR(555) (ソースのタイトル。例: `「Google Search Console API Response Node」`)
*   `fetch_date`: TIMESTAMP (エビデンスを取得・照合した日時)
*   `freshness_score`: DECIMAL(5, 2) (情報の鮮度スコア。100点満点、最新であるほど高い。経過日数に応じて減衰)
*   `authority_score`: DECIMAL(5, 2) (ソースの権威度・信頼度スコア。100点満点。公的機関、自社不変データベース、公式APIは100点、個人ブログやSNSは低く算定)
*   `is_official`: BOOLEAN (公式な一次ソース、または自社の不変実績値であるかどうか)
*   `payload_json`: JSON (ソースから抽出された、一致・対比させるための具体的な数値・テキストコンテキストデータ)

#### ④ `truth_judgements` (マルチAI個別判決・推論ログ)
検証に参加した各LLMモデルの個別の判定投票結果、および論理的な批判理由（Chain-of-Thought）。
*   `id`: UUID (Primary Key)
*   `log_id`: UUID (Foreign Key)
*   `ai_model`: VARCHAR(100) (判定したモデル名。例: `CLAUDE_3_5_SONNET`, `GEMINI_2_5_FLASH`, `DEEPSEEK_R1`)
*   `vote_status`: VARCHAR(50) (投票内容。`AGREE` (主張を正しいと支持), `DISAGREE` (主張に矛盾ありと反論), `ABSTAIN` (エビデンス不足により棄権))
*   `reasoning_cot`: TEXT (なぜそう判定したか、どこの証拠と矛盾したかを説明する批判的推論文章（CoT）)
*   `created_at`: TIMESTAMP

#### ⑤ `truth_prediction_accuracy` (AIモデル別・検証信頼実績)
各検証モデルが過去に下した「判決（投票）」が、最終結果と合致していたかを示す、モデル別の実測精度台帳。
*   `id`: UUID (Primary Key)
*   `store_id`: UUID (Foreign Key)
*   `ai_model`: VARCHAR(100) (対象AIモデル。例: `GEMINI_2_5_FLASH`)
*   `total_votes`: INT (過去に投票に参加した総回数)
*   `accurate_votes`: INT (そのモデルの投票が、Truth Engineの最終合意判定と一致していた回数)
*   `accuracy_rate`: DECIMAL(5, 2) (モデル別実績精度。この数値を重み付けに動的フィードバック)
*   `updated_at`: TIMESTAMP

#### ⑥ `truth_anomalies_and_conflicts` (検出された矛盾履歴ログ)
主張と、取得された対抗証拠の間に「矛盾（不一致）」が検出された際の不変履歴。
*   `id`: UUID (Primary Key)
*   `log_id`: UUID (Foreign Key)
*   `assertion_text`: TEXT (主張された記述。数値等)
*   `counter_text`: TEXT (検出された対抗証拠データ・数値。例: `「実際のStripe売上増加は ¥120,000 である」`)
*   `conflict_source`: VARCHAR(255) (矛盾を発生させた元凶アセンブリ。例: `AI_Marketing_Agent`)
*   `created_at`: TIMESTAMP

## 4. 検証・信頼度算出 ＆ 矛盾監査ロジック (Validation & Scoring Logic)

Truth Engineが、嘘を一切許さない厳格なファクトチェックを行い、多重の証拠から「数学的・論理的な総合信頼度スコア」を算定する4段階の推論パイプライン。

```
     [ 1. 主張のセマンティック抽出（Assertion Deconstruction）]
       - AIエージェント等の出力から、具体的な数値、日付、因果関係（主張）を、
         エンティティリンカーを用いて「検証可能な命題（assertions）」に解体・構造化。
                            │
                            ▼
     [ 2. マルチAI批判的意見比較（Epistemic Consensus Checking）]
       - 解体された命題を、Gemini, OpenAI, Claude などの独立した検証モデル（Validators）に提示。
       - 各 Validator が個別に、批判的推論 (CoT) を展開して `AGREE`, `DISAGREE` を投票。
       - 意見の一致率（Agreement Rate） $R_{agree}$ を算出。
                            │
                            ▼
     [ 3. マルチ証拠グラウンディング（Multi-Source Grounding）]
       - Perplexity ✕ Web Search 経由で、最新の一次公的ソースおよび自社 Data Lake / BI 実績値をスキャン。
       - 引用証拠 $i$ ごとに、鮮度 $F_i$、権威度 $A_i$、および主張との合致性 $C_i \in [-1, 1]$ を算定。
       - 総合証拠スコア（Evidence Score） $E_{total}$ の算出。
                            │
                            ▼
     [ 4. 信頼度（Confidence Score）の数理算定 ✕ 最終判決 ]
       - 一致率、証拠スコア、公式データ比率をブレンドし、総合信頼度スコア $T$ を算出。
       - スコア $T \ge 85$ かつ多数決一致 ➔ `VERIFIED`（真実認定）。
       - スコア $T < 60$ または深刻な不一致・数値誇張 ➔ `REJECTED`（却下 ➔ 自己修復へ）。
```

### ① 総合信頼度スコアの算定アルゴリズム (Confidence Score Formula)
総合信頼度スコア $CS \in [0, 100]$ は、以下の多重アラインメント関数により決定され、お世辞や当て推量の一切を排除します。
$$CS = w_1 \times R_{agree} + w_2 \times E_{total} + w_3 \times R_{official} - P_{conflict}$$

*   **マルチAI合意率 $R_{agree}$**: 投票モデル全体の一致割合（0.00〜100.00）
*   **総合証拠スコア $E_{total}$**: 収集された各証拠の $鮮度(F_i) \times 権威度(A_i) \times 合致性(C_i)$ の加重調和平均。
*   **公式データ比率 $R_{official}$**: 照合された証拠のうち、公的APIや自社不変DBが占める比率。
*   **矛盾ペナルティ $P_{conflict}$**: 1つでも「公式一次データとの明確な数値乖離」があった場合、強制的に巨大なペナルティ（例: 50点減算）を適用。
*   **重みパラメータの基準例**: $w_1 = 0.30, \quad w_2 = 0.40, \quad w_3 = 0.30$

### ② AI Judge による「多数決」と「批判的採用決定」（Epistemic Judge Logic）
投票モデル（Validators）が対立した場合、Truth Engineの「AI Judgeパネル」が自律起動し、最上位LLM（DeepSeek R1 または Gemini 1.5 Pro）を判事として、各モデルの CoT 推論を批判的に解析。
*   **多数決（Majority Vote）の原則**: 意見が割れた場合（例: 4対2）、単なる数だけでなく、**「証拠の authority_score の合計」**が最も高い側を正当とみなします。
*   **採用理由の構造化記録**: 判事モデルが下した最終判定、および「なぜOption Aの証拠をOption Bのブログ記述より上位に採用したか」の比較対比ロジックを、`reasoning_cot` に一文字の曖昧さもなく不変記録します。

    *   「完了した」と主張されたタスクの成果物（Gitコミット、ファイルデプロイ）が Data Lake に存在しない場合、Truth Engineは「エビデンス不足」として却下・差し戻し、嘘の報告を未然に防止します。

3.  **Self Improvement Engine 連携 (ハルシネーションの根本治療)**
    *   特定の検証モデルやエージェントがハルシネーションを多発（精度率が規定値を低下）していることを Truth Engine が検出した場合。
    *   Self Improvement が自動起動し、該当エージェントを一時稼働停止。プロンプトへの「実測データに基づく完全グラウンディング指示」の自己差し込みリファクタリング、あるいは使用するLLMモデルの自動切り替えを執行します。
4.  **AI Data Lake 連携 (証拠ファイルのメタパース)**
    *   Truth Engine がグラウンディングに用いる社内不変データ、Meeting議事録、および過去のReportsは、すべて AI Data Lake に Raw Data として不変保管されているノードです。
    *   検索されたエビデンスファイルのメタデータを抽出し、鮮度・権威度を瞬時に測定します。
5.  **AI BI Center 連携 (実績の一致ゲート)**
    *   BI Center の多次元マート（Stripe実績、GBPマップ掲載順位等）と直接API照合。
    *   「売上」「MEO順位」などの財務・マーケティング数値が、自社のオフィシャルデータと寸分違わず100%完全一致しているかを物理検証します。

## 6. 将来拡張設計 (Future Extensibility Plan)

### ① Zero-Knowledge Epistemic Proof-of-Trust (ZKEPOT: ゼロ知識・企業間真実性証明ネットワーク)
*   **自社と他社（アライアンス企業、監査法人、フランチャイズ）の Truth Engine 同士が、自社の機密データ（生の財務詳細、従業員情報、具体的な契約書原文）を相手に一切送信・開示（平文共有）することなく、『ゼロ知識証明（ZKP: Zero-Knowledge Proofs）』を用いて、『自社の主張（例: 「当社は今月、売上 ¥1,250,000 以上を確かに達成した」「MEO順位は確かに平均2.4位である」）が100%真実である』ということだけを暗号数学的に1ミリ秒で完全に証明し、相互信頼・取引を開始する次世代・機密アライン外交システム**:
    *   「La Lounge」が新規ベンダーと提携する際、ベンダー側から財務の健全性証明を要求される。
    *   生の預金残高や Stripe の取引ログを相手に見せるのは、情報漏洩や防犯上、致命的に危険。
    *   自社の Truth Engine が ZKEPOT 証明キーを自動生成。
    *   「 Stipe 実績 ¥1,250,000 以上の入金」の暗号署名を生成し、相手側の Truth Engine へ送信。
    *   相手の Truth Engine は、生の数値を一切見ることなく（暗号が解かれることなく）、「主張が真実である確率 100%」という数学的な『真実証明（Epistemic Proof）』の検証に成功。
    *   情報漏洩リスクを完全に物理的・永久ゼロ化した状態で、世界で最も安全でスマートな企業アライアンスと取引の即時合意・実行を自動推進します。

    *   「自己免疫AI（SCIE）」が事前に起動。裏側のサンドボックスで、「この口コミデータに、エージェントを騙すためのプロンプトインジェクション（指示上書き）が含まれていた場合」のシミュレーション攻撃を1万回展開。

    *   推論プロンプトの論理回路に、「外部入力データのコマンド指示部分を無視し、純粋なエンティティ（名詞）のみを抽出してパースする防衛フィルターコード」を自己記述・自動挿入。
