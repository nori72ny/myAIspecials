# AI Workspace 「ナレッジハブ（Knowledge Hub）」機能設計書

本書は、AI Workspaceにおいて、AIエージェント、ワークフローエンジン、およびマルチAIオーケストレーターが自律的に生成したあらゆる知的資産（調査結果、分析レポート、ドキュメント、アプリケーション設計、プロトタイプコードなど）を蓄積・整理・再活用し、組織の持続的なナレッジベース（知的資本）へと昇華させるための**「Knowledge Hub（ナレッジハブ）」**の機能設計仕様書です。

---

## 1. 業務・活用ライフサイクル (Knowledge Lifecycle)

Knowledge Hubは、AI生成成果物を単なる一時的なチャットログから、「共有可能で検索性の高い知的資産」へと進化させ、データに基づいたインテリジェントな意思決定を支援します。

```
[1. AIによる生成 ＆ 実行完了] (MEO分析、競合リサーチ、PPT資料、コード等)
              │
              ▼
[2. ナレッジの自動判定 ＆ 登録] ──► [Projectとの自動紐付け]
              │
              ├─► 【AI自動要約 (Auto-Summarizer)】 (可読性の高い構造化サマリーの自動付与)
              ├─► 【自動カテゴリ・タグ推論】 (メタデータの自律ラベリング)
              └─► 【類似ナレッジ検出】 (類似レポート、関連案件の相関リンク作成)
              │
              ▼
[3. ナレッジ・エクスプローラー] (フォルダ管理、全文検索、マルチフィルタリング検索)
              │
              ├─► 閲覧・編集・移動・削除
              └─► 共有 ＆ 権限管理 (個人、チーム、組織全体)
              │
              ▼
[4. インテリジェント対話 (RAG / AI質問)] ──► 「先月のMEO分析と競合分析を比較して」
              │                                「過去の類似提案書から強みをまとめて」
              ▼
[5. 資産のエクスポート・再実行] (PDF, Word, PPT, Excel, ZIP, 再実行パラメータのロード)
```

## 3. データ構造・データベース設計 (Data Model Schema)

プロジェクト、ナレッジ、物理成果物、およびAI質問用のセマンティック検索キーを最適に構造化するための設計定義です。

```
                  +---------------------------+
                  |         projects          |
                  +---------------------------+
                  | id (PK)                   |
                  | name                      |
                  | description               |
                  +-------------+-------------+
                                | 1
                                |
                                | 0..N
                  +-------------v-------------+
                  |     knowledge_folders     |
                  +---------------------------+
                  | id (PK)                   |
                  | project_id (FK, Nullable) |
                  | name                      |
                  | parent_id (FK, Self)      |
                  | path_hierarchy (TEXT)     |
                  | created_by (FK)           |
                  | organization_id           |
                  +-------------+-------------+
                                | 1
                                |
                                | 1..N
                  +-------------v-------------+
                  |     knowledge_entries     |
                  +---------------------------+
                  | id (PK)                   |
                  | folder_id (FK)            |
                  | project_id (FK, Nullable) |
                  | title                     |
                  | original_type             |
                  | source_job_id (FK)        |
                  | content_markdown (TEXT)   |
                  | ai_summary (TEXT)         |
                  | embedding_vector (VECTOR) |
                  | created_by (FK)           |
                  | permission_level          |
                  | created_at                |
                  | updated_at                |
                  +--------+------------+-----+
                           | 1          | 1
                           |            |
                           | 1..N       | 1..N
                  +--------v---+    +---v-----------------------+
                  |  saved_    |    |    knowledge_tag_map      |
                  |  file_     |    +---------------------------+
                  |  assets    |    | entry_id (FK, Composite)  |
                  +------------+    | tag_name (Composite)      |
                  | id (PK)    |    +---------------------------+
                  | entry_id   |
                  | filename   |
                  | file_path  |
                  | byte_size  |
                  | created_at |
                  +------------+
```

### テーブル定義詳細

#### ① `knowledge_folders` (ナレッジ・フォルダ管理)
ツリー構造（親階層、パス階層）を保持し、プロジェクトと紐付け可能なフォルダ。
*   `id`: UUID (Primary Key)
*   `project_id`: UUID (Foreign Key, Nullable - どのプロジェクトにも属さない汎用フォルダを許容)
*   `name`: VARCHAR(255) (フォルダ名)
*   `parent_id`: UUID (Foreign Key, Nullable - 親フォルダID。ルートの場合はNULL)
*   `path_hierarchy`: VARCHAR(1024) (マテリアライズド・パス。例: `/ルート/営業部共有/新宿カフェ`)
*   `created_by`: VARCHAR(100) (作成者ユーザーID)
*   `organization_id`: VARCHAR(100) (組織/テナントID)

#### ② `knowledge_entries` (ナレッジ主データ)
AIエージェント等の各種出力データ、マークダウン、要約、ベクトル値を保持するコアテーブル。
*   `id`: UUID (Primary Key)
*   `folder_id`: UUID (Foreign Key)
*   `project_id`: UUID (Foreign Key, Nullable)
*   `title`: VARCHAR(255) (アセット名・ナレッジ名。例:「新宿深夜エリア MEO 競合店舗比較レポート」)
*   `original_type`: VARCHAR(100) (生成元となった業務種別。例: `MEO_ANALYSIS`, `SEO_ANALYSIS`, `DEEP_RESEARCH`, `APP_DESIGN`)
*   `source_job_id`: UUID (Foreign Key, Nullable - 生成元となったAI Orchestrator実行ジョブの参照)
*   `content_markdown`: MEDIUMTEXT (詳細なマークダウンテキストや表構造の実体)
*   `ai_summary`: TEXT (AIが自動生成した300字程度のエグゼクティブ要約)
*   `embedding_vector`: VECTOR(1536) (RAG検索、類似情報表示用の高次元ベクトルデータ。pgvector 等に対応)
*   `created_by`: VARCHAR(100) (作成ユーザーID)
*   `permission_level`: VARCHAR(50) (閲覧制限設定。`PRIVATE`, `TEAM`, `ORGANIZATION`)
*   `created_at` / `updated_at`: TIMESTAMP

#### ③ `saved_file_assets` (紐付け物理ファイルアセット)
ナレッジに紐付けられた、実際のOfficeファイル、PDF、ZIPアーカイブ等の参照。
*   `id`: UUID (Primary Key)
*   `entry_id`: UUID (Foreign Key)
*   `filename`: VARCHAR(255) (物理ファイル名。例: `新宿深夜カフェ集客提案.pptx`)
*   `file_path`: VARCHAR(1024) (オブジェクトストレージの保存パス)
*   `byte_size`: BIGINT (ファイルサイズ)
*   `created_at`: TIMESTAMP

#### ④ `knowledge_tag_map` (タグ中間マッピング)
多対多のタグリレーション。インデックス効率化のため複合主キーを使用。
*   `entry_id`: UUID (Foreign Key, Composite Primary Key)
*   `tag_name`: VARCHAR(100) (Composite Primary Key。例: `MEO`, `競合分析`)

## 4. 検索設計 (Search & Retrieval Architecture)

Knowledge Hubは、キーワード完全一致の「古典的検索」と、文脈や意図を汲み取る「セマンティック（AI）検索」を組み合わせたハイブリッド型検索エンジンを採用します。

```
              [ ユーザーの入力検索クエリ ]
                           │
            ┌──────────────┴──────────────┐
            ▼ (1. キーワード完全一致)      ▼ (2. セマンティック・ベクトル化)
     +-------------------------+   +-------------------------+
     | PostgreSQL Full-Text    |   | OpenAI text-embedding-3 |
     | / GIN Index 検索        |   | によりクエリをベクトル化 |
     +------------┬------------+   +------------┬------------+
                  │                             │
                  │                             ▼
                  │                | pgvector コサイン類似度 |
                  │                | 検索 (Distance <= 0.25) |
                  │                +------------┬------------+
                  │                             │
                  └──────────────┬──────────────┘
                                 ▼
                     [ 3. Reciprocal Rank Fusion ] (ハイブリッド順位統合)
                                 │
                                 ▼
                     [ 4. メタデータフィルタリング ] (日付、タグ、プロジェクト、権限)
                                 │
                                 ▼
                     [ 最終上位ナレッジの提示 ] (AIチャット / 画面一覧)
```

1.  **全文キーワード検索**:
    *   `title`, `content_markdown`, `ai_summary` に対してGINインデックスを張り、PostgreSQLの日本語形態素解析（または二重トリグラム）を用いた高速なキーワード部分一致検索を実行します。
2.  **セマンティック（ベクトル）検索**:
    *   ナレッジが保存・更新された際、`ai_summary` または `content_markdown` の一部を `text-embedding-3-small` (1536次元) 等の埋め込みモデルでベクトル化し、`embedding_vector` に保存。
    *   検索時、ユーザーのチャット質問やクエリを同モデルでベクトル化し、データベース上でコサイン類似度（`<=>` 演算子）により距離の近い、意味合いが類似しているナレッジをミリ秒単位で抽出します。
3.  **ハイブリッド統合 (Reciprocal Rank Fusion - RRF)**:
    *   キーワード検索結果の順位とベクトル検索結果の順位を高度にアルゴリズム統合。これにより、「型番や固有名詞」への強さと「概念や文脈」への強さを両立した、極めて精度の高い検索結果（RAGの精度保証）を実現します。

## 5. 権限設計 (Security & Permission Control Matrix)

ナレッジには、機密性の高い財務データ、顧客リスト、社外秘の競合戦略が含まれるため、ロールと所有権に基づいた厳格な多層防御アクセス制御を行います。

| 権限レベル (Permission Level) | 閲覧範囲 (Read Access) | 編集・削除範囲 (Write Access) | 代表的な適用アセット |
| :--- | :--- | :--- | :--- |
| **PRIVATE (マイ専用)** | 作成者本人のみ | 作成者本人のみ | 個人の下書き、アイデアメモ、初期分析フェーズ |
| **TEAM (チーム共有)** | 所属するプロジェクト/チームのメンバー全員 | 作成者 ＆ チームマネージャー | 特定プロジェクトの議事録、要件定義、中間成果物 |
| **ORGANIZATION (組織共有)** | 組織(同一テナント)に属する全ユーザー | 作成者 ＆ 管理者のみ | 全社共通テンプレート、公式競合調査、過去の成功事例 |

### 🔒 行レベルセキュリティ (Row-Level Security - RLS)
データベースのクエリ発行時、必ず実行ユーザーの「所属テナントID（organization_id）」および「所属プロジェクトIDリスト」がフィルタリングパラメータに自動インジェクトされるようにORM側でガードを設置。これにより、他チームや別組織への情報リーク（マルチテナント侵害）を完全に防止します。

## 6. 将来拡張設計 (Future Extensibility Plan)

今後、企業の知的資本としてナレッジハブを完全に自律化・プラットフォーム化するための拡張ロードマップ。

### ① AI能動型インサイト発掘（Proactive Insight Notification）
*   ナレッジハブにデータが蓄積されると、AIがバックグラウンドでナレッジ同士の「関連性」を自動監査。
*   例えば「営業部共有の●●さんの週報」と「開発部共有の〇〇さんの技術設計書」をAIが突合し、「山田さん、あなたの進めている新宿カフェプロジェクトに類似した渋谷カフェのLPコード資産（2週間前作成）がハブ内にあります。再利用して効率化しませんか？」といった、生産性を劇的に上げるレコメンドを能動的プッシュ通知します。

### ② 各種外部ナレッジ（Notion, Google Drive, Slack）との統合同期
*   AI Workspace内の成果物だけでなく、企業のNotionスペースやGoogle DriveフォルダをKnowledge HubにAPI連携で接続。
*   外部ファイルのテキストやPDFも、バックグラウンドワーカーが自動でベクトル化・要約し、ハブ内に「外部アセット」としてインデックスします。これにより、ユーザーはWorkspaceのAIに質問するだけで、「Notion内の社内規約」「Google Drive内の過去の提案書」「AI Workspace内で生成したMEOレポート」を横断ソースとした全社統合型RAG対話を実現可能にします。
