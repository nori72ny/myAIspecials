# AI Workspace 「AIデジタルツイン（AI Digital Twin）」機能設計仕様書

本書は、AI Workspaceにおいて、ユーザー本人の「思考プロセス」「価値観」「判断基準」「行動パターン」「仕事観」を自律的に学習し、本人に極限まで酷似した「AIの分身（デジタルツイン）」を生成・運用するためのアーキテクチャ、多層記憶構造、学習フロー、および画面設計を定義した最上位の製品設計仕様書です。

単なる汎用的なアシスタントではなく、ユーザーが過去に作成したチャット履歴、プロジェクトの成果物、執筆したドキュメント、判断を自律学習し、オーナー不在時にも「本人ならどう決定するか」を高精度に予測、または難解な経営判断の「未来予測シミュレーション」を本人に代わって実施する「セルフ・クローン・エンジン」として機能します。

## 1. 業務フロー ＆ デジタルツイン・コンセプト

AI Digital Twinは、ユーザーの「思考の痕跡（学習データ）」を吸い上げる「Learning Layer」、それを短期・長期・プロジェクト別の3つに構造化する「Personal Memory Engine」、価値観に基づいてシミュレーションを行う「Decision Engine」、そして本人との一致率を絶えずフィードバックする「Optimization Cycle」の4つの階層で構成されます。

```
              [1. 学習データの自動集約 (Continuous Learning)]
    (チャット履歴、Projects成果物、Knowledge、メモ、ドキュメント、決定履歴)
                                  │
                                  ▼
                [2. パーソナル記憶エンジン (Personal Memory)]
      ├─ 短期記憶 (Short-term): 直近の進行タスク、会話の文脈
      ├─ 長期記憶 (Long-term): ユーザーの根本的価値観、キャリア、譲れないこだわり
      └─ プロジェクト記憶 (Project-specific): 過去の特定の成功/失敗、業界ルール
                                  │
                                  ▼
                [3. 5大領域における意思決定・判断 (Learning Areas)]
      ├─ 意思決定 / 優先順位: どの案件を優先し、どれを捨てるか (仕事観)
      ├─ 営業・マーケ方針: クライアントへの提案トーン、割引許容度、集客ターゲット軸
      ├─ 開発・デザイン方針: 好みのトポロジー、好むUIテーマ、コーディング規約
      └─ 投資・リスク判断: 予算配分の限界値、許容できる納期遅延・セキュリティ基準
                                  │
                                  ▼
                    [4. ツイン実行モード (Twin Modes)]
        ┌─────────────────────────┼─────────────────────────┐
        ▼                         ▼                         ▼
  【A. Twin Chat (相談)】   【B. Simulation (予測)】  【C. Autonomous Agent (自律)】
   - 本人の思考で壁打ち      - A案・B案・C案の未来予測   - 本人になり代わり、
   - 客観的な内省の誘発      - 本人との一致率・リスク    - OSや会議で判断を自動執行
        │                         │                         │
        └─────────────────────────┼─────────────────────────┘
                                  ▼
            [5. 本人によるフィードバック ➔ 精度向上 (Feedback Loop)]
                 「その判断は85%正しい。残りの15%は…」 ➔ 記憶を動的補正
```

## 3. 記憶構造 ＆ データモデル設計 (Memory Architecture & Schema)

デジタルツインの脳（LLM）が、過去のデータを単なるテキスト情報としてではなく、「思考の優先順位（価値観）」や「感情・譲れない一線」として構造的に引き出すための3階層データベース設計。

```
                  +---------------------------+
                  |         twin_profiles     |
                  +---------------------------+
                  | id (PK)                   |
                  | user_id (FK)              |
                  | twin_name                 |
                  | model_alias (LLM_name)    |
                  | consistency_rate (INT)    |
                  | status (LEARNING, ok)     |
                  | preference_vector_json    |
                  | created_at                |
                  | updated_at                |
                  +-------------+-------------+
                                | 1
                                |
                                ├─────────────────────────────┬─────────────────────────────┐
                                | 1..N                        | 1..N                        | 1..N
                  +-------------v-------------+   +-----------v-------------+   +-----------v-------------+
                  |       twin_memories       |   |      twin_decisions     |   |    twin_audit_feedback  |
                  +---------------------------+   +-------------------------+   +-------------------------+
                  | id (PK)                   |   | id (PK)                 |   | id (PK)                 |
                  | twin_id (FK)              |   | twin_id (FK)            |   | decision_id (FK)        |
                  | memory_type (LONG/PROJ)   |   | scenario_title          |   | user_agreement_rate     |
                  | memory_key                |   | option_a_text (TEXT)    |   | user_comments (TEXT)    |
                  | memory_summary (TEXT)     |   | option_b_text (TEXT)    |   | adjustment_made (TEXT)  |
                  | vector_embedding_id       |   | predicted_choice (TEXT) |   | created_at              |
                  | last_synced_at            |   | reasoning_markdown(TEXT)|   +-------------------------+
                  +---------------------------+   | simulated_at            |
                                                  +-------------------------+
```

### テーブル定義詳細

#### ① `twin_profiles` (デジタルツイン基本マスタ台帳)
*   `id`: UUID (Primary Key)
*   `user_id`: VARCHAR(100) (所有者となる実ユーザーのID)
*   `twin_name`: VARCHAR(255) (ツインの名前。例: 「山田太郎 AIクローン」)
*   `model_alias`: VARCHAR(100) (ツインの思考エンジン。例: `gemini-1.5-pro`, `claude-3-5-sonnet`)
*   `consistency_rate`: INT (現在の本人判断一致率。0〜100%)
*   `status`: VARCHAR(50) (`LEARNING` (同期学習中), `STANDBY` (稼働可能), `ACTIVE` (自律エージェントとしてOSで稼働中))
*   `preference_vector_json`: JSON (ユーザーの価値観・嗜好パラメーターのマトリクス。例: `{"quality_vs_cost": 9.5, "speed_vs_perfection": 4.2, "risk_tolerance": 2.0}`)
*   `created_at` / `updated_at`: TIMESTAMP

#### ② `twin_memories` (3階層デジタルツイン記憶テーブル)
ツインの思考・価値観の核となる多層化された「体験・知識・こだわり」のレコード。
*   `id`: UUID (Primary Key)
*   `twin_id`: UUID (Foreign Key)
*   `memory_type`: VARCHAR(50) (`LONG_TERM` (ユーザーの基本性格・根本思想), `SHORT_TERM` (直近数日間の対話・タスク脈絡), `PROJECT_KNOWLEDGE` (プロジェクト固有のやり方、過去の成功物ノウハウ))
*   `memory_key`: VARCHAR(255) (検索インデックスキー。例: `high_quality_shaping_mind`)
*   `memory_summary`: TEXT (要約された高濃度コンテキスト。例: 「私は顧客体験を最大化するため、予算が多少超過しても防音個室の設備仕様は削らないポリシーを持つ」)
*   `vector_embedding_id`: VARCHAR(255) (ベクター検索データベースに格納された高次元ベクトルのUUID)
*   `last_synced_at`: TIMESTAMP

#### ③ `twin_decisions` (ツイン判断・未来シミュレーション履歴)
ツインが過去にどのような仮説、シナリオに対してどのような判断を下したかの軌跡。
*   `id`: UUID (Primary Key)
*   `twin_id`: UUID (Foreign Key)
*   `scenario_title`: VARCHAR(255) (シミュレートした課題。例: 「新宿カフェ予算10%超過対策」)
*   `option_a_text`: TEXT (選択肢Aの定義)
*   `option_b_text`: TEXT (選択肢Bの定義)
*   `predicted_choice`: TEXT (ツインが「本人ならこれを選ぶ」と予測した結論の要約)
*   `reasoning_markdown`: TEXT (ツインが結論に至った、本人の「価値観DNA」に基づいた思考プロセス説明)
*   `simulated_at`: TIMESTAMP

#### ④ `twin_audit_feedback` (人間による評価・精度向上ログ)
ツインの判断に対して、ユーザーが下した「採点」と「軌道修正」のログ。
*   `id`: UUID (Primary Key)
*   `decision_id`: UUID (Foreign Key)
*   `user_agreement_rate`: INT (人間がツインの判断にどれだけ納得したか。0〜100%)
*   `user_comments`: TEXT (「この結論は合っているが、表現をもう少しマイルドにすべき」「MEOに広告費を振る優先順位を忘れている」などの修正指示)
*   `adjustment_made`: TEXT (フィードバックを受けて、`twin_profiles` や `twin_memories` の重み付けをどう再調整したかの技術ログ)
*   `created_at`: TIMESTAMP

## 4. 学習フロー ＆ 動的判断エンジンの仕組み (Pipeline & Engine)

本人のクローンとして振る舞うために、Unified AI Brainの調停技術をパーソナライズに特化させた「動的思考トレースアルゴリズム」。

### ① 継続的・多重自律学習パイプライン (Continuous Learning Pipeline)
1.  **データインジェクション (Data Ingest)**:
    ユーザーのAI Workspace内での行動履歴（チャットでの発言、編集したMarkdown、作成したLPのデザインスタイル、承認した予算、作成したスライド）を毎晩バックグラウンドで自動バッチクロール。
2.  **意味的チャンキング (Semantic Chunking & Embedding)**:
    単に全テキストを読み込ませるのではなく、テキストを「事実（Facts）」と「判断基準（Directives/Preferences）」にAIが自律分離。
    *   *事実（Facts）*: 「新宿店は24時間営業である」 ➔ Knowledge Hubへ。
    *   *判断基準（Preferences）*: 「夜カフェは、ノイズを遮断して集中できる没入空間こそが最大の強みになる」 ➔ デジタルツインの `LONG_TERM` 記憶へ。
3.  **ベクターマッピング (Vector Mapping)**:
    「判断基準」を多次元ベクトル空間に埋め込み、コサイン類似度で本人の思考キーワードと常時連動させます。

### ② 意思決定シミュレーション・アルゴリズム (Decision Engine)
ユーザーから「相談（クエリ）」を投げられた際、ツインは以下の3ステップで回答を生成します。

```
1. クエリ（相談）を受け取る
   └─ 「予算超過に対して、仕様を削るか、予算を拡大するか？」
                       │
                       ▼
2. セマンティックRAGによる「本人のDNA（こだわり記憶）」の高速抽出
   ├─ クエリベクトルと、`twin_memories` の高次元ベクトルを照合
   ├─ 類似度上位の「私の仕事観」「過去の渋谷での決定履歴」を引き出す
   └─ 抽出例：「私は高単価差別化モデルを好む」「過去、初期費用を仕様維持に全振りした」
                       │
                       ▼
3. 本人の思考になり代わったロールプレイング（Persona Simulation）
   ├─ プロンプトに、抽出した「私の仕事観」を制約条件（System Instruction）としてインジェクション
   ├─ 「私は山田太郎です。私の絶対のこだわりは…」という文脈でシミュレート回答を生成
   └─ A案・B案のそれぞれに対する「想定される自分の反応（リスク・効果）」を出力する
```

## 5. 他システム（OS、会社、プロジェクト、ブレイン）との統合・連携

AI Digital Twinは、AI Workspace全体の「最上位のオーナーコパイロット」として、あらゆる自律システムに人間の意思決定を反映させます。

1.  **AI Company OS / Personal AI OS 連携**
    *   「社長（会長）」であるユーザーが不在時、AI Company OS内の「CEO Agent (Ken)」や「COO Agent (Rei)」が重大な意思決定（例：LP公開の是非、予算超過の補填、MEO対策方針）を迫られた際、デジタルツインを「オーナーの代理（クローン役員）」として会議室（AI Boardroom）に招集。
    *   ツインが「本人なら、新宿LPのテーマカラーは宇宙的ネイビーを選ぶだろう」と会議で発言・合意形成し、そのまま業務を自律進行させます。これにより、人間がパソコンの前にいなくても、ビジネスが本人の美学に従って24時間365日自律的に成長し続けます。
2.  **ナレッジハブ ＆ プロジェクト連携 (Knowledge / Projects Sync)**
    *   ツインが自身の判断精度を上げるために使用する「学習アセット」は、ナレッジハブの個人知識ベースから常時同期。
    *   特定のプロジェクト（新宿店オープン等）が開始されると、ツインは自動的に「新宿プロジェクト記憶」という短期・専用記憶ブロックを脳内に動的確保し、そのプロジェクトのタスク進行やタイムラインに適合したアドバイスを開始します。
3.  **Unified AI Brain 連携**
    *   ツインが未来予測シミュレーションを実行する際、その技術的裏付けやハルシネーションの排除は、Unified AI Brainの「Fact Check Engine（ファクトチェック）」をバックグラウンドで駆動。
    *   「自分の価値観に合致し、かつ現実世界のデータ的にも100%正しい」という、極めて強固で説得力のある戦略シナリオを生成・提示します。
