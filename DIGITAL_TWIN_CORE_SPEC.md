# AI Workspace 「自律デジタルツイン・コア（Digital Twin Core）」機能設計仕様書

本書は、AI Workspaceにおけるユーザー自身（経営者、チームリーダー、スペシャリスト等）の「思考・価値観・専門知識・判断ロジック・意思決定の歴史」を継続的かつ多角的に学習し、本人に極めて近い判断・行動傾向を持つ自律的な代理人**「デジタルツイン（Digital Twin）」**を構築・運行するための核心システム**「Digital Twin Core（デジタルツイン・コア）」**のシステム構成、データスキーマ、学習・記憶構造、意思決定アルゴリズム、および画面設計を定義した製品設計仕様書です。

単なる「チャットボット」や「定型アシスタント」を超え、日々の会話、関与したプロジェクト、作成された成果物、そして人間が行った「意思決定のログ（承認・却下の理由）」をセマンティックに学習。オーナーが不在の間でも、プロジェクトマネージャーやエージェントと自律対話し、優先順位を判断し、リスクを予見し、本人に代わって「1次承認」や「事業改善提案」をデリバリーする、AI Workspace全体の**「自律最高執行代理（Autonomous Executive Proxy）」**の役割を担います。

---

## 1. システム概要 ＆ デジタルツインコンセプト (Overview & Concept)

Digital Twin Coreは、人間（オーナー）の言動から「暗黙知」や「判断基準」を抽出し、構造化された「プロファイル」および「多層記憶」として保持します。これにより、実務の現場（Project Manager, Goal Engine）で発生する複雑な問いに対し、「もしオーナーならどう判断するか」を高精度に予測・代行します。

```
        ===================================================================
                       [ 多層データソースからのリアルタイム学習 ]
        ===================================================================
        ├─ 💬 会話学習     ➔ スラック、チャット、指示出しのトーン、表現、意図
        ├─ 📂 案件・PJ学習  ➔ どのタスクに高い優先度を置き、何を差し戻したか
        ├─ 📦 成果物学習   ➔ 承認した資料、コード、デザイン、レポートの品質基準
        └─ 🎯 意思決定ログ  ➔ 承認/却下時の手動メモ、修正指示のセマンティック抽出
                                        │
                                        ▼
        ===================================================================
                         [ Digital Twin Core Engine ]
        ===================================================================
        ├─ ① プロファイル層 (Profile) : 価値観、興味、専門分野、判断・行動傾向
        ├─ ② 記憶エンジン層 (Memory)  : 短期記憶、長期記憶、案件記憶、経験記憶
        └─ ③ 意思決定予測エンジン (Decision) : 判断予測、優先順位、提案、リスク評価
                                        │
                     ┌──────────────────┴──────────────────┐
                     ▼ (ツインによる自律支援)                  ▼ (人間によるアライメント)
        【🏢 Executive AI / PM 連携】               【👥 人間（オーナー）との対話】
         - プロジェクトの遅延リスク自律評価          - ツイン一致率（精度）の監査
         - ゴールに対する改善策の自律起案            - 思考傾向のチューニング、記憶修正
         - 成果物に対する1次レビューと承認           - 自律代行権限（ガバナンス）の付与
```

## 3. データベース・スキーマ設計 (Data Schema Design)

デジタルツインのプロファイル、多層記憶、会話・成果物からの学習履歴、および意思決定の予測と結果を保持するためのスキーマ。

```
                  +---------------------------+
                  |    digital_twin_cores     |
                  +---------------------------+
                  | id (PK)                   |
                  | company_id (FK)           |
                  | owner_user_id             |
                  | current_concordance_rate  |  -- オーナーとの判断一致率 (0.00〜100.00)
                  | total_memory_nodes        |  -- 記憶総数
                  | auto_proxy_level (INT)    |  -- 自律レベル (0:参照のみ, 1:1次承認, 2:完全代行)
                  | created_at                |
                  | updated_at                |
                  +-------------+-------------+
                                | 1
                                |
         ┌──────────────────────┼──────────────────────┬──────────────────────┐
         | 1..N                 | 1..N                 | 1..N                 | 1..N
  +------v------+        +------v------+        +------v------+        +------v------+
  |  twin_      |        |  twin_      |        |  twin_      |        |  twin_      |
  |  profiles   |        |  memories   |        |  learning_  |        |  decisions  |
  +-------------+        +-------------+        |  logs       |        +-------------+
  | id (PK)     |        | id (PK)     |        +-------------+        | id (PK)     |
  | twin_id (FK)|        | twin_id (FK)|        | id (PK)     |        | twin_id (FK)|
  | dimension   |        | memory_type |        | twin_id (FK)|        | scenario_   |
  |  --'VALUES' |        |  --'SEMANTIC|        | source_type |        |  desc (TEXT)|
  |  --'BIAS'   |        |  --'EPISODIC|        |  --'CHAT'   |        | twin_       |
  |  --'SKILL'  |        |  --'HEURIST'|        |  --'FILE'   |        |  decision   |
  | key_name    |        | content     |        |  --'DECISION|        | owner_      |
  |  (VARCHAR)  |        |  (TEXT)     |        | source_ref_ |        |  decision   |
  | weight_score|        | memory_     |        |  id (UUID)  |        | is_matched  |
  |  (DECIMAL)  |        |  vector     |        | extracted_  |        |  (Boolean)  |
  | payload_    |        |  (VECTOR)   |        |  json (JSON)|        | confidence_ |
  |  json (JSON)|        | last_       |        | created_at  |        |  rate (DEC) |
  | updated_at  |        |  accessed_at|        +-------------+        | created_at  |
  +-------------+        | created_at  |                               +-------------+
                         +-------------+
```

### テーブル定義詳細

#### ① `digital_twin_cores` (ツイン基本マスタ)
デジタルツインの稼働権限、全般的な学習精度、およびガバナンス設定を記録。
*   `id`: UUID (Primary Key)
*   `company_id`: UUID (Foreign Key - 組織スペースのID)
*   `owner_user_id`: UUID (ツインの主人となる人間（オーナー）のユーザーID)
*   `current_concordance_rate`: DECIMAL(5, 2) (オーナーとの判断一致率。0.00〜100.00 %)
*   `total_memory_nodes`: INT (ツインが保持する記憶の合計数)
*   `auto_proxy_level`: INT (自律代理執行権限レベル。`0`: 提案のみ（自律承認不可）、`1`: 1次承認代行（人間へ最終承認は求めるが下準備は完了）、`2`: 完全自動執行（低コスト・低リスクのタスクを全自動で代行・パブリッシュ）)
*   `created_at` / `updated_at`: TIMESTAMP

#### ② `twin_profiles` (ツイン特性定義)
価値観、判断バイアス、興味、専門分野の数値重み、および思考ベクトル。
*   `id`: UUID (Primary Key)
*   `twin_id`: UUID (Foreign Key)
*   `dimension`: VARCHAR(50) (プロファイル次元。`VALUES` (価値観), `BIAS` (判断傾向・バイアス), `SKILL` (専門分野), `INTEREST` (興味・関心))
*   `key_name`: VARCHAR(255) (属性キー。例: `quality_over_cost` (コストより品質優先), `speed_first` (スピード至上主義), `risk_averse` (リスク回避性))
*   `weight_score`: DECIMAL(5, 2) (属性の強さ・重要度。-1.00〜1.00、または 0.00〜1.00)
*   `payload_json`: JSON (追加情報。例: `{"description": "一時的なAPI支出増加を許容する傾向", "examples": ["La Loungeでの開発遅延対策"]}`)
*   `updated_at`: TIMESTAMP

#### ③ `twin_memories` (多層記憶エンジンテーブル)
長期記憶（定常知識）、短期記憶（直近文脈）、および経験記憶（思考パターン、解決策の引き出し）を格納。
*   `id`: UUID (Primary Key)
*   `twin_id`: UUID (Foreign Key)
*   `memory_type`: VARCHAR(50) (記憶の種類。`SEMANTIC` (長期長期: 知識・方針), `EPISODIC` (短期記憶: 直近の出来事・会話), `HEURISTIC` (経験・発見的記憶: 判断の歴史・Heuristics))
*   `content`: TEXT (記憶される具体的な記述・文章)
*   `memory_vector`: VECTOR(1536) (記憶のセマンティック検索用1536次元ベクトル)
*   `last_accessed_at`: TIMESTAMP (この記憶が最後に読み込まれた、または想起された日時。忘却ロジックや優先度算出で使用)
*   `created_at`: TIMESTAMP

#### ④ `twin_learning_logs` (学習パイプライン実行ログ)
会話、成果物、意思決定ログを、いつ、どのような形式で抽出してツインに反映したかの証跡。
*   `id`: UUID (Primary Key)
*   `twin_id`: UUID (Foreign Key)
*   `source_type`: VARCHAR(100) (情報源。`CHAT` (会話ログ), `FILE` (ドキュメント・成果物), `DECISION` (人間の意思決定承認履歴))
*   `source_ref_id`: UUID (情報源のオリジナルテーブルへの参照ID。例: `system_approvals` のID)
*   `extracted_json`: JSON (抽出された思考メタデータ。例: `{"user_sentiment": "Urgently angry", "priority_assigned": "CRITICAL"}`)
*   `created_at`: TIMESTAMP

## 4. 学習構造 ＆ 記憶のライフサイクル設計 (Learning & Memory Engine Architecture)

デジタルツインが「人間の心身のデジタルコピー」として進化するための、4つの学習ソースと多層記憶管理システム。

```
    [ 会話学習 (Chat) ]      [ 案件学習 (Task) ]      [ プロジェクト (PM) ]      [ 成果物 (Files) ]
            │                       │                        │                         │
            ▼                       ▼                        ▼                         ▼
  +─────────────────────────────────────────────────────────────────────────────────────────+
  |                                 学習パイプライン (Pipeline)                              |
  |  - LLMが言動から「価値観」「判断の背景」「専門知識」「優先順位の配分」をメタ抽出          |
  +────────────────────────────────────┬────────────────────────────────────────────────────+
                                       │ (記憶の格納)
                                       ▼
  +─────────────────────────────────────────────────────────────────────────────────────────+
  |                              多層記憶 Vault (Multi-Layer Memory)                        |
  |                                                                                         |
  |  1. 短期記憶 (Episodic Memory)  : 直近24時間の出来事・アクティブタスクのステータス。     |
  |                                  (高頻度アクセス、24h経過後に重要ノードのみ長期記憶へ)   |
  |                                                                                         |
  |  2. 長期記憶 (Semantic Memory)  : 経営理念、美学、デザインルール、知識、人脈マップ。     |
  |                                  (コサイン類似度で想起、生涯保持)                       |
  |                                                                                         |
  |  3. 経験記憶 (Heuristic Memory) : 過去に解決した障害、承認・差し戻したロジックの歴史。   |
  |                                  (「Aという状況 ➔ Bと判断して成功した」というセット)    |
  +─────────────────────────────────────────────────────────────────────────────────────────+
```

### ① 4大インプット学習構造 (The Four Learning Channels)
1.  **会話学習（Episodic / Semantic Extraction）**:
    オーナーとAI、または他のメンバーとのチャットログから、「指示トーン（怒り、安心、急ぎ）」「よく使う言葉」「好む表現」「興味関心のホットワード」を毎日深夜にバッチ処理抽出し、プロファイル特性（`twin_profiles`）を微細アップデート。
2.  **案件・プロジェクト学習（Task Execution Preference）**:
    どのような状況でどのタスクを「自ら作成したか」「他エージェントに割り当てたか」「進捗遅延に対してどう動いたか」という行動ログから、「オーナーの優先順位（Priority）の設定癖」を抽出。
3.  **成果物学習（Qualitative Acceptance Standard）**:
    オーナーが「🟢 合格（本番デプロイやクライアント提出を承認）」としたコード、資料、デザインファイルのメタデータ（例: テストカバレッジ、文量、トーン）を監査。オーナーの「暗黙の品質基準」を、ツインの長期記憶（美学）へ蓄積。
4.  **承認意思決定学習（Core Consent Learning）**:
    人間による承認待ちタスク（`system_approvals`）に対し、人間が「承認」または「却下（修正指示）」をクリックした際に入力した理由（例: `「デザインがポップすぎる。新宿店は高級感重視」`）から、**「判断傾向（If-Thenルール）」**を逆引き抽出して経験記憶へバインド。

### ② 記憶の忘却 ＆ 蒸留ライフサイクル (Memory Distillation Loop)
*   **短期記憶の減衰（Decay）**:
    短期記憶（`Episodic`）は作成後、アクセスされない時間が経過するにつれて「想起スコア」が自動的に線形減衰。
*   **記憶の蒸留（Distillation）**:
    週に一度、AIツインは減衰した短期記憶をスキャン。
    「新宿店のLPで、ロゴの色をゴールドに変更した」といった個別の出来事を抽象化し、「オーナーは新宿店においてゴールドやダークグレーといったシックなカラーコードを強く好む」という**長期記憶（`Semantic`）**、および「デザインが軽い場合 ➔ カラーを重厚に変更して承認する」という**経験ルール（`Heuristic`）**に変換・蒸留（コンパイル）し、不要になった古い短期記憶ノードを安全にパージ（忘却）します。

## 5. 判断構造 ＆ 意思決定予測アルゴリズム (Decision Engine Logic)

デジタルツインが、新しい課題や承認依頼に直面した際、オーナーの思考・判断傾向を精巧にエミュレートするための予測ロジック。

```
                      [ 意思決定が必要な新規タスク ]
                  (例: "LPデザインをNetlifyに即デプロイして良いか")
                                   │
                                   ▼
              [ 1. 思考・価値観コンテキストの想起 (Retrieval) ]
      ├─ データベースの `twin_profiles` (価値観・バイアス) から重み取得
      └─ クエリとのコサイン類似度で `twin_memories` から関連する長期・経験記憶をロード
                                   │
                                   ▼
                 [ 2. 多角的意思決定エミュレータ (Engine) ]
      ├─ 評価関数 A (品質適合) : 成果物はオーナーの「美学長期記憶」に合格しているか
      ├─ 評価関数 B (スピード) : 現在のプロジェクトの遅延リスクに適合しているか
      └─ 評価関数 C (コスト)   : 予算ポリシーとLLMコスト最適化ポリシーに合致しているか
                                   │
                                   ▼
                    [ 3. 意思決定予測 ＆ 確信度算出 ]
      ├─ 各評価関数のスコアを重み付け合算し、「承認（APPROVE）確率」を算出
      ├─ 予測回答を生成、確信度（Confidence Score）を算出
      └─ 自律レベルに応じて「即時自律実行」または「人間へ1次承認として提示」
```

### ① 思考一致率（Concordance Rate / Alignment Score）の計算
ツインの判断 $D_{twin} \in \{-1, 1\}$（非承認、承認）と、オーナーの最終判断 $D_{owner} \in \{-1, 1\}$ の整合性を監査。
$$\text{Concordance Rate} = \frac{\text{過去 } N \text{ 回の代行判断で } D_{twin} = D_{owner} \text{ となった回数}}{N} \times 100$$
一致率が `90%` 以上の領域を「ゴールデンアライメント」と定義し、この基準を超えたツインにのみ、高度な自律代行権（自律レベル2）がガバナンス上付与されます。

### ② 動的優先順位 ＆ リスク自律評価モデル
*   **予測提案**:
    新規タスク発生時、ツインは「プロファイル」の専門知識（`SKILL`）を参照。
    「オーナーなら、このMEO問題に対してこう提案するはずだ」という対策ドキュメントを RAG 経由でコンテキスト生成し、Project Managerへインプット。
*   **リスク自己評価（Twin Risk Evaluator）**:
    ツインは自分自身（またはシステム全体）のエラーや、計画遅延、コスト高騰に対し、「オーナー自身の危機感（ストレス閾値）」をシミュレート。
    「この予算超過ペースだと、オーナーは非常に強いアラートを出す傾向がある」と判断した場合、自律的なリプランを起動し、オーナーへ「緊急事態対策書」を自動発行します。

## 6. 他システム（OS、ツイン、ブレイン、ハブ、グラフ、目標、PM、ストア、チーム）との統合

Digital Twin Coreは、AI Workspace全体の「最高の意思決定中枢（CEO Executive Proxy）」として、全システムの動きをオーナーの意志・ポリシーのコントロール下に統治・同期します。

1.  **AI Company OS / Personal AI OS 連携 (意思決定の代行)**
    *   OS全体の「自律運行」における最高ガバナンスとして稼働。
    *   外部からのメッセージ（LINE、Slack経由の顧客連絡など）に対し、ツインがオーナーのトーンと言葉遣いで「1次返信案」を全自動生成。
2.  **ナレッジグラフ・エンジン（Knowledge Graph Engine）連携 (概念世界の同期)**
    *   ツインの長期記憶、短期記憶、および行動バイアスは、すべてナレッジグラフ上の「特権ノード（Twin Domain）」として表現。
    *   「オーナー山田太郎 ➔ VALUES ➔ スピード ➔ MOTIVATES ➔ NextJS開発タスク」といったセマンティック関係（エッジ）が常にグラフ内に動的に張り巡らされます。
3.  **Goal Management Engine 連携 (人生・ビジネスの羅針盤の同期)**
    *   Goal Engine内の「健康目標」「自由設定目標（オーナーのストレス軽減）」を達成するため、ツインが能動的に実務（タスク、アサイン）を調整。
    *   オーナーの労働時間が週に30時間を超えた場合、ツインが自動的に自動化比率（Automation Rate）を引き上げるポリシーをキックします。
4.  **AI Project Manager 連携 (自律ディレクション)**
    *   Project Manager内の各タスクに対する成果物を、ツインが「オーナーの厳しい目」に代わって1次セマンティック監査（Quality Check）。
    *   「文脈がオーナーの方針とズレている」とツインが感知した場合、エージェントへ差し戻し（Reject）を行い、自律的なデバッグループを実行します。

### ① Federated Multi-Twin Alliance Council (企業間ツインサミット・自律外交合意システム)
*   **自社（自プロジェクト）のデジタルツインと、提携パートナー企業・取引先のデジタルツイン同士が、お互いの人間（オーナー）を介さずに、バックグラウンドの安全な暗号化スレッド上で「超高速交渉・業務スケジュール合意」を自律的に締結する外交システム**:
    *   「La Lounge新宿店オープン」プロジェクトで、自社オーナーツインと、内装施工会社オーナーツインが自律通信。
    *   お互いのプライベートスケジュール、希望価格、および譲れない価値観（品質 vs 納期）を暗号化（ゼロ知識証明を用いたセキュアなマッチング）を介して瞬時に突き合わせ。
    *   人間がチャットで何往復も調整する時間コストを完全ゼロ化し、「お互いのオーナーの価値観を98%満たした、最適な納期・価格の合意契約書」をツイン同士が1秒でドラフト（ツイン外交協議体: Decentralized Twin Council）。
    *   人間（双方のオーナー）は、ツインが作成した「パーフェクトな合意書」に最後にサインするだけでビジネスが完全に成立する、究極のアライアンス自動化インフラへと拡張します。

### ② Multi-Personality Sandboxed Hive (多重人格意思決定シミュレータ)
*   **1つのツインだけでなく、オーナーの「若手時代」「現在の経営者」「競合カフェオーナー」「熱狂的なファン（顧客ペルソナ）」など、様々な『仮想パーソナリティ（多重ツイン）』を個別にシミュレーション空間に召喚し、ビジネスプランの弱点をあらゆる視点からストレステスト・監査するシステム**:
    *   新規事業「新宿 La Lounge」のLPをデプロイする前に、ツインコアが「バーチャル・ボードルーム（仮想取締役会ハイブ）」を開設。
    *   1. オーナーツイン（経営者目線）
    *   2. 競合店舗オーナーツイン（ライバル目線でのネガティブ攻撃想定）
    *   3. 20代新宿で働く女性ツイン（コアターゲット層のリアルな感情・口コミ予測）
    *   4. 財務専門ツイン（極限のコストカット目線）
    *   この4つの仮想ツインが、LP、メニュー、および集客計画に対して同時並行で激しい自律ディベートを展開。
    *   「ライバル目線だと、この価格設定は即座に対抗値下げキャンペーンを打たれる隙がある」「ターゲット女性目線だと、内装のゴールドは高級すぎて入りづらい」といった、極めてリアルで客観的なバグやリスクをリリース前に 100% 炙り出す（Hive Swarm Brainstorming）。
