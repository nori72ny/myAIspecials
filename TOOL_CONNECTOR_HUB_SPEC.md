# AI Workspace 「ツール・コネクター・ハブ（Tool Connector Hub）」機能設計仕様書

本書は、AI Workspace内のすべてのAIエージェント、ワークフロー、およびオートメーションが、外部のクラウドサービスやSaaS、各種開発・マーケティングプラットフォームと安全、柔軟、かつシームレスにデータ同期・機能実行を行うための統合インフラ**「Tool Connector Hub（ツール・コネクター・ハブ）」**のシステム構成、データスキーマ、認証・認可アーキテクチャ、セキュリティ設計、および画面レイアウトを定義した設計仕様書です。

本ハブは、各API認証情報（OAuthトークン、APIキー、WebHook秘密鍵）のセキュアな一元管理、全自動のアクセストークン更新（Token Rotation）、詳細な実行履歴ログの監査、およびエージェント/ワークフロー単位での厳格な権限管理（粒度の細かいポリシー制御）を実現する、AI Workspaceにおける「神経網」の役割を担います。

---

## 1. システム概要 ＆ 接続コンセプト (Overview & Architecture Concept)

Tool Connector Hubは、人間（組織の管理者や各ユーザー）が一度認証した外部コネクタへの「接続チャネル」をプーリングし、全エージェントや全ワークフローが透過的に呼び出せる共通サービスレイヤーを提供します。

```
      +-------------------------------------------------------------------------+
      |                           [ 人間（管理者・一般ユーザー） ]               |
      |   - 外部サービス（Google, Microsoft, GitHub 等）の認証実行 (OAuth/API Key) |
      +------------------------------------┬------------------------------------+
                                           │ (1. コネクタ接続・認可の実行)
                                           ▼
      +-------------------------------------------------------------------------+
      |                        [ Tool Connector Hub ]                           |
      |  ├─ 1. 認証マネージャ (Credentials Vault - AES-256 暗号化保存)            |
      |  ├─ 2. トークンローテーター (OAuth Refresh Token 自動更新エンジン)        |
      |  ├─ 3. 権限・ポリシーチェッカー (エージェントごとの実行許可範囲の確認)      |
      |  └─ 4. ダッシュボード ＆ 監査ログ (利用回数、失敗履歴、接続ステータス)     |
      +------------------------------------┬------------------------------------+
                                           │ (2. セキュアな実行チャネルの開放)
                                           ├────────────────────────────────────┐
                                           ▼                                    ▼
      +-------------------------------------------------+  +------------------------------------+
      |             [ AI エージェント群 ]               |  |     [ ワークフロー・オートメーション ] |
      | (すべてのAI社員が役割プロンプトに応じツール利用)  |  | (定時実行、イベントトリガーによる自動化) |
      +------------------------------------+------------+  +--------------------+---------------+
                                           │                                    │
                                           └─────────────────┬──────────────────┘
                                                             │ (3. 共通APIゲートウェイ経由でコール)
                                                             ▼
                                      ==================================
                                         外部連携プラットフォーム (SaaS API)
                                      ==================================
                                       [Google] [Microsoft] [Development]
                                       [Marketing] [Business Solutions]
```

## 3. データベース・スキーマ設計 (Data Schema Design)

コネクタの資格情報、認可スコープ、利用制限ポリシー、実行履歴ログ、およびトークン更新管理情報を効率よくセキュアに保持するデータモデル。

```
                  +---------------------------+
                  |     connector_registry    |
                  +---------------------------+
                  | id (PK)                   |
                  | company_id (FK)           |
                  | service_provider (VARCHAR)|  -- 'GOOGLE', 'SLACK', 'GITHUB', etc.
                  | connection_name (VARCHAR) |  -- 例: '社内Slack主幹チャネル'
                  | auth_type (VARCHAR)       |  -- 'OAUTH2', 'API_KEY', 'WEBHOOK'
                  | credentials_vault_id      |  -- 暗号化データ保存への参照
                  | is_active (Boolean)       |
                  | scopes_granted (JSON)     |  -- 許可されたスコープ一覧
                  | token_expires_at          |
                  | created_at                |
                  | updated_at                |
                  +-------------+-------------+
                                | 1
                                |
                                ├─────────────────────────────┬─────────────────────────────┐
                                | 1..N                        | 1..N                        | 1..N
                  +-------------v-------------+   +-----------v-------------+   +-----------v-------------+
                  | connector_access_policies |   |    connector_usage_logs |   | connector_token_rotation|
                  +---------------------------+   +-------------------------+   +-------------------------+
                  | id (PK)                   |   | id (PK)                 |   | id (PK)                 |
                  | connector_id (FK)         |   | connector_id (FK)       |   | connector_id (FK)       |
                  | target_type (VARCHAR)     |   | caller_agent_id (FK)    |   | action_type (VARCHAR)   |  -- 'ROTATE', 'REVOKE'
                  | target_id (UUID/Null)     |   | caller_workflow_id (FK) |   | rotation_status         |  -- 'SUCCESS', 'FAILED'
                  | permission_level (VARCHAR)|   | api_endpoint_called     |   | error_message (TEXT)    |
                  | restriction_rules (JSON)  |   | http_method (VARCHAR)   |   | performed_at            |
                  | created_at                |   | response_status (INT)   |   +-------------------------+
                  +---------------------------+   | error_payload (TEXT)    |
                                                  | executed_at             |
                                                  +-------------------------+
```

### テーブル定義詳細

#### ① `connector_registry` (ツールコネクタ登録マスタ)
*   `id`: UUID (Primary Key)
*   `company_id`: UUID (Foreign Key - 所属する企業またはワークスペーススペースのID)
*   `service_provider`: VARCHAR(100) (サービス名。例: `GOOGLE_SHEETS`, `SLACK`, `GITHUB`, `NOTION`)
*   `connection_name`: VARCHAR(255) (接続の表示名。例: `営業部リード追跡スプレッドシート`)
*   `auth_type`: VARCHAR(50) (認証方式。`OAUTH2`, `API_KEY`, `WEBHOOK_TOKEN`, `BASIC`)
*   `credentials_vault_id`: VARCHAR(255) (セキュリティで保護された外部Vault（KMS、Google Secret Manager等）に格納されている認証情報キーへの参照ID)
*   `is_active`: BOOLEAN (コネクタが現在有効かつ使用可能か)
*   `scopes_granted`: JSON (実際にSaaS側から付与されている認可スコープの配列。例: `["files.metadata.read", "files.content.write"]`)
*   `token_expires_at`: TIMESTAMP (OAuthアクセストークンの有効期限。期限が近い場合、後述のトークンローテーターが自動更新)
*   `created_at` / `updated_at`: TIMESTAMP

#### ② `connector_access_policies` (詳細権限・アクセス制限ポリシー)
どのエージェント、どのワークフローにコネクタの使用を許すかを定義するセキュリティマッピング。
*   `id`: UUID (Primary Key)
*   `connector_id`: UUID (Foreign Key)
*   `target_type`: VARCHAR(50) (制御の対象種別。`AGENT` (個別AI社員), `WORKFLOW` (自動化フロー), `ROLE` (ロールベース（例: 開発者ロール全体）), `ALL` (全公開))
*   `target_id`: UUID (Foreign Key, Nullable - 対象となるエージェントIDやワークフローID)
*   `permission_level`: VARCHAR(50) (権限レベル。`READ_ONLY` (読み取り専用), `WRITE_ONLY` (書き込み専用), `FULL_ACCESS` (フルアクセス), `ADMIN_ONLY` (設定変更まで許可))
*   `restriction_rules`: JSON (詳細な実行制約条件。例: `{"allowed_slack_channels": ["#general"], "max_file_size_mb": 10}`)

#### ③ `connector_usage_logs` (コネクタAPI実行・監査履歴)
エージェントやワークフローが外部APIを実行したすべての履歴。セキュリティ監査やデバッグに必須。
*   `id`: UUID (Primary Key)
*   `connector_id`: UUID (Foreign Key)
*   `caller_agent_id`: UUID (Foreign Key, Nullable - 実行を指示したエージェントのID)
*   `caller_workflow_id`: UUID (Foreign Key, Nullable - 実行をトリガーしたワークフローのID)
*   `api_endpoint_called`: VARCHAR(1024) (実行されたAPIエンドポイント。例: `/v2/pages`)
*   `http_method`: VARCHAR(10) (例: `POST`, `GET`, `PATCH`)
*   `response_status`: INT (HTTPレスポンスコード。例: `200`, `401`, `429`)
*   `error_payload`: TEXT (APIエラー発生時の生のエラーメッセージ、スタックトレース)
*   `executed_at`: TIMESTAMP

#### ④ `connector_token_rotation` (認証トークンローテーション履歴)
OAuth2のRefreshTokenを使用した自動トークンリフレッシュの監査証跡。
*   `id`: UUID (Primary Key)
*   `connector_id`: UUID (Foreign Key)
*   `action_type`: VARCHAR(50) (処理内容。`ROTATE` (更新), `REVOKE` (剥奪))
*   `rotation_status`: VARCHAR(50) (処理結果。`SUCCESS`, `FAILED`)
*   `error_message`: TEXT (トークン更新に失敗した場合の理由。例: `Refresh token has expired or revoked`)
*   `performed_at`: TIMESTAMP

## 4. 認証設計 ＆ セキュリティ設計 (Authentication & Security Vault Design)

Tool Connector Hubは、外部サービスへの「鍵」を握る最重要インフラであるため、厳格な認証・暗号化セキュリティプロトコルを採用します。

### ① トークン・シークレットの二重暗号化保存（Credentials Vault）
*   **データの局所化と暗号化**:
    *   データベース（`connector_registry`）には、生のアクセストークンやシークレットキー、リフレッシュトークンを直接保存しません。
    *   認証情報は、システムが管理する隔離された「シークレット・ストレージ（例: Google Secret Manager / HashiCorp Vault）」に保存。または、アプリケーションの暗号化キー（**Envelope Encryption**方式：データ暗号化キー(DEK)をマスター暗号化キー(MEK)で包む方式）を使い、データベース内でAES-256を用いて暗号化したバイト列として保管します。

### ② 自動アクセストークン更新（Token Rotation）
*   **シームレスな自動更新プロセス**:
    1.  エージェントまたはワークフローがAPIを実行する際、Connector Hubは保存されている `token_expires_at` を確認。
    2.  有効期限が残り5分未満の場合、Hubは即時リクエストを保留。
    3.  バックグラウンドで、暗号化保存された `refresh_token` を復号し、対象SaaS（例：Google Auth、GitHub API）のOAuthトークン更新エンドポイントにリフレッシュリクエストを自動送信。
    4.  新しいアクセストークンおよびリフレッシュトークン（※トークン回転に対応したワンタイムリフレッシュの場合）を受け取り、安全に再暗号化して保存。
    5.  保留していたエージェントのAPIリクエストに「最新の有効なトークン」を載せて実行。この一連の作業はAIエージェントやユーザーには完全に不透明（隠蔽）で行われます。

## 5. 権限設計 ＆ 実行時ガバナンス (Fine-Grained Permission Design)

AIエージェントが自律行動する中での予期せぬ「外部APIの暴走」や「不要なデータの書き込み」を防ぐため、Connector Hubは強力な「ゲートウェイフィルター」として稼働します。

```
[ AIエージェント/自動化フロー ] ➔ 外部へのリクエスト
                                       │
                                       ▼
                     +-----------------------------------+
                     |    Connector Gateway Validator    |
                     +-----------------------------------+
                     | [Check 1] 認証は有効か？           | ➔ 無効時はエラー・自動更新
                     | [Check 2] 呼び出し元の権限はあるか？| ➔ ポリシー違反時は遮断・警告
                     | [Check 3] パラメータ・内容の検証    | ➔ 禁止動作（例: Slack全通知など）はサニタイズ
                     +-----------------┬-----------------+
                                       │ (検証合格)
                                       ▼
                              SaaS APIの実行送信
```

1.  **最小特権の原則 (POLP: Principle of Least Privilege)**:
    *   エージェントが使用するコネクタは、可能な限り制限的なスコープ（例: Slackは「指定チャンネルへの書き込み専用（`incoming-webhook`）」、GitHubは「特定リポジトリのReadのみ」）で認可することをシステムが推奨・案内します。
2.  **APIパラメータの動的監査・フィルタリング**:
    *   **Gmail / Outlook連携**: エージェントがメール送信APIを実行する際、送信元のアドレスが「社内の制限されたアカウント」から逸脱していないか、および本文中に特定の機密情報（クレジット番号や個人情報など、正規表現で検知）が含まれていないかをフィルタリングし、自動サニタイズ、または人間の承認ステップを差し込みます。
    *   **Slack / Discord連携**: エージェントに割り当てられたシステムルールにより、「@here」や「@channel」といった大規模メンションの送信をGatewayで強制遮断します。
3.  **テナント・組織レベルのガードレール (Tenant-Level Guardrails)**:
    *   組織のシステム管理者は、特定のコネクタ（例：全社共有のGitHubリポジトリ）の利用について、開発部のエージェントへの配備を許可する一方で、マーケティング部やカスタマーサポートのエージェントへの共有配備をグローバル設定から一括禁止にできます。

## 6. 他システム（OS、エージェント、自動化、ブレイン）との統合

Tool Connector Hubは、AI Workspace全体に安全な「ツールキット」を供給し、シームレスな業務執行を支援します。

1.  **AI Agent & Workflow Engine 連携 (動的ツールインジェクション)**
    *   エージェントがタスクを実行する際、そのエージェントのプロンプト内に「Tool Connector Hubに接続されているツールの名前と、使用可能なAPIのスキーマ（OpenAPI定義形式）」がシステムによって自動的にインジェクションされます。
    *   エージェントは自律的に「現在、SlackコネクタとGoogleスプレッドシートコネクタが利用可能である」ことを認識し、自身の脳（LLM）から「ツール実行指示（Tool Call）」を発行。Connector Hubを介して実際にデータを読み書きします。
2.  **Automation ＆ Scheduler 連携 (定時トリガーの自動実行)**
    *   「毎日朝9時にGoogle Analyticsから昨日のアクセスデータを取得し、社内Slackにレポートを自動投稿する」といった定期実行オートメーション。
    *   スケジューラーが時間到来を検知すると、GA4コネクタとSlackコネクタの認証情報を安全に取得し、ワークフローエンジンに流し込んでタスクを実行します。
3.  **Unified AI Brain 連携 (検索グラウンディングと情報補強)**
    *   Unified AI Brainが「最新の自社売上トレンドについて教えて」と聞かれた際、ナレッジベースだけでなく、Tool Connector Hubを通じて直接GoogleスプレッドシートやNotion、Salesforceのデータを動的に検索・ロード。常に「本物かつ最新のデータ」に基づいた高度な推論を導出（Grounding）します。

## 7. 将来拡張設計 (Future Extensibility Plan)

### ① Community Tool Registry（エコシステム・カスタムプラグインストア）
*   **サードパーティ開発者が自作のカスタムコネクタを登録・公開できるストア**:
    *   現在はシステム標準の24サービスのみですが、将来的にAPI仕様（Swagger/OpenAPI形式）とOAuth接続パラメータを設定するだけで、誰でも「カスタムコネクタ（例: 自社製ERP、特定の業界特化SaaS）」をノーコードで作成し、ハブに登録できるフレームワークを構築します。
    *   優れたカスタムコネクタを「プラグインマーケット」を通じて配布・販売（エコシステム化）し、あらゆる業務ソフトとAI Workspaceがノーコードで直結する環境を拡張します。

### ② AI-to-AI Connection Protocol（AI間自律インターフェース調整）
*   **接続先API仕様が変わっても、AIがAPI仕様書を自律学習し、コード変更なしで配線を自己修復する仕組み**:
    *   外部サービス（例：SlackやClickUp）のAPIアップデートにより、エンドポイント名や必要なパラメータ構成が変更され、既存の自動化システムが破損（400 / 404エラーを検知）した場合。
    *   Connector Hubのエラー監視AIがエラーを捕捉し、インターネット経由で最新の「外部サービスAPI仕様書」を自動収集（Web Search Grounding）。
    *   API仕様の違いを自律解析し、接続スキーマを自動で書き換えて接続を維持（自己回復・Self-Healing API Pipeline）する、究極のメンテナンスフリー・インフラへと拡張します。
