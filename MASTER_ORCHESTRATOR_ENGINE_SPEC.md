# AI Workspace 「超統合協調生成・マスタオーケストレーター（Master Orchestrator Engine）」機能設計仕様書

本書は、AI Workspaceにおけるあらゆる創造物（文章、アプリケーションコード、ウェブサイト、社内資料、プロモーション画像、解説動画、提案プレゼン、大規模システムなど）の制作依頼に対し、AIがその「制作物要件（Creation Analysis）」を自律解析し、最適な「複数AIモデル（Gemini, OpenAI, Claude, Perplexity, DeepSeek, Grok, Flux等）」「自律エージェント群」「自動ワークフロー」を動的に召喚・統治。それぞれに役割を与えて並列・協調制作を行わせ、AI同士による厳格な多重クロスレビュー（自己採点・品質評価）および「6大ペルソナ（経営者、営業、開発、デザイン、マーケティング、ユーザー）視点」での最終評価を経て、欠点ゼロの究極の成果物（Golden Answer / Golden Creation）を自律生成する、システム全体の**「自律最高デジタル創造・製品責任者（Autonomous CDO / Chief Product Officer Engine）」**のシステム構成、データスキーマ、共同制作フュージョンアーキテクチャ、品質保証（QA）ワークフロー、および画面レイアウトを定義した製品設計仕様書です。

## 1. システム概要 ＆ マスタオーケストレーションコンセプト (Overview & Concept)

Master Orchestrator Engineは、下図のようにユーザーからの多大な創作要請を「受託」すると、単一のAIに丸投げするのではなく、複数の「専門AIエージェント（ディレクター、ライター、コーダー、デザイナー、QA監査等）」による**「自律バーチャルスタジオ（Virtual Swarm Studio）」**を瞬時に構築。
各AIが作成した中間成果物を「フュージョン・エンジン」がセマンティックにマージし、人間（経営者、ユーザー）や各種ビジネス視点の「多角シミュレーション」で磨き上げることで、最高峰の「品質保証された成果物」を出力します。

```
     [ 👥 ユーザーの創作要請 (文章, コード, アプリ, プレゼン, 画像, 動画, システム) ]
                               │
                               ▼
     ===================================================================================
                            [ Master Orchestrator Engine ]
     ===================================================================================
     ├─ ① 自律インテリジェント・ルーティング (AI Routing)
     │    - 制作物解析・質問解析、最適なAI（Gemini/OpenAI/Claude等）、Agent、Workflowの動的アサイン
     ├─ ② 共同創造フュージョンコア (Creation Fusion Pipeline)
     │    - 各AI（文章、コード、画像、UI）の非同期並列制作、セマンティック・マージ（統合マージ）
     ├─ ③ AI相互品質監査ループ (AI Quality Improvement)
     │    - AI同士のクロスレビュー、改善提案の自動フィードバック、文章/コード/UIの動的自己リファクタ
     ├─ ④ 自律最高回答・完成版生成器 (Golden Answer / Golden Creation Engine)
     │    - 複数制作案の動的生成、多次元マトリクス比較、自己改善ループの反復（最低品質保証）
     ├─ ⑤ 認知自己採点 ✕ 6大ビジネス視点フィルター (Perspective Review Panel)
     │    - [経営者視点]  : ROI、コスト、将来性、アライン
     │    - [営業視点]    : 売りやすさ、訴求力、リード獲得
     │    - [開発視点]    : 保守性、クリーンコード、堅牢性
     │    - [デザイン視点]: UI/UX美、タイポグラフィ、余白・トーン
     │    - [マーケ視点]  : MEO/SEO適合、コピーライティング、CTR
     │    - [ユーザー視点]: 使いやすさ、アクセシビリティ、読みやすさ
     └─ ⑥ 物理パブリッシュ & 改善伝播 (Publish & Deployment Layer)
                         │
                     ┌───┴──────────────────────┐
                     ▼ (完全無欠の成果物)               ▼ (継続的な自己進化)
        【💾 Golden Answer / Golden Creation】   【🛠️ Self Improvement / Project Manager】
         - 欠陥ゼロのコード、極上UIアプリ、          - プロジェクトカンバンの自動完了・配備
           完璧な財務プレゼン、超高解像度画像          - 過去の採用履歴に基づくツイン認知学習
```

### テーブル定義詳細

#### ① `orchestrator_projects` (制作プロジェクト統合マスタ)
どのような種類の成果物を、どのプロセスで共同制作し、最終的にどの案が「Golden Creation（完成版）」として採択されたかを統括管理。
*   `id`: UUID (Primary Key)
*   `company_id`: UUID (Foreign Key - 組織スペースのID)
*   `title`: VARCHAR(255) (プロジェクト名。例: `「新宿店LP ✕ MEO改善アプリ共同制作」`)
*   `creation_type`: VARCHAR(50) (制作区分。`CODE` (コード), `WEB_SITE` (Webサイト), `PRESENTATION` (プレゼン資料), `APP` (フルアプリ), `IMAGE` (画像アセット), `VIDEO` (解説動画), `DOCUMENT` (長文ドキュメント), `SYSTEM` (統合システムシステム))
*   `status`: VARCHAR(50) (ステータス。`ROUTING` (AIアサイン中), `CREATING` (共同執筆・制作中), `REVIEWING` (相互品質監査中), `GOLDEN_READY` (完成・パブリッシュ可能), `PUBLISHED` (現実世界へマウント・公開済み))
*   `best_candidate_id`: UUID (Foreign Key, Nullable - 厳重に選定された完成版 `orchestrator_candidates` のID)
*   `average_quality_score`: DECIMAL(5, 2) (完成版に対して6大視点レビューが下した総合平均点。0.00〜100.00点)
*   `created_at` / `updated_at`: TIMESTAMP

#### ② `orchestrator_teams` (召喚された自律エージェントチーム)
プロジェクトごとに動的に割り当てられた、役割の異なる複数のAIモデルの配備ログ。
*   `id`: UUID (Primary Key)
*   `project_id`: UUID (Foreign Key)
*   `ai_model`: VARCHAR(100) (割り振られたAIモデル名。例: `GEMINI_1_5_PRO`, `CLAUDE_3_5_SONNET`, `FLUX_3`)
*   `role`: VARCHAR(50) (役割。`DIRECTOR` (進行管理監), `WRITER` (文章執筆), `CODER` (ロジック実装), `DESIGNER` (UI/UXビジュアル), `QA_AUDITOR` (バグ・品質監査))
*   `status`: VARCHAR(50) (状態。`ASSIGNED`, `WORKING`, `COMPLETED`, `FAILED`)
*   `created_at`: TIMESTAMP

#### ③ `orchestrator_candidates` (制作案・中間成果物テーブル)
AIチームによって記述・マージされた、世代別の成果物ソースコード、JSON、画像URLなどの不変記録。
*   `id`: UUID (Primary Key)
*   `project_id`: UUID (Foreign Key)
*   `version`: INT (世代・サイクル数。例: `3` は第3回改善サイクルの成果物を示す)
*   `raw_content`: TEXT (成果物の本体。HTML/TypeScriptコード、Markdown文章など)
*   `assets_json`: JSON (マウントされた副次的アセット情報。画像URL配列、動画URL、CSS変数等)
*   `is_golden`: BOOLEAN (これが完成版（Golden Creation）として採択されたかどうか)
*   `created_at`: TIMESTAMP

#### ④ `orchestrator_dialogues` (AI同士の品質改善クロスレビュー対話)
制作中にAIエージェント間で交わされた、バグ指摘、コード修正要望、およびそれに対する合意リファクタのログ。
*   `id`: UUID (Primary Key)
*   `project_id`: UUID (Foreign Key)
*   `sender_ai`: VARCHAR(100) (発言したエージェントモデル。例: `CLAUDE_3_5_SONNET (DESIGNER)`)
*   `receiver_ai`: VARCHAR(100) (指摘先の対象エージェントモデル。例: `GPT_5 (CODER)`)
*   `message`: TEXT (具体的な指摘または修正確認内容。例: `「ボタンにホバーエフェクト（transition）を追加してください」`)
*   `created_at`: TIMESTAMP

#### ⑤ `orchestrator_reviews` (6大ビジネス視点品質評価テーブル)
特定の生成案に対し、6つの専門ペルソナ（経営者、営業、開発、デザイン、マーケ、ユーザー）が下した個別スコアと批判的分析文章。
*   `id`: UUID (Primary Key)
*   `candidate_id`: UUID (Foreign Key)
*   `perspective`: VARCHAR(50) (ペルソナ視点。`CEO` (経営者・投資対効果), `SALES` (営業・コンバージョン), `DEV` (開発・コード品質/堅牢性), `DESIGN` (デザイン・UIUX/視覚美), `MARKETING` (マーケティング・SEO/AIO), `USER` (エンドユーザー・使いやすさ))
*   `score`: DECIMAL(5, 2) (100点満点の評価点。AIが各ペルソナのバイアスで客観算定。85点以上が最低品質の合格ライン)
*   `critique_text`: TEXT (ペルソナごとの詳細な品質講評、および未達部分の改善要望文章)
*   `created_at`: TIMESTAMP

#### ⑥ `orchestrator_publishes` (パブリッシュ物理伝播ログ)
完成した「Golden Creation」を、Reality Engineや外部デプロイプラットフォームにどのような構成で本番公開（デプロイ）したかの監査。
*   `id`: UUID (Primary Key)
*   `project_id`: UUID (Foreign Key)
*   `target_platform`: VARCHAR(100) (パブリッシュ先。例: `Vercel`, `Cloud_Run`, `Shopify`, `GBP_Posts`)
*   `credentials_json`: JSON (デプロイ時に使用した暗号化セキュリティメタデータインデックス)
*   `executed_at`: TIMESTAMP (パブリッシュが開始された日時)

## 4. 共同制作 ＆ 品質保証・自己採点ロジック (Fusion & Quality Logic)

Master Orchestrator Engineが、ハルシネーションや質の低い妥協を完全に許さず、複数のAIを束ねて最高傑作を導き出す4段階の自律生成プロセス。

```
     [ 1. 動的 AI Swarm フォーメーション (Swarm Routing) ]
       - ユーザーのクリエイション要請をセマンティック解析。
       - 「アプリ制作（コード＋UI）」の場合、自動的に『Gemini 1.5 Pro (Director)』をリーダーとし、
         『Claude 3.5 (Designer)』『GPT-5 (Coder)』『DeepSeek R1 (QA)』による開発 Swarm を編成。
                            │
                            ▼
     [ 2. 協調共同制作 ✕ セマンティック・フュージョン (Semantic Fusion) ]
       - 各エージェントに詳細なシステム要件（SLA）と、独立した開発コンテキストを配備。
       - 各自が生成した部分コード（CSS、ビジネスロジック、データベース定義）を、
         Director LLM がインポート整合性を担保しつつ、1つのコンパイル可能なファイルにセマンティック・マージ。
                            │
                            ▼
     [ 3. 多重 AI クロスレビュー ✕ 品質改善ループ (Critique Loop) ]
       - マージされた成果物に対して、QAエージェント（DeepSeek R1）がコードを実行・静的解析。
       - エラーやUI欠陥、非効率なロジックを発見した際、他の担当AIへ「名指しでリファクタ」を要求。

### ① 6大ビジネス視点の数理評価モデル (Multi-Perspective Evaluation)
成果物 $C$ に対し、各ビジネス視点 $p$ が下すスコア $S_p(C) \in [0, 100]$ は、各専門ペルソナプロファイル（評価関数） $f_p$ によって厳格に算定されます。
$$Score_{golden}(C) = \sum_{p=1}^{6} W_p \times S_p(C)$$
*   **経営者視点 (CEO)**: $S_{ceo}$ ➔ 投資効果（期待ROI、開発のシンプルさ、不要ライセンス排除、隠れたコストの低さ）
*   **開発者視点 (DEV)**: $S_{dev}$ ➔ ソフトウェア・エンジニアリング原則（結合度の低さ、バグ未検出率、変数命名規則、セキュリティ脆弱性ゼロ、テスト網羅度）
*   **デザイン視点 (DESIGN)**: $S_{design}$ ➔ UI/UX原則（余白の十分さ、視覚的階層の明確さ、アクセシビリティ4.5:1コントラスト適合度、一貫した配色ルール）
*   **営業・マーケティング視点 (SALES/MARKET)**: $S_{sales}$ ➔ 成約弾力性（直感的な価値訴求、コンバージョンフックの存在、MEO/SEO用メタタグ適合度、読み込み速度）
*   システムは、どれほど優れたコード（$S_{dev} = 100$）であっても、デザインや使いやすさが破綻（$S_{design} = 30$）している場合、全体の $Score_{golden}$ を著しく減点し、全方位に配慮された真の傑作のみをゲート通過させます。

### ② 最低品質保証の強制改善ルール (SLA Refinement Rule)
1つでも 80点 を下回る評価次元があった場合、または総合 Golden Score が 90点 を下回る場合は、**「最低品質保証契約（SLA: Service Level Agreement）の違反」**と自律判定され、Directorは自動的にその評価内容と具体的指摘をフィードバックし、担当AIたちに「第 $N+1$ 世代の改善バッチ」を実行させます（人間の手を1秒も煩わせることなく、100%の自律品質チェックを回しきります）。
