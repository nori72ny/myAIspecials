# AI Workspace 「統合自律 AI データレイク（AI Data Lake）」機能設計仕様書

本書は、AI Workspace内のあらゆるシステムや外部連携（Connector Hub）から生み出されるすべてのデータ（MEO, SEO, AIO, 営業, 顧客, Projects, Knowledge, Memory, Agents, Workflow, Meeting, Apps, Files, Reports, AI Logs）をリアルタイムに集約・一元管理し、セマンティック（意味的）解析、多次元相関分析、自然言語検索、およびAI自律発見（AI Discovery）を可能にする次世代インテリジェントデータ統合基盤**「AI Data Lake（AIデータレイク）」**のシステム構成、データスキーマ、検索・インデックスアーキテクチャ、相関分析・推論パイプライン、および画面レイアウトを定義した製品設計仕様書です。

本システムは、単なる静的なデータ保管庫（Object Storage）に留まらず、すべての非構造化データと構造化データをセマンティックにマージし、AIが「成功・失敗パターン」や「異常・改善の予兆」を自律的に発見・解釈する、Workspace全体の**「データ中枢脳・自律解析エンジン（Autonomous Data Lake & Discovery Engine）」**の役割を果たします。

---

## 1. システム概要 ＆ データレイクコンセプト (Overview & Concept)

AI Data Lakeは、下図のように15種類の異なる性質のデータを「インジェスト・パイプライン」を介して吸収し、メタデータの抽出、チャンク分割、およびベクトル化（Embedding）を瞬時に実行します。これにより、従来のRDBや単一の検索エンジンでは実現できなかった「多次元での相関分析」や「コンテキスト（文脈）を跨いだ高精度な意味検索・Discovery」を可能にします。

```
        ===================================================================================
                       [ 📥 15種類の統合インジェスト・データソース (Stored Data) ]
        ===================================================================================
        ├─ 🌐 マーケティング  : MEO順位, SEO流入, AIO引用率, 広告ROI, SNS動向
        ├─ 💰 ビジネス・顧客  : 営業案件（42件）, 顧客プロファイル, 商談履歴
        ├─ ⚙️ プロジェクト    : AI Project Manager タスク、カンバン、マイルストーン
        ├─ 🧠 ナレッジ＆メモリ: 知識グラフ（Knowledge Graph）、長期記憶（Memory）、ミーティング議事録（Meeting）
        ├─ 🤖 エージェント活動: 稼働エージェント（27体）、実行ワークフローログ、AI生成アプリ（16件）、各種レポート、AI Logs
        └─ 📁 実ファイル情報  : Drive等から同期した生ドキュメント（Files）、各種財務・経営Reports
                                                │
                                                ▼ [リアルタイム・ストリーミング・パーサー]
        ===================================================================================
                                      [ AI Data Lake ]
        ===================================================================================
        ├─ 📁 3層データ構造 (Storage Layer) : Raw Data (不変ログ), Structured Data (マート), Semantic Data (ベクトル)
        ├─ 🔍 多元インデックス (Multi-Index): 全文 (BM25) ✕ 意味検索 (Vector) ✕ 自然言語 ✕ タグ ✕ 期間
        ├─ 📜 統合タイムライン (Timeline Engine): 案件・営業・AI・改善の全歴史をシームレスに一本化
        ├─ 📊 因果相関分析器 (Correlation): 順位、売上、営業、口コミ、広告、SEO、MEO、AIOの動的相関係数算定
        └─ 💡 自律発見脳 (AI Discovery) : 成功/失敗パターン、システム/経営異常、改善候補の自律ドキュメント化
                                                │
                     ┌──────────────────────────┴──────────────────────────┐
                     ▼ (経営とリプランの最終合意)                               ▼ (実行・最適化)
        【📊 AI BI Center / Executive】                          【🔌 Reality Engine / Self Improvement】
         - 未来予測への相関特徴量（因果モデル）の提供               - 発見された失敗パターンに基づく自動バグ修復
         - RAG経営インサイトのための厳格な根拠データ供給             - 異常検知時における自律的なフォールバック
```

### テーブル定義詳細

#### ① `data_lake_stores` (データレイク総合管理マスタ)
データレイク全体の規模、ノード数、接続されている15種類のアクティブソースの状態とハイドレーション実績をキャッシュ管理。
*   `id`: UUID (Primary Key)
*   `company_id`: UUID (Foreign Key - 組織スペースのID)
*   `total_ingested_size`: DECIMAL(20, 2) (データレイクに蓄積された全実ファイルのバイト総数。1.25 TB)
*   `total_nodes_count`: INT (パースされデータベースに登録された全構造化・非構造化ノードの総数。142,500 ノード)
*   `active_sources_json`: JSON (インジェストが有効化されているソースリストとステータス。`{"MEO": "ACTIVE", "SEO": "ACTIVE", "STRIPE": "ACTIVE", "GITHUB": "ACTIVE", "AI_LOGS": "ACTIVE", ...}`)
*   `created_at` / `updated_at`: TIMESTAMP

#### ② `data_lake_nodes` (データレイク・コアノードテーブル)
15種のデータソースからパースされた汎用的な構造化メタデータノード。JSONスキーマレス設計であり、あらゆる種類の属性と元のコンテンツデータをフラットに格納。
*   `id`: UUID (Primary Key)
*   `store_id`: UUID (Foreign Key)
*   `source_type`: VARCHAR(50) (データソース種別。`MEO`, `SEO`, `AIO`, `SALES` (営業), `CUSTOMER` (顧客), `PROJECTS`, `KNOWLEDGE`, `MEMORY`, `AGENTS`, `WORKFLOW`, `MEETING`, `APPS`, `FILES`, `REPORTS`, `AI_LOG`)
*   `payload_json`: JSON (パースされたセマンティック本体。例: 会議議事録の場合 `{"title": "MEO会議", "transcript": "...", "action_items": [...]}`)
*   `tags_json`: JSON (自動/手動で付与された分類タグの配列。`["#meo", "#marketing", "#meeting"]`)
*   `ingested_at`: TIMESTAMP (データレイクがデータを取り込んだ日時)
*   `created_at`: TIMESTAMP

#### ③ `data_lake_timelines` (データレイク統合タイムラインテーブル)
「営業、案件、AI、改善」の4つの重要なビジネスイベント歴史を時間軸に沿って一本化し、時系列スクロール追跡を高速化するためのインデックス。
*   `id`: UUID (Primary Key)
*   `store_id`: UUID (Foreign Key)
*   `event_type`: VARCHAR(50) (イベントの系統。`DEAL_HISTORY` (営業案件進捗), `SALES_HISTORY` (売上・商談完了), `AI_HISTORY` (AIエージェント/ワークフロー実行履歴), `IMPROVEMENT_HISTORY` (自己修復・最適化実行履歴))
*   `event_title`: VARCHAR(255) (イベントタイトル。例: `「MEOオートFAQ 3件をGBPへ自律配信完了」`)
*   `event_details`: TEXT (イベントの詳細説明文章、または関連ノードへのリンクUUID)
*   `occurred_at`: TIMESTAMP (イベントが実際に発生した日時。タイムライン描画の基準キー)
*   `created_at`: TIMESTAMP

#### ④ `data_lake_correlations` (多次元因果相関マトリクステーブル)
売上、MEO、SEO、AIO、広告、口コミ、コストなどの実指標データから、AIが定期的に算定した時系列ピアソン相関係数および時間ラグの記録。
*   `id`: UUID (Primary Key)
*   `store_id`: UUID (Foreign Key)
*   `metric_x`: VARCHAR(100) (第1指標。例: `google_ad_roi`)
*   `metric_y`: VARCHAR(100) (第2指標。例: `meo_average_rank`)
*   `correlation_score`: DECIMAL(5, 4) (相関係数 $r \in [-1.0, 1.0]$。正負の相関関係を示す)
*   `lag_days`: INT (時間差（ラグ）日数。例: `3` の場合、指標Xが変化した3日後に指標Yが追従変化する時系列相関が最大であることを示す)
*   `p_value`: DECIMAL(10, 8) (相関の統計的有意性を示す P値。一般に $p < 0.05$ で有意とみなす)
*   `updated_at`: TIMESTAMP

#### ⑤ `data_lake_discoveries` (AI Discovery 自律インサイトテーブル)
時系列相関や異常検知、ログ監査からAIが自律的・能動的に発見した「成功パターン」「失敗パターン」「異常・予兆」「改善候補」の記録。
*   `id`: UUID (Primary Key)
*   `store_id`: UUID (Foreign Key)
*   `category`: VARCHAR(50) (発見された性質。`SUCCESS_PATTERN` (成功法則), `FAILURE_PATTERN` (失敗・バグパターン), `ANOMALY` (異常値/予兆検知), `IMPROVEMENT` (システム/経営改善候補))
*   `content`: TEXT (AIによる詳細な状況の解釈、因果関係のロジック説明、および対策の文章)
*   `evidence_json`: JSON (発見の数学的根拠。関連する相関係数、異常値が発生したノードID、AI Logsなどの証跡データ)
*   `action_payload_json`: JSON (改善のための具体的な物理実行メタデータ。1クリックで Reality Engine をキックするための構造化コマンド)
*   `created_at`: TIMESTAMP

#### ⑥ `data_lake_semantic_embeddings` (セマンティック・ベクトルインデックステーブル)
不変ノード `data_lake_nodes` のコンテンツを RAG（検索拡張生成）およびコサイン類似度検索にマウントするための、多次元ベクトルデータの保管。
*   `id`: UUID (Primary Key)
*   `node_id`: UUID (Foreign Key - データレイク元のノード `data_lake_nodes` のID)
*   `embedding_vector`: VECTOR(1536) (例: `text-embedding-3-small` などで生成された1536次元または多次元の実数配列)
*   `model_name`: VARCHAR(100) (使用したベクトル化モデル名)
*   `created_at`: TIMESTAMP

## 4. 検索インデックス ＆ 相関・自律発見ロジック (Search & Discovery Logic)

AI Data Lakeが、膨大な非構造化ドキュメントをパースし、瞬時にハイブリッド検索を実行した上で、自律的に経営・開発の成功/失敗法則（AI Discovery）を導き出す4段階のプロセス。

```
     [ 1. リアルタイム・ストリーミング・パース（Ingestion）]
       - 15種のデータソースが流入すると、セキュアなストリームリスナーが起動。
       - ファイル、Meeting議事録、AI Logs、Stripeトランザクションを即座に「テキストチャンク」へ解体。
       - メタデータ、タイムスタンプ、付与するタグ（#meo, #finance）を自律マッピング。
                            │
                            ▼
     [ 2. 多元ハイブリッド検索アルゴリズム（Hybrid Search Matrix）]
       - ユーザーが自然言語（NL）やクエリで検索を要求した際。
       - レキシカル全文検索（BM25）による「キーワードの完全一致」と、
         セマンティック意味検索（Embedding Cosine Similarity）による「意味概念の一致」を相互並行実行。
       - 逆ランク融合（RRF: Reciprocal Rank Fusion）を用いて、二つの結果をブレンド・リランキング。
       - メタデータ（タグ #meo、期間 過去90日間）による高速プレフィルタリングを適用。
                            │
                            ▼

       - データベース内の時系列実績データから、異なるドメイン間の相関係数を毎日自動算定。
       - X系列と、時間差 $\Delta t$（ラグ日数）を持たせたY系列の相関関数 $R_{xy}(\Delta t)$ を計算。
         $$R_{xy}(\Delta t) = \frac{\sum (X_t - \bar{X})(Y_{t + \Delta t} - \bar{Y})}{\sqrt{\sum (X_t - \bar{X})^2 \sum (Y_{t + \Delta t} - \bar{Y})^2}}$$
       - 相関が最大となるラグ日数 $\Delta t$ と $p < 0.05$ の有意性を自動特定し、時系列因果関係としてマッピング。
                            │
                            ▼

       - 相関エンジンで特定された有意な因果モデル、異常値アラート、失敗パターン（例: API障害 ➔ ワークフロー停止）を
         AI Executive LLM のコンテキストへ、データレイクのノードエビデンスを添えて厳格に入力。
       - 「成功」「失敗」「異常」「改善」の4つに体系化し、1文字の嘘もない厳密な「自律発見ドキュメント」として保存。
```

### ① ハイブリッド検索リランキング（Reciprocal Rank Fusion - RRF）
BM25スコアによる順位 $R_{bm25}$ と、ベクトル類似度スコアによる順位 $R_{vector}$ を組み合わせ、ノード $d$ の最終スコア $RRF(d)$ を数理的に算定します。これにより、「意味の近さ」と「固有名詞の完全一致」を完璧に両立した次世代検索を実現します。
$$RRF(d) = \frac{1}{k + R_{bm25}(d)} + \frac{1}{k + R_{vector}(d)} \quad (k = 60)$$

### ② AI Discovery による「自己治癒」への展開（Closed-Loop Self Healing）
AI Data Lake が「失敗パターン（Failure Pattern）」を発見した場合、単に画面に表示するだけでなく、以下のような「自己治癒ループ」を自動的に回します。
1.  **失敗の特定**: `「Workflow成功率が 95% を割り込む原因は、GitHub API Tokenの有効期限切れによる HTTP 401 認証障害の連続発生である」`とデータレイクが検出。
2.  **改善指示のパッケージ化**: 失敗ノードの `action_payload_json` に、`{"trigger": "RE_AUTHENTICATE", "provider": "GITHUB"}` という自動修正コマンドをバインド。
3.  **Self Improvement への伝播**: Self Improvement Engine がこのペイロードを検知し、自動的にOAuthリフレッシュシーケンスを実行。人間の手を介さずに24時間自律稼働する強靭なシステム構造を自動維持します。
