# AI Workspace 「オートメーション・センター（Automation Center）」機能設計仕様書

本書は、AI Workspaceにおけるすべての調査、分析、レポート出力、資料作成、アプリ開発、AIエージェントの自律実行、およびワークフロー全体の定常運用をスケジューリングまたはイベント駆動によって完全自動化し、進捗や生成物をマルチチャネルへ通知・永続化するためのバックボーンシステム**「Automation Center（オートメーション・センター）」**の機能設計仕様書です。

本システムを導入することにより、人間が手動で都度AIへ指示を出すことなく、例えば「毎週月曜日の朝に競合店のMEO順位を自動監視し、前週比の改善レポート（PDF/Excel）を自動生成してSlackと担当者メールに通知し、同時にナレッジハブとプロジェクトルームに成果物をコミットする」といった、企業の実務に完全に定着した**「自律稼働型AIワークフォース（自律業務プロセス）」**を構築可能にします。

---

## 1. 業務フロー ＆ 自律自動化ライフサイクル

オートメーションは、時間・イベントを検知する「Trigger」、定義された一連のAIプロセスやワークフローを稼働させる「Executor」、失敗時に原因を自己分析しAIモデルや接続パラメーターを自律チューニングする「Optimizer」、そして通知や保存を行う「Exporter」の4レイヤーで構成されます。

```
                       [1. Trigger Layer (起動検知)]
     ┌──────────────────────────────┼──────────────────────────────┐
     ▼                              ▼                              ▼
  【時間トリガー】               【イベントトリガー】           【手動トリガー】
  (毎日/毎週/毎月/指定日時)       (競合の更新/SEO順位変動/       (ワンクリック実行/
  例: 毎週月曜 朝 9:00           ニュース等)                    API外部連携)
     │                              │                              │
     └──────────────────────────────┼──────────────────────────────┘
                                    ▼
                       [2. Executor Layer (実行)]
     ┌──────────────────────────────┼──────────────────────────────┐
     ▼                              ▼                              ▼
  【Agent チームの自律起動】     【Workflow の自動シーケンス】 【各種単一エンジン実行】
  (リサーチャー/アナリスト)       (入力 ➔ 検索 ➔ PPT生成 ➔ ZIP) (AIO/SEO/MEO/Website等)
     │                              │                              │
     └──────────────────────────────┼──────────────────────────────┘
                                    ▼
                       [3. AI 自己改善 & QA レイヤー (Optimizer & Self-Heal)]
     ├─ 実行失敗時のエラー・ログ解析 (エラー原因：API制限 / タイムアウト / 型不整合 / ハルシネーション)
     ├─ 代替AIモデルの自動選定 (例: Gemini Pro ➔ Claude 3.5 Sonnet への自動切り替え・再実行)
     └─ ワークフロー構成・プロンプトの自律修復およびパラメーター最適化
                                    │ (正常完了時)
                                    ▼
                       [4. Export & Notify Layer (エクスポート・通知)]
     ├─ 成果物の保存: [Knowledge Hub] [Projects (Files/Creation/Documents)]
     └─ マルチ通知: [アプリ内通知] [e-mail] [Slack] [Discord] [Webhooks]
```

## 3. データベース設計 (Database Schema Design)

オートメーションの設定ルール、トリガー条件、実行結果、自己改善ログ、通知アセットを強固に管理するデータベーススキーマ。

```
                  +---------------------------+
                  |     automation_rules      |
                  +---------------------------+
                  | id (PK)                   |
                  | name                      |
                  | description               |
                  | trigger_type              |
                  | cron_expression (Nullable)|
                  | event_type (Nullable)     |
                  | action_type (AGNT/WFLW/etc)
                  | action_ref_id (UUID)      |
                  | project_id (FK, Nullable) |
                  | is_active (Boolean)       |
                  | enable_self_healing (Bool)|
                  | created_by (FK)           |
                  | created_at                |
                  | updated_at                |
                  +-------------+-------------+
                                | 1
                                |
                                ├─────────────────────────────┐
                                | 1..N                        | 1..N
                  +-------------v-------------+   +------------v------------+
                  |      automation_runs      |   |   automation_notifiers  |
                  +---------------------------+   +-------------------------+
                  | id (PK)                   |   | id (PK)                 |
                  | rule_id (FK)              |   | rule_id (FK)            |
                  | status (SUCCESS/FAIL/etc) |   | channel_type (SLACK,etc)|
                  | start_time                |   | destination_address     |
                  | end_time                  |   | message_template (TEXT) |
                  | duration_ms               |   +-------------------------+
                  | cost_usd                  |
                  | error_message (TEXT)      |
                  +-------------+-------------+
                                | 1
                                |
                                | 0..N
                  +-------------v-------------+
                  |  automation_healing_logs  |
                  +---------------------------+
                  | id (PK)                   |
                  | run_id (FK)               |
                  | step_name                 |
                  | detected_error (TEXT)     |
                  | analysis_by_ai (TEXT)     |
                  | action_taken              |
                  | outcome_status            |
                  | created_at                |
                  +---------------------------+
```

### テーブル定義詳細

#### ① `automation_rules` (自動化ルール基本定義台帳)
*   `id`: UUID (Primary Key)
*   `name`: VARCHAR(255) (ルール名。例: 「週次 MEO競合監視レポート」)
*   `description`: TEXT (自動化ルールの目的、処理フロー概要)
*   `trigger_type`: VARCHAR(50) (起動条件。`SCHEDULE` (時間指定/定時実行), `EVENT` (外部データ・イベント連携), `MANUAL`)
*   `cron_expression`: VARCHAR(100) (定時実行用のCron表記。例: `0 9 * * 1` ➔ 毎週月曜 朝9時)
*   `event_type`: VARCHAR(100) (イベント起動用。例: `COMPETITOR_WEBSITE_UPDATED`, `SEO_RANKING_DROPPED`)
*   `action_type`: VARCHAR(50) (実行するアクション形式。`SINGLE_JOB`, `AGENT_RUN`, `WORKFLOW_RUN`)
*   `action_ref_id`: UUID (実行対象となる Workflow, Agent, または特定処理のレコードID)
*   `project_id`: UUID (Foreign Key, Nullable - 紐付けるプロジェクトID)
*   `is_active`: BOOLEAN (稼働・停止切り替えスイッチフラグ)
*   `enable_self_healing`: BOOLEAN (AIエラー自己修復・最適化を有効にするかどうか)
*   `created_by`: VARCHAR(100) (作成ユーザーのID)
*   `created_at` / `updated_at`: TIMESTAMP

#### ② `automation_runs` (ジョブ実行履歴テーブル)
*   `id`: UUID (Primary Key)
*   `rule_id`: UUID (Foreign Key)
*   `status`: VARCHAR(50) (結果。`RUNNING`, `SUCCESS`, `PARTIAL_SUCCESS` (自己修復して成功), `FAILED`)
*   `start_time` / `end_time`: TIMESTAMP (実行開始と完了時刻)
*   `duration_ms`: BIGINT (処理にかかった実ミリ秒数)
*   `cost_usd`: DECIMAL(10, 4) (今回の実行で消費されたAPI・インフラ合計コスト)
*   `error_message`: TEXT (最終的に失敗した際のエラーログ詳細)

#### ③ `automation_healing_logs` (AI自己修復・エラー改善ログ)
実行失敗時、AIがどのように障害を分析し、最適化対応したかの監査ログ。
*   `id`: UUID (Primary Key)
*   `run_id`: UUID (Foreign Key)
*   `step_name`: VARCHAR(100) (失敗・検証が発生したノード・ステップ名)
*   `detected_error`: TEXT (検出された生のエラーメッセージ)
*   `analysis_by_ai`: TEXT (AIが自己分析したエラーの主原因、改善策)
*   `action_taken`: VARCHAR(255) (適用した改善アクション。例: `CHANGE_MODEL_TO_GEMINI_1.5`, `REPARSE_JSON_STRICT`, `RETRY_WITH_HIGHER_TIMEOUT`)
*   `outcome_status`: VARCHAR(50) (適用後の再実行結果。`RESOLVED`, `FAILED_AGAIN`)
*   `created_at`: TIMESTAMP

#### ④ `automation_notifiers` (通知宛先・マッピング定義)
自動処理完了時、どこにどんなメッセージで連携するか。
*   `id`: UUID (Primary Key)
*   `rule_id`: UUID (Foreign Key)
*   `channel_type`: VARCHAR(50) (通知形式。`APP` (アプリ内ポップアップ), `EMAIL`, `SLACK`, `DISCORD`, `WEBHOOK`)
*   `destination_address`: VARCHAR(1024) (メールアドレス、チャンネルのWebhookURLなど)
*   `message_template`: TEXT (AIが通知作成時に参照するひな形。例: 「【MEOレポート】当週の調査結果が完了。星評価は{stars}でした」)

## 4. 実行エンジン設計 (Execution Engine Architecture)

オートメーションを正確、セキュラ、かつスケーラブルに動かすための「バックグラウンド実行エンジン」の仕組み。

### ① スケジューラー＆トリガー監視 (Scheduler & Polling Loop)
*   **Cronベースのスケジューリング**:
    Node.jsの `node-cron` もしくは クラウドサーバー（Cloud Scheduler）と連動し、 `automation_rules` テーブルに保存されたCron式をミリ秒単位で同期監視。
*   **イベントポーリング・フック (Event-driven hooks)**:
    「競合サイトの更新監視」「MEO順位変動」などをトリガーにする場合、バックグラウンドクローラーが指定対象（ショップ、URL）を定期ポーリング。前回のハッシュ値（checksum）との差分を検知した瞬間、イベントキュー（RabbitMQ, BullMQ等）にタスクをパブリッシュし、ジョブを即時キックします。

### ② タスクキュー ＆ コンテキストアイソレーション (Task Queue & Isolation)
*   **並行実行キュー管理**:
    大量のバッチ処理（例: 月曜の朝9時に100個の自動化が一斉に起動）が発生した場合に備え、処理をキューイング。優先順位（Priority）に応じて各ジョブにワーカーを割り当てます。
*   **メモリ・インプット連携**:
    実行されるエージェントやワークフローに対し、所属する「プロジェクト」や「ナレッジ」から「最新の文脈（コンテキスト）」を取り出して実行エンジンに自動インジェクションします。

### ③ 自己改善・自己修復アルゴリズム (AI Self-Optimization Algorithm)
API接続エラーや型崩れによってジョブが異常終了するのを防ぐため、AI自身がテスト・修復ループを実行します。

```
1. ジョブ実行中、あるNodeでエラー（Exception）をキャッチ
                       │
                       ▼
2. 障害ログを「AI自己改善エンジン（Self-Healing Manager）」へディスパッチ
                       │
                       ▼
3. 専用のプロンプト「エラー分析プロンプト」を使い、エラー文とコンテキストから原因を特定
                       │
                       ▼
4. 【原因が外部API制限（Rate Limit等）の場合】
      └─ フォールバックモデル（例: Claude ➔ Gemini）に自動切替 ＆ 30秒バックオフ待機
   【原因が出力フォーマット不整合の場合】
      └─ パーサーに「ルーズパーシング」指示を出し、出力を再加工
                       │
                       ▼
5. 対象のNodeから、修復されたパラメータで自動再実行
      ├─ 成功 ➔ 正常稼働ラインに復帰し、実行ログに「自己修復成功」を記録
      └─ 3回リトライしても失敗 ➔ 管理者へ緊急通知（Error Notification）を送信
```

## 6. 将来拡張設計 (Future Extensibility Plan)

オートメーション・センターが、単なるバッチスケジューラーから、企業の「真の経営ダッシュボード＆自律型AI運用ハブ」に進化するための拡張設計計画。

### ① 組織横断のAI意思決定に基づく「人間承認（Human-in-the-Loop）」フロー
*   **自律生成 ➔ 承認待ち ➔ 公開デプロイ**:
    「Webサイト生成」や「LP更新自動生成」など、顧客の目に触れるアセットを扱う場合、AIが「自動生成完了。人間による承認（Approve）を待機中」という状態でジョブを一時ストップ。
*   Slackの承認ボタンを押す、またはメール内リンクを1クリックするだけで、ワークフローが自動再開され、本番公開（ホスティング）が行われる「人間承認ハブ（Human-in-the-Loop Framework）」を追加設計します。

### ② オートメーション自身の自律最適化（Autonomous CRO & AI Selection Optimization）
*   過去の履歴（`automation_runs`）から、「このMEO分析ルールは、Claudeを使うよりもGeminiを使用した方が、成功率が同等でAPIコストを70%削減できる」といったコストパフォーマンスをAIが自律分析。
*   実行エンジンが「次回から自動でAIモデルを切り替える設定」を提案、または完全自律的にワークフローの「AI選定最適化」を実行し、組織のAI運用コストを極限までセルフチューニングし続ける機能を構築します。
