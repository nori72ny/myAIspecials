# AI Workspace 「ナレッジグラフ・エンジン（Knowledge Graph Engine）」機能設計仕様書

本書は、AI Workspaceにおけるあらゆるアセット（プロジェクト、ナレッジ、文書、メモ、成果物、エージェント、自動化フロー、外部連携ツール等）と、現実のビジネスエンティティ（顧客、競合、キーワード等）を多次元的かつ意味的に紐付け、点在する断片的な情報を相互に関連した「巨大な知のネットワーク」として構築・管理するセマンティックハブシステム**「Knowledge Graph Engine（ナレッジグラフ・エンジン）」**のシステム構成、データスキーマ、グラフ探索アルゴリズム、AIセマンティック検索設計、および画面設計を定義した最上位の製品設計仕様書です。

従来のフォルダ階層やキーワード検索の限界を突破し、AIが「新宿カフェの競合A社のMEO対策」を調査した際、自動的に「競合A」「MEO対策」「新宿店LPプロジェクト」「担当デザイナーエージェント」「Googleビジネスプロフィール（GBP）」「山田太郎（顧客）」といった点と点を結線。エージェントやワークフローが、情報の「コンテキスト（文脈）と依存関係」をグラフ構造で瞬時にたどり、人間の脳細胞（シナプス）さながらに有機的な判断と爆発的な生産性を発揮するためのセマンティックデータ脳幹の役割を担います。

## 1. 業務フロー ＆ 知識グラフ統合コンセプト (Overview & Concept)

Knowledge Graph Engineは、Workspace内で情報が作成・更新、あるいは外部から取得された瞬間、自動的にその意味内容（セマンティクス）を解析し、グラフ空間上に「ノード（点：実体）」と「エッジ（線：関係）」を動的にマッピングします。

```

   (Projects, Documents, Notes, Websites, Apps, Agents, Workflows, Clients, Competitors)

                  [ 2. セマンティックエンティティ抽出 ＆ 関係定義 ]

      ├─ ベクター埋め込み（Embedding）による意味的コサイン類似度の算出

   - コンテキスト全体の依存関係     - 関連するアクション・ツールの   - フォルダを超え、「関連情報
     を把握しハルシネーション防護     依存トリガーを自律探索          を一網打尽」にする高度な検索
        │                               │                               │
        └───────────────────────────────┼───────────────────────────────┘
                                        ▼
                  [ 5. インタラクティブ可視化 (Canvas UI) ]
      ├─ Force-Directed Graph 3D/2D による関係性のビジュアル描画
```

## 2. 管理対象エンティティ ＆ リレーションシップ定義 (Graph Schema & Triples)

ナレッジグラフは、「主語 (Subject) ➔ 述語 (Predicate) ➔ 目的語 (Object)」からなる**セマンティック・トリプル（セマンティック・リレーションシップ）**をベースに構造化されます。

### ① 管理対象エンティティ (Graph Nodes)
*   **Projects (プロジェクト)**: 現在進行中の各種案件、出店計画等。
*   **Knowledge / Documents / Notes**: 蓄積された自社ノウハウ、PDF資料、個人用スクラッチメモ。
*   **Websites / Apps / Presentations / Spreadsheets**: Action Center等を介して生成・管理される、具体的なデジタル成果物アセット。
*   **Agents / Workflows / Automation**: 業務を執行するAI社員、自動バケツリレーのフロー、スケジュール。
*   **Clients (顧客)**: 案件のオーナー、または自社サービスを提供する顧客プロファイル。
*   **Competitors (競合)**: 対象エリアや対象ドメインにおける直接・間接の競合他社・店舗。
*   **Keywords (ターゲットキーワード)**: SEO/MEO/AIOなどで狙うべき重要集客ワード（例: 「新宿 深夜 個室カフェ」）。

### ② 関連定義マトリクス (Graph Edges / Predicates)

| 主ノード (Subject) | エッジ種類 (Predicate) | 目的ノード (Object) | 関係の意味・ユースケース |
| :--- | :--- | :--- | :--- |
| **Projects** | `HAS_CLIENT` | **Clients** | プロジェクトのクライアントを特定 |
| **Projects** | `PRODUCES_ASSET` | **Websites / Spreadsheets** | プロジェクトの直接的な成果物（LPや集計シート） |
| **Agents** | `ASSIGNED_TO` | **Projects** | どのAI社員がどのプロジェクトを担当しているか |
| **Websites** | `TARGETS_KEYWORD` | **Keywords** | LPや制作サイトがどのSEO/MEOキーワードを狙っているか |
| **Competitors** | `COMPETES_WITH` | **Projects / Clients** | 特定の顧客や店舗を競合としてマッピング |
| **Automation** | `TRIGGERS_ACTION` | **Workflows / Tools** | 自動化スケジュールがトリガーする具体的なアクション |
| **Documents** | `REFERENCES_KNOWLEDGE` | **Knowledge** | 作成された提案書が依拠している自社の根本ノウハウ |

## 4. データベース設計 (Database & Triple-Store Schema)

グラフデータベースのデータ構造（Node-Edge Relational Mapping）を、リレーショナル・データベース（PostgreSQL等）で超高速かつ整合性高く処理するためのスキーマ設計。

```
                  +---------------------------+
                  |        graph_nodes        |
                  +---------------------------+
                  | id (PK)                   |
                  | company_id (FK)           |
                  | entity_type (VARCHAR)     |  -- 'PROJECT', 'AGENT', 'CLIENT', 'KEYWORD'
                  | entity_id (UUID/Null)     |  -- 実際のアセット・マスタテーブルへの可変リンク
                  | name (VARCHAR)            |
                  | summary_text (TEXT)       |
                  | vector_embedding_id       |  -- セマンティック検索用のEmbeddingベクター参照
                  | importance_score (INT)    |  -- ノードの重み・中心性
                  | created_at                |
                  | updated_at                |
                  +-------------+-------------+
                                | 1
                                |
                                ├─────────────────────────────┐
                                | 1..N (Source)               | 1..N (Target)
                  +-------------v─────────────────────────────v-------------+
                  |                        graph_edges                      |
                  +---------------------------------------------------------+
                  | id (PK)                                                 |
                  | company_id (FK)                                         |
                  | source_node_id (FK)                                     |
                  | target_node_id (FK)                                     |
                  | edge_type (VARCHAR)                                     |  -- 'CREATED_BY', 'COMPETES_WITH', etc.
                  | confidence_score (DECIMAL)                              |  -- AIが自動結線した際の一致度・信頼度(0.0-1.0)
                  | usage_count (INT)                                       |  -- この接続パスがAIによって読み取られた累計回数
                  | created_at                                              |
                  | updated_at                                              |
                  +---------------------------------------------------------+
```

### テーブル定義詳細

#### ① `graph_nodes` (グラフ・ノードマスタテーブル)
Workspace内のあらゆる概念・オブジェクトを点として一般化・登録するマスタ。
*   `id`: UUID (Primary Key)
*   `company_id`: UUID (Foreign Key - 組織スペースのID)
*   `entity_type`: VARCHAR(100) (実体の種類。`PROJECT`, `KNOWLEDGE`, `DOCUMENT`, `NOTE`, `WEBSITE`, `APP`, `PRESENTATION`, `SPREADSHEET`, `AGENT`, `WORKFLOW`, `AUTOMATION`, `CLIENT`, `COMPETITOR`, `KEYWORD`)
*   `entity_id`: UUID (Nullable - 対応する具体的な実体テーブル（例: `company_agents`, `company_profiles`, `action_integrations` 等）のプライマリキー。これにより、グラフ上のノードをクリックした際に実アセットへ瞬時に紐付けマッピングが可能)
*   `name`: VARCHAR(255) (ノード表示名。例: `NextJS_Coder_Pro`, `新宿個室カフェ La Lounge`)
*   `summary_text`: TEXT (ノードを要約した短いテキスト。セマンティック類似度ベクトルの生成元)
*   `vector_embedding_id`: VARCHAR(255) (ベクター検索エンジン、またはpgvector等のインデックスID)
*   `importance_score`: INT (このノードの近傍ハブ度、重要度指標。0〜100)
*   `created_at` / `updated_at`: TIMESTAMP

#### ② `graph_edges` (グラフ・エッジ（関係）テーブル)
ノード同士の意味的依存関係、取引関係、担当関係を定義するリレーショナル定義。
*   `id`: UUID (Primary Key)
*   `company_id`: UUID (Foreign Key)
*   `source_node_id`: UUID (Foreign Key - 矢印の起点となるノードID)
*   `target_node_id`: UUID (Foreign Key - 矢印の終点となるノードID)
*   `edge_type`: VARCHAR(100) (エッジの種類。`HAS_CLIENT`, `ASSIGNED_TO`, `PRODUCES_ASSET`, `TARGETS_KEYWORD`, `COMPETES_WITH`, `TRIGGERS_ACTION`, `REFERENCES_KNOWLEDGE`, `BELONGS_TO_DEPT`)
*   `confidence_score`: DECIMAL(3, 2) (この関係の信頼度・合意率。0.00〜1.00。AIによる自動マッピングの場合はセマンティック類似度から動的に算出。人間が手動で結線した場合は `1.00` 固定)
*   `usage_count`: INT (AIエージェント、検索エンジン、ワークフローが、文脈補強のためにこのパス（リレーションシップ）を参照・ロードした累計コール回数。パスの人気・重要度の監査用)
*   `created_at` / `updated_at`: TIMESTAMP

## 5. AIセマンティック検索設計（GraphRAG エンジン）

ナレッジグラフの圧倒的強みは、単純なテキスト類似度で文書を探す従来のRAG（Retrieval-Augmented Generation）を遥かに凌駕する**「GraphRAG（ナレッジグラフ補強型生成）」**を実現できる点にあります。

### ① GraphRAG 探索アルゴリズム (Semantic Graph Retrieval)
ユーザーからクエリ（例: 「新宿LPのデザイン方針を決定して」）が入力された際、GraphRAGエンジンは以下のパイプラインで情報を引き出します。

```
1. クエリ（質問）の意味的ベクター（Embedding）を作成。
   └─ 「新宿LPのデザイン方針」
                       │
                       ▼
2. グラフノード（graph_nodes）からコサイン類似度で最も近い「起点ノード」を検出。
   ├─ 類似度トップ: [ノード A: 新宿個室カフェ La Lounge プロジェクト]
   └─ 類似度セカンド: [ノード B: 新宿カフェLP]
                       │
                       ▼
3. 起点ノードから 1次・2次近傍のエッジ（関係）を高速トラバース（グラフ探索）。
   ├─ ノード A ➔ (PRODUCES) ➔ [ノード C: UI_UXデザインスタイルガイド.pdf]
   ├─ ノード A ➔ (ASSIGNED_TO) ➔ [ノード D: Designer_B (AIエージェント)]
   └─ ノード B ➔ (TARGETS_KEYWORD) ➔ [ノード E: 「新宿 深夜 個室カフェ」]
                       │
                       ▼
4. 抽出された関連ノード一式から「コンテキスト網（知識サブグラフ）」を自動編成。
   ├─ 「新宿LPを狙うキーワードは新宿・深夜・個室。デザインガイドラインは、Cosmic Midnightの
   │   ネイビー＆シアン。担当デザイナーはDesigner_Bであり、山田太郎オーナーの美学を踏襲中…」
   └─ 単一フォルダにとらわれず、プロジェクトを包む「文脈情報（コンテキスト）」が漏れなく完成。
                       │
                       ▼
5. Unified AI Brain へこのコンテキスト網をシステム制約として入力し、回答を生成。
   └─ 極めて具体的で、ハルシネーションのない、完璧な意思決定回答を出力。
```

### ② 多層記憶（Personalized Graph Memory）
グラフ上に格納される「記憶」は、以下の4階層で整理され、エージェントがいつでも引き出せるようにマッピングされます。
*   **長期記憶 (Long-term System Memory)**:
    ユーザーの生涯的な美学、不変のビジネス哲学。グラフの最高位に配属（例: 山田太郎 ➔ `PREFERS_QUALITY` ➔ 高単価プレミアムモデル）。
*   **案件記憶 (Project-specific Memory)**:
    特定プロジェクト内の全アセット、開発コード、LPカラー、MEO順位の推移履歴。プロジェクトノードに直接紐づく近傍グラフ。
*   **顧客記憶 (Client Relationship Memory)**:
    顧客ごとの過去のトラブル、好みの返信トーン、決済履歴。クライアントノードから過去の別プロジェクトへとエッジを結線。
*   **学習記憶 (Self-Improving Memory)**:
    エージェントが自律業務中に獲得した新たな教訓、失敗からのプロンプト補正ルール。エージェントノード ➔ `LEARNED_RULE` ➔ 改善メモ、として蓄積。

## 6. 他システム（OS、ツイン、ブレイン、アクション、ハブ、マーケット）との統合

Knowledge Graph Engineは、AI Workspaceを構成するすべての自律エンジンの「共有メモリコア」として機能し、データベースの分断を解消します。

1.  **AI Company OS ＆ デジタルツイン (Twin & Org Integration)**
    *   AI組織図内の全部署、全AI社員エージェントは、グラフ上のノードとして完全に管理。
    *   「オーナー山田太郎のデジタルツイン」がAI役員会で意見を述べる際、ツインはナレッジグラフの山田太郎ノードからエッジをトラバースし、「山田太郎の不変の美学（長期記憶）」と「直近の新宿プロジェクト成果物（案件記憶）」をシームレスに合成して発言。本人との一致率94.2%の奇跡的精度を担保します。
2.  **Unified AI Brain 連携 (コンセンサス監査 ＆ ハルシネーション完全排除)**
    *   Unified AI Brainが複数の異なるLLMで「自動討論（Debate Mode）」を行わせる際、各LLMが出力した固有名詞や数値ファクトを、ナレッジグラフに登録されている「実データノード」と常時突き合わせて整合性を検査（Fact Check Engine）。
    *   グラフ上の事実（例: 「新宿La Loungeは完全個室」）と矛盾する意見（例: 「新宿はオープン席主体です」）を述べたモデルの出力を、調停エンジンが自動的に棄却・修正（誤情報防護）します。
3.  **Action Center ＆ Tool Connector Hub (動作のコンテキスト配線)**
    *   エージェントが「Slackに告知文を投稿し、NetlifyでLPを公開する」というアクションを実行する際。
    *   アクションオブジェクトがグラフ上で「新宿プロジェクト ➔ PRODUCES ➔ 新宿LP ➔ DEPLOYED_VIA ➔ Netlify ➔ CONNECTED_TO ➔ Tool Connector Hub」として視覚的かつ論理的に接続。
    *   実行エラーが発生した場合、エラーログノードがグラフに動的追加され、COO Agentがエラーの原因関係（どのAPIの期限切れかなど）をグラフから特定し、自動再接続アクション（自己修復）をキックします。
4.  **Private Agent Marketplace (タレントとスキルのインデックス)**
    *   マーケットプレイスから雇用されたエージェントは、ナレッジグラフ上に登録され、エージェントが過去に作成した「全成果物アセット（HTML、Excel、スライド）」との関係性が常時マッピング。
    *   「最も多くの成果物と結ばれており、高い評価（KPI）エッジを持つ人気AI社員」を、マーケットプレイスのランキングダッシュボードに動的反映させます。

## 7. 将来拡張設計 (Future Extensibility Plan)

### ① Cross-Workspace Graph Sync（企業間知識グラフセキュア連動）
*   **他社の企業システム（別のAI Workspace）と自社の知識グラフを、機密を保持したまま自律接続・折衝する仕組み**:
    *   自社が「新宿店オープンのための不動産仕入れ」を行う際、パートナー不動産会社の「不動産知識グラフ」とセキュアにP2P通信。
    *   お互いの機密情報（顧客個人情報や社内コスト等）にはマスキング（ゼロ知識証明等の暗号技術）を施しつつ、外部に公開可能なノード（物件間取り、坪単価、対象キーワード等）のみを相互リンクさせ、AIエージェント同士がグラフ上のパスを辿って最適なJV（共同プロジェクト）をマッチング・推進する「グラフフェデレーション（AIアライアンスネットワーク）」へと拡張します。

### ② Neural-Graph Self-Growing Logic（自己進化型ニューラル・グラフ自律増殖）
*   **AIエージェント自身が人間（オーナー）の指示なしで、日々の実務から隠れたインサイト（関係性）を自律推理し、グラフを自己増殖する仕組み**:
    *   「新宿店の売上データ」と「周辺競合カフェのMEO順位」「天候データ」「ターゲット客層の個人メモ」をバックグラウンドで走る「グラフ進化エージェント」が多次元分析。
    *   人間が「新宿店は、雨の日の深夜24時以降、客単価が40%急上昇する」という明示的なメモを書いていなくても、AIが統計データからその関係性を発見。
    *   グラフ空間上に `RAINY_LATE_NIGHT` ➔ `BOOSTS_REVENUE` ➔ `新宿店` という「発見された法則（インサイトノード）」を自動で新規追加。
    *   このインサイトを元に、Action Centerを介して「雨の日の23時、Xで『深夜個室暖房完備・雨の日特別クーポン』を完全自動で予約投稿する」という超先回りの経営オートメーションを自律起動させる、真の知的生命体（AI Corporate Brain）へと進化させます。
