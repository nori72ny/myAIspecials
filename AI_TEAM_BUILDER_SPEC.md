# AI Workspace 「AIチーム・ビルダー（AI Team Builder）」機能設計仕様書

本書は、AI Workspaceにおける複数の専門AIエージェントを束ね、目標達成に向けて協調・連携して自律稼働する「バーチャルAI組織・混成チーム」を直感的に構築、管理、シミュレーションするための統合指揮システム**「AI Team Builder（AIチーム・ビルダー）」**のシステム構成、データスキーマ、コラボレーションプロトコル、実行（オーケストレーション）設計、および画面レイアウトを定義した最上位の製品設計仕様書です。

単体のAIエージェントでは解決が困難な、長期かつ複雑な業務プロセス（例：「Next.jsアプリの設計・コーディング・テスト・自動デプロイまでの一気通貫開発」や「競合分析からLP文章生成、SNS投稿、検索順位追跡に及ぶSEO/MEOマーケティング」）に対し、適切なスキルセットを持つAI社員を「リーダー」「メンバー」「サポート」としてアサインし、チーム内での自律的なディスカッション、ピアレビュー、相互修正、合意形成（投票）を通じて、極めて品質の高いアウトプットを全自動で生成する**「自律型バーチャルオフィス（Multi-Agent Swarm Integration）」**の役割を担います。

---

## 1. システム概要 ＆ マルチエージェント協調コンセプト (Overview & Multi-Agent Concept)

AI Team Builderは、複数のエージェントが相互にメッセージを交換し、お互いの成果物をレビュー・修正し合う「エージェント間コラボレーション空間」を提供します。

```
                                [ 1. 人間による指令 (Goal Definition) ]
                    「新宿個室カフェの、集客用LPをMEOに最適化した状態で制作・公開して」
                                                   │
                                                   ▼
                                     +---------------------------+
                                     |      [ Leader Agent ]     | (1名・進行指揮、最終承認、要約)
                                     +-------------┬-------------+
                                                   │
                        ┌──────────────────────────┼──────────────────────────┐
                        ▼                          ▼                          ▼
             +--------------------+     +--------------------+     +--------------------+
             |  [ Member Agent ]  |     |  [ Member Agent ]  |     |  [ Support Agent ] |
             |    (SEO Marketer)  |     |   (NextJS Coder)   |     |  (Analytics Bot)   |
             +----------┬---------+     +----------┬---------+     +----------┬---------+
                        │                          │                          │
                        └──────────────────────────┼──────────────────────────┘
                                                   │
                                                   ▼
                                  [ 2. チーム内コラボレーション (Co-Working) ]
                     ├─ A. 順次実行（バケツリレー型）：調査 ➔ 執筆 ➔ コーディング ➔ 検証
                     ├─ B. 並列実行（分散作業）：デザイン作成とSEOキーワード抽出を同時並行
                     ├─ C. 条件分岐 ＆ レビュー：記述にバグやSEO最適化不足があれば差し戻し
                     └─ D. 合意形成・投票：3体のエージェントによる成果物の相互品質チェックと採点
                                                   │
                                                   ▼
                                  [ 3. セマンティック脳・成果物連携 ]
                     ├─ Knowledge Graph：関係情報のリアルタイム共有、長期・短期記憶の同期
                     ├─ Project / Files：生成したコードや分析レポートの共有フォルダへの保存
                     └─ Workflow Engine：ツール・APIコネクタによる自律的な公開・実行（Netlify等）
```

## 3. データベース・スキーマ設計 (Data Schema Design)

AIチーム、チーム内での役割（アサイン）、実行ワークフロー定義、およびチーム内対話ログをセキュアかつ効率的に管理するスキーマ設計。

```

                  | id (PK)                   |
                  | company_id (FK)           |

                  | description (TEXT)        |
                  | is_active (Boolean)       |
                  | created_at                |
                  | updated_at                |
                  +-------------+-------------+
                                | 1
                                |
         ┌──────────────────────┼──────────────────────┬──────────────────────┐
         | 1..N                 | 1..N                 | 1..N                 | 1..N
  +------v------+        +------v------+        +------v------+        +------v------+
  | team_members|        |team_workflow|        | team_collab_ |        |  team_kpi_  |
  |             |        |  _steps     |        | _messages    |        |  statistics |
  +-------------+        +-------------+        +-------------+        +-------------+
  | id (PK)     |        | id (PK)     |        | id (PK)     |        | id (PK)     |
  | team_id (FK)|        | team_id (FK)|        | team_id (FK)|        | team_id (FK)|
```

### テーブル定義詳細

#### ① `ai_teams` (AIチーム定義マスタ)
*   `id`: UUID (Primary Key)
*   `company_id`: UUID (Foreign Key - 組織スペースのID)
*   `team_name`: VARCHAR(255) (チーム名。例: `Marketing Team Beta`)
*   `team_type`: VARCHAR(100) (チームの種類。`RESEARCH`, `SEO`, `AIO`, `MEO`, `MARKETING`, `SALES`, `DEVELOPMENT`, `CONTENT`, `CUSTOM`)
*   `description`: TEXT (チームの目的、得意業務の解説)
*   `is_active`: BOOLEAN (チームが現在稼働可能・アサイン可能か)
*   `created_at` / `updated_at`: TIMESTAMP

#### ② `team_members` (チームメンバー構成・役割定義)
チームに所属する個別エージェントと、チーム内での役割階層を定義するマッピング。
*   `id`: UUID (Primary Key)
*   `team_id`: UUID (Foreign Key)
*   `agent_id`: UUID (Foreign Key - `marketplace_agents` 等への参照キー)
*   `role_type`: VARCHAR(50) (チーム内階層・権限。`LEADER` (指示出し、他メンバーの成果物の統合、人間とのやり取り窓口), `MEMBER` (実務、コード生成、執筆等), `SUPPORT` (補助、テスト実行、APIコネクタによる成果物投稿))
*   `status`: VARCHAR(50) (アサイン状態。`ACTIVE`, `STANDBY`, `ON_LEAVE`)

#### ③ `team_workflow_steps` (チーム内並列・直列ワークフロー進行配線)
チームがタスクを解決する際の「自律バケツリレー」の進行順序と条件分岐。
*   `id`: UUID (Primary Key)
*   `team_id`: UUID (Foreign Key)
*   `step_order`: INT (実行順序。例: 1, 2, 3)
*   `assigned_agent_id`: UUID (Foreign Key - このステップを処理するエージェントID)
*   `execution_type`: VARCHAR(50) (実行モード。`SEQUENTIAL` (前のステップが完了したら開始), `PARALLEL` (他ステップと並行実行), `CONDITIONAL_BRANCH` (条件分岐), `AUDIT_REVIEW` (監査・ピアレビュー))
*   `parent_step_id`: UUID (Foreign Key, Nullable - 条件分岐や差し戻しが発生した際の戻り先、または分岐元の親ステップID)
*   `created_at`: TIMESTAMP

#### ④ `team_collab_messages` (チーム内対話 ＆ ピアレビュー履歴)
エージェント同士が成果物を良くするために交わしたディスカッション、投票、採点の記録。
*   `id`: UUID (Primary Key)
*   `team_id`: UUID (Foreign Key)
*   `sender_agent_id`: UUID (Foreign Key - 発言したエージェントのID)
*   `message_text`: TEXT (発言内容、または生成された中間成果物)
*   `vote_score`: INT (Nullable - レビュー時の採点スコア。0〜100。合意形成アルゴリズムで使用)
*   `created_at`: TIMESTAMP

## 4. チーム実行（オーケストレーション）設計 (Multi-Agent Execution Engine)

複数のAIエージェントを自律協調させるための、コア実行・合意形成エンジン（Consensus Algorithm）の仕様。

### ① 実行フローの制御パターン (Workflow Patterns)
*   **順次実行（Sequential Pipeline）**:
    リーダーが作成した「設計書」を、実装メンバーが受け取って「コード」にし、サポートエージェントが「テスト」を走らせる直列型プロセス。
*   **並列実行（Parallel Swarm）**:
    「新宿カフェ」「渋谷カフェ」「池袋カフェ」の3大エリアの競合調査を、3体のエージェントに並行して分散アサインし、最後にリーダーが1枚の比較スライドにマージする並列型プロセス。
*   **ピアレビュー ➔ 条件分岐差し戻し（Critic-Loop）**:
    メンバーが執筆したLP原稿を、監査エージェントがSEO/MEOの観点から判定。設定された合格点（例：85点）に満たない場合、自動的に修正フィードバック文脈を乗せて「執筆エージェントへ差し戻し（ループ実行）」を繰り返します（最大リトライ回数を設定し、無限ループを防止）。

### ② チームコラボレーションプロトコル (Agent Collaboration & Voting Protocol)
*   **エージェント間対話 (Inter-Agent Chat)**:
    各エージェントは、相手エージェントの「システム指示（System Instructions）」を理解した上で、メッセージAPI（JSON形式）を介してやり取りを行います。
*   **合意形成・投票システム (Consensus Voting)**:
    重要な意思決定（例：「NetlifyへLPを公開するかどうか」）の前に、チーム内の全メンバーによる「自律投票」を行います。
    *   **多数決プロトコル (Majority Vote)**: チームメンバーの過半数（例：3名中2名以上）が「合意」した場合にアクションを実行。
    *   **リーダー専決プロトコル (Leader Decides)**: メンバーのレビュー点数を参考にしつつ、最終決定権はリーダーエージェントのみに帰属。
    *   **人間承認プロトコル (Human-in-the-Loop)**: AI同士の合意スコアが一定水準（例：80%）を超えた場合でも、本番デプロイ直前に人間のUI画面へ「最終承認ボタン」を掲示して一時停止（安全弁）。

## 5. チームメモリ ＆ コンテキスト同期設計 (Team Unified Memory & State)

チームがひとつの有機体として動作するためには、全メンバーが「同じビジョン、同じファイル、同じ顧客の好み」をタイムラグなく共有（脳の共有）する必要があります。

```
[ 共通フォルダ: Project Files ]  ◀───────────┐ (全アセットの同期読み書き)
                                             │
                                             ▼
[ ナレッジグラフ: Knowledge Graph ] ◀─────▶ [ AI チームメンバー (全員) ]
                                             ▲
                                             │
[ 短期記憶: Session History (Redis) ] ◀──────┘ (リアルタイム対話コンテキスト)
```

1.  **ナレッジグラフ（Knowledge Graph）の一元マッピング**:
    *   チームがアサインされたプロジェクトに関するナレッジグラフノード（顧客の美学、ターゲットキーワード、競合他社のデータ）は、チームに所属するすべてのエージェントの「推論時コンテキスト（グラウンディング）」に最初から統合・インジェクションされます。
    *   メンバーAが新たに発見した競合店舗の情報は、即座にナレッジグラフにノード登録され、メンバーBが次の開発ステップを実行する際には「すでにその競合の存在を知っている状態」で処理が行われます。
2.  **プロジェクトファイル（Project / Files）の動的共有**:
    *   チーム専用の共有ディレクトリスペースを用意。エージェントが作成中のコード、ドキュメント、CSVは、共有Files上で一元管理されます。
    *   「ファイルロック」や「バージョン管理（Gitライクな追跡）」をバックグラウンドで行い、エージェント同士の書き込み競合（コンフリクト）を防止します。
3.  **セッション短期対話メモリ (Real-time Session Memory)**:
    *   チーム内での議論（チャット履歴）は、ひとつの会話セッションとしてRedis等の高速メモリにキャッシュされ、エージェントが発言するたびに「過去のディスカッションの流れ」がシステムプロンプトに動的付加されます（Context Windowの許容範囲内で要約・保持）。

## 6. 他システム（OS、グラフ、ハブ、マーケット、アクション）との統合

AI Team Builderは、AI Workspaceにおける「複数AI社員のオペレーションシステム（OSのチーム管制）」として、各インフラと高度に連動します。

1.  **AI Company OS / Personal AI OS 連携**
    *   AI Company OSの「組織図（Org Chart）」は、このTeam Builderをビジュアル化したものです。
    *   ユーザーが「新規に『飲食集客タスクチーム』を組織図に追加」した際、Team Builder APIが走り、MEO Teamテンプレートから各メンバーエージェントを自動召喚・マッピングします。
2.  **Tool Connector Hub ＆ Action Center 連携**
    *   チームにアサインされた「サポートエージェント（Support Agent）」は、Tool Connector Hubを通じて組織が許可した外部サービス（Google Sheets、Slack、GitHub等）への安全なゲートウェイを開放されます。
    *   メンバーが生成したコードやデータを、サポートエージェントがバトンを受け取り、Action Centerを介して実際にネットへ公開、成果をSlackへ報告します。
3.  **Private Agent Marketplace 連携 (タレント召喚)**
    *   チームを構成する各エージェントは、すべてPrivate Agent Marketplaceから「調達（雇用）」されます。
    *   「Next.jsアプリを開発したい」となった場合、チーム構成編集画面のメンバー追加候補として、Marketplaceで社内評価No.1の `NextJS_Coder_Pro` が推薦表示されます。

## 7. 将来拡張設計 (Future Extensibility Plan)

### ① Self-Organizing Swarms（自律チーム自律組成 ＆ 自己リファクタリング）
*   **人間が目標を1つ与えるだけで、AIが自ら最適な「チーム構成」を提案・召喚・自律実行する仕組み**:
    *   ユーザーが「新宿に防音個室カフェのLPを新規立ち上げして」と指示した際。
    *   システムが過去の類似成功データ（Knowledge Graph）を探索。
    *   「このタスクの完遂には、MEO Teamテンプレートに加えて、Next.jsの実装経験がある『NextJS_Coder_Pro』を1名増員すべきである」とAIが判断。
    *   自動的にメンバーを追加雇用し、業務分担フロー（Workflow Step）を自律設計して、チーム稼働を開始する「自己組織化スウォーム（Self-Organizing AI Swarm）」へと拡張します。

### ② AI Team vs. AI Team Debate (切磋琢磨・競合シミュレーション)
*   **2つの異なるAIチーム（チームA：攻め、チームB：守り）を仮想空間上で対決させ、限界までアウトプットを研ぎ澄ます仕組み**:
    *   **チームA (Red Team)**: セキュリティの脆弱性やプロンプトインジェクション、バグ、SEOの隙を徹底的に攻撃する、悪魔のデバッガー＆ホワイトハッカーチーム。
    *   **チームB (Blue Team)**: 開発を推進し、高品質なコードや文章、強固なインフラを構築する開発チーム。
    *   本番デプロイ前に、この2つのチームをバーチャルサンドボックス内で戦わせ、ハッキングやバグをすべて出し尽くさせ、自己修復した後の「究極に安全かつ頑健なシステム」のみを市場に送り出す、超進化型ゲーム理論（Generative Adversarial Multi-Agent Engine）へと拡張します。
