# AI Workspace 「マルチAIオーケストレーター (AI Orchestrator v2)」機能設計書

本書は、AI Workspaceにおける従来の「単一AIモデルへの問い合わせ・回答」から、ユーザーのインテント（意図）をインテリジェントに解析し、世界中の主要な超巨大言語モデル（LLM）群を最適に組み合わせ・並行実行・検証・統合する**「マルチAIオーケストレーター (AI Orchestrator v2)」**の機能設計書です。

---

## 1. システムアーキテクチャ ＆ ルーティングロジック

### 💡 コンセプト
**「最速・最高品質・最高精度の自律的ルーティング ＆ 合意形成（コンセンサス）エンジン」**
指示内容の属性（検索、調査、分析、開発等）およびユーザーの「実行モード」に基づき、最適なAIモデル群を自動選定。個別回答を並行生成した上で、それらの「共通点」「相違点」を抽出し、信頼度の高い「最終統合回答」と「推奨アクション」を1つのインターフェースに統合して提示します。

### 🔌 対応AIプロバイダー ＆ 統合モデル群
*   **OpenAI**: GPT-4o, GPT-4o-mini (汎用性、コード、ロジック分析)
*   **Gemini**: Gemini 1.5 Pro, Gemini 1.5 Flash (大容量コンテキスト、推論、データ調査)
*   **Claude**: Claude 3.5 Sonnet, Claude 3.5 Opus (高品質文章、論文分析、複雑なプログラミング)
*   **Perplexity**: Perplexity Sonnet/Llama (リアルタイムWeb検索・引用ファクト検証)
*   **DeepSeek**: DeepSeek-V3, DeepSeek-R1 (超低コスト思考、高度な数理・コード生成)
*   **Grok**: Grok 2, Grok 2 mini (最新ニュース、SNSセンチメント分析)
*   **Qwen**: Qwen2.5-72B (多言語対応、アジア圏市場調査、構造化データ出力)
*   **Mistral**: Mistral Large 2 (欧州規格準拠、高精度コーディング)
*   **OpenRouter経由モデル**: Llama-3-70B, Phi-3, Claude/OpenAI各種微調整モデル等

| 業務ドメイン | 優先AIモデル 1 | 優先AIモデル 2 | 優先AIモデル 3 | 優先ルーティング条件 |
| :--- | :--- | :--- | :--- | :--- |
| **一般検索** | Perplexity | Gemini | Qwen | 最新ファクト、Web情報への引用が必要な場合 |
| **最新情報** | Perplexity | Grok | Gemini | 直近の時事問題、トレンド、SNS反響 |

| **コード生成** | ChatGPT | Claude | DeepSeek | 型安全、ライブラリ整合性、効率的アルゴリズム |
| **Webサイト生成** | ChatGPT | Claude | Qwen | Tailwind CSS、UIの審美性、コンポーネント調和 |
| **アプリ生成** | ChatGPT | Claude | DeepSeek | フルスタック接続性、DB定義、API設計ロジック |

## 2. 3つの実行モード (Execution Modes)

タスクの緊急度、求められる品質、コスト効率のトレードオフに合わせて、ユーザーは3つの実行モードから最適な組み合わせをワンクリックで選択できます。

```
                    +---------------------------------------+
                    | ユーザーのプロンプト / テンプレート入力 |
                    +---------------------------------------+
                                        │
                                        ▼
                         [AIルーティング・ディスパッチャー]
                                        │
         ┌──────────────────────────────┼──────────────────────────────┐
         ▼ (Fast モード)                ▼ (Balanced モード)            ▼ (Expert モード)
  +---------------+              +------------------+           +------------------+
  | 最適なAI 1つ   |              |  最適なAI 3つ     |           |  最適なAI 5〜8つ  |
  | (例: Perplexity) |           | (例: Claude/GPT/ |           | (例: Claude/GPT/ |
  +──────┬────────+              |  DeepSeek)       |           | DeepSeek/Gemini/ |
         │                       +────────┬─────────+           | Perplexity/Grok) |
         │                                │                     +────────┬─────────+
         │                                │                              │
         └──────────────────────────────┼──────────────────────────────┘
                                        ▼
                        [コンセンサス・統合エンジン (Aggregator)]
                                        │
                                        ├─► AI別の個別回答をパース
                                        ├─► 重なり合う「共通点（Fact）」を抽出
                                        ├─► 対立・乖離する「相違点（Perspective）」を特定
                                        └─► 知的統合した「最終統合回答」＆「信頼度スコア」を算出
```

### ① Fast モード (スピード重視)
*   **起動AI数**: 1モデル（ルーターが「最有力」と判定したモデル）
*   **特徴**: 最も応答速度が早く、コスト効率が高い。一般的な事実確認や、シンプルなコード修正、日常のクイックドラフト作成に最適。

### ② Balanced モード (バランス重視)
*   **起動AI数**: 3モデル（ルーターが選定した各特性のトップ3）
*   **特徴**: 回答の偏りを防ぎ、多角的な視点を取り入れる。「要約」「構成案」「長文執筆」などの多重検証が必要な中規模タスクに最適。平均10〜15秒でコンセンサスが仕上がる。

### ③ Expert モード (超高精度・最高品質重視)
*   **起動AI数**: 5〜8モデル（全プロバイダーから適性スコアが高いものを並行召喚）
*   **特徴**: 
    競合分析、ビジネス提案書、高度なシステムアーキテクチャ設計などの絶対的な正確性・品質が求められるミッションクリティカルな業務。
    各エージェントの出力結果を「AI査読エンジニア」が相互検証し、ファクトの不整合（ハルシネーション）を自動で排除します。

## 3. 出力コンポーネント (Output Interface Details)

オーケストレーターが並行実行を完了すると、ユーザーに提供される結果は以下の6層構造で高度に統合・可視化されます。

1.  **AI別回答 (Model-Specific Answers)**:
    並行実行された各AI（例: Claude、Gemini、ChatGPTなど）の生の出力を、タブ切り替えUIでそれぞれ美しく表示（マークダウンレンダリング）。
2.  **共通点 (Commonalities / points of consensus)**:
    選定した複数のAIが「共通して認めている事実、推奨事項、推奨ツール」などを客観的に要約。ファクトとしての信頼性が極めて高い部分です。
3.  **相違点 (Differences / discrepancies)**:
    各AIモデル間で意見が分かれた部分、アプローチの差、見積もり数値のズレ、推奨優先順位の不一致などを対比表形式で表示。人間に思考を促す「論点」として抽出。
4.  **最終統合回答 (Final Integrated Synthesis)**:
    共通点を軸に、相違点の論点整理を加え、最も信頼性が高く体系化された「究極のマスター回答」をコンポーネントとして自動出力。
5.  **信頼度スコア (Confidence Score)**:
    モデル間の回答合意率、参照した最新ソース（Web引用）のエビデンス密度、ハルシネーション検出スコアから、全体の信頼度を100%満点で算出。
6.  **推奨アクション (Recommended Actions)**:
    最終統合回答から導き出される、「ユーザーが次に起こすべき具体的なビジネスステップ、コマンド、または適用可能なテンプレート」をチェックリスト化。

## 4. データベース設計 (Data Model Schema)

マルチAIの呼び出し設定、実行時の各モデルパラメーター、統合されたメタ解析テキストを整合性高く蓄積するスキーマ。

```
                  +---------------------------+
                  |    orchestration_jobs     |
                  +---------------------------+
                  | id (PK)                   |
                  | prompt (TEXT)             |
                  | execution_mode            |
                  | routing_strategy          |
                  | status                    |
                  | overall_confidence        |
                  | created_at                |
                  | updated_at                |
                  +-------------+-------------+
                                | 1
                                |
                                | 1..N
                  +-------------v-------------+
                  |     individual_runs       |
                  +---------------------------+
                  | id (PK)                   |
                  | job_id (FK)               |
                  | model_name (e.g., claude) |
                  | api_provider              |
                  | latency_ms                |
                  | raw_response (MEDIUMTEXT) |
                  | token_count (JSON)        |
                  | status                    |
                  +-------------+-------------+
                                | 1
                                |
                                | 1..1
                  +-------------v-------------+
                  |    orchestrated_synthesis |
                  +---------------------------+
                  | id (PK)                   |
                  | job_id (FK, UNIQUE)       |
                  | commonalities (TEXT)      |
                  | differences (TEXT)        |
                  | final_synthesis (TEXT)    |
                  | recommended_actions (JSON)|
                  | analysis_metadata (JSON)  |
                  | created_at                |
                  +---------------------------+
```

### テーブル定義

#### ① `orchestration_jobs` (オーケストレーション案件)
マルチAIを束ねる全体プロセスのマスター。
*   `id`: UUID (Primary Key)
*   `prompt`: TEXT (ユーザーの指示内容)
*   `execution_mode`: VARCHAR(50) (`FAST`, `BALANCED`, `EXPERT`)
*   `routing_strategy`: VARCHAR(100) (ルーターが判定したドメイン。例: `CODE_GEN`, `COMPETITOR_ANALYSIS`)
*   `status`: VARCHAR(50) (全体状態: `ROUTING`, `RUNNING_MODELS`, `SYNTHESIZING`, `SUCCESS`, `FAILED`)
*   `overall_confidence`: INT (最終算出された全体信頼度: 0-100)
*   `created_at` / `updated_at`: TIMESTAMP

#### ② `individual_runs` (個別モデル並行実行履歴)
実行された個別のモデルのログ、応答速度、出力を個別に追跡。
*   `id`: UUID (Primary Key)
*   `job_id`: UUID (Foreign Key)
*   `model_name`: VARCHAR(100) (モデルID。例: `claude-3-5-sonnet`)
*   `api_provider`: VARCHAR(100) (プロバイダー。例: `anthropic`, `openrouter`)
*   `latency_ms`: INT (モデル応答速度。ミリ秒単位)
*   `raw_response`: MEDIUMTEXT (各AIから返ってきた未加工回答)
*   `token_count`: JSON (消費したインプット・アウトプットトークン数)
*   `status`: VARCHAR(50) (個別状態: `PENDING`, `RUNNING`, `SUCCESS`, `FAILED`)

#### ③ `orchestrated_synthesis` (コンセンサス統合成果物)
AI群の合意形成アルゴリズムによって最終生成された出力。
*   `id`: UUID (Primary Key)
*   `job_id`: UUID (Foreign Key, Unique)
*   `commonalities`: TEXT (各モデルの合意ファクト要約)
*   `differences`: TEXT (各モデル間の相違点・対立意見対比)
*   `final_synthesis`: MEDIUMTEXT (最終統合回答)
*   `recommended_actions`: JSON (優先度付き推奨アクションの配列データ)
*   `analysis_metadata`: JSON (実行時のソースURL、参照エビデンス等のメタデータ)
*   `created_at`: TIMESTAMP

## 6. 将来的なコンセンサス品質の向上 ＆ セルフコレクト (Self-Correction)

AI Orchestrator v2の信頼度を本質的に引き上げるための、知的自律ループ拡張設計。

1.  **AI査読付きダブルチェックループ (Peer-Review Loop)**:
    *   Balanced/Expertモードの実行時、回答をいきなりユーザーに提示するのではなく、回答生成後に「Claudeが作成した戦略をGPT-4oがファクトレビューする」「DeepSeekが作ったコードをClaudeがセキュア監査する」といった、AI同士のクローズドな批評・修正ラウンドを内部的に1ステップ挟むことで、コードの動作バグやテキストの論理破綻をほぼゼロにします。
2.  **ハルシネーション（嘘）のリアルタイム検知スコアリング**:
    *   Perplexityが検索して得た「一次情報（Webページ本文）」と、他の推論AIが作成した記述に矛盾がないかを意味論的（Semantic）に照合。
    *   もし事実と異なる年号、数値、または存在しないAPI規約をAIが捏造していた場合、システムログに「WARN: ハルシネーションの可能性を検知」を記録し、その該当記述を自動でマスク、または検索結果をベースにした正しいファクトに自動置換して最終回答を作成します。
