# AI Workspace 「自律型マルチエージェント実行エンジン」設計書 (Autonomous AI Agent Engine Specification)

本書は、AI Workspaceに統合されるアドバンスドコア機能**「自律型マルチエージェント実行エンジン（AI Agent）」**のシステムアーキテクチャ、自律プロセス、データモデル、および画面設計を定義したものです。
ユーザーが「ゴール（達成したい抽象的な目的）」を入力すると、AIが実行計画を自律策定し、複数ステップの作業やエージェント間協調を自動実行しながら、詳細な実行ログ、リアルタイム進捗、そして最終的な完了レポートを出力します。

### 🚀 自律実行の対応ドメイン・テンプレート例
*   **市場調査**: 最新トレンド、主要プレイヤー、価格相場情報の収集・整理
*   **競合分析**: ポジショニングマップ（SWOT/3C）作成とバリュープロポジション設計
*   **AIO分析 (AI検索最適化)**: 各種対話型AI検索での露出度調査・推奨参照コンテキスト作成
*   **SEO分析**: キーワード難易度、競合コンテンツ構造、上位表示記事の構成案作成
*   **MEO分析 (地図検索最適化)**: Googleマップ検索順位、口コミ数・センチメント、競合店舗の露出状況分析
*   **提案書作成**: 説得力の高い構成骨子の策定とプレゼンテーション用スライド原稿の自動執筆
*   **アプリ設計**: 要件定義、ユーザーアクター設計、CRUD用データ構造スキーマ、画面レイアウト設計
*   **Webサイト設計**: ワイヤーフレーム構造、Tailwind CSSを用いたUIコードプロトタイプ自動生成

## 1. エージェント自律実行のライフサイクル (Agentic Execution Lifecycle)

自律型エージェントは、人間から与えられた曖昧なゴールに対し、以下の5つのループプロセス（OODAループおよびReActパターン）を繰り返して自律的に業務を完了させます。

```
[1. ユーザーのゴール入力] (例:「2026年夏の最新AI動向を調べ、競合差別化提案書を作って」)
      │
      ▼
[2. 実行計画生成 (Planner Agent)] ── (目標達成に必要なサブタスク、優先度、依存関係を自動決定)
      │
      ├─► [人間による承認・調整] (※オプショナル: 人間が介入して計画をブラッシュアップ可能)
      │
      ▼
[3. ステップ自律実行 (Execution Loop)]
      │  ├─► [タスク 1 実行]: Webクローリング・検索 (Perplexity / Gemini)
      │  ├─► [タスク 2 実行]: AIO/SEO露出度データ構造化 (Analyst Agent)
      │  └─► [タスク 3 実行]: 提案用マーケテキスト起草 (Writer Agent)
      │
      ▼
[4. 実行ログ ＆ 思考トレースの収集] ──── (AIの"思考(Thought)", "行動(Action)", "観察(Observation)"をリアルタイム追跡)
      │
      ▼
[5. 品質判定＆リトライ (QA Agent)] ─────► [不合格・再実行] ──► [ステップ3へ差し戻し]
      │ (パス)
      ▼
[6. 完了レポート作成 (Final Aggregator)] ─ (全タスク成果物を統合、結論・エグゼクティブサマリーとして出力)
```

## 2. データ構造・データベース設計 (Data Model Schema)

自律実行される各エージェントの動作、タスクプラン、リアルタイム実行ログ、および最終レポートを整合性を保って格納するためのスキーマ設計です。

```

                  | created_at                |
                  | completed_at              |
                  +-------------+-------------+
                                | 1
                                |
                                | 1..N
                  +-------------v-------------+
                  |      agent_plans          |
                  +---------------------------+
                  | id (PK)                   |
                  | goal_id (FK)              |
                  | task_title                |
                  | task_description          |
                  | execution_order           |
                  | status                    |
                  | assigned_agent            |
                  | input_context (JSON)      |
                  | output_context (JSON)     |
                  +-------------+-------------+
                                | 1
                                |
                                | 1..N
                  +-------------v-------------+
                  |     agent_execution_logs  |
                  +---------------------------+
                  | id (PK)                   |
                  | plan_id (FK)              |
                  | timestamp                 |
                  | log_level (INFO/DEBUG/ERR)|

                                | 1

                  | generated_assets (JSON)   |
                  | created_at                |
                  +---------------------------+
```

### テーブル定義詳細

#### ① `agent_goals` (ユーザーのゴール設定)
ユーザーがエージェントに指示した大目標と進捗状態。
*   `id`: UUID (Primary Key)
*   `user_id`: VARCHAR(100) (ユーザー識別子)
*   `title`: VARCHAR(255) (AIが自動要約した案件名。例: 「MEO/SEO 競合調査と最適化プラン」)
*   `original_goal`: TEXT (ユーザーの入力した指示原文)
*   `status`: VARCHAR(50) (ステータス: `PLANNING` (計画中), `RUNNING` (実行中), `WAITING_USER` (確認待ち), `SUCCESS` (完了), `FAILED` (エラー終了))
*   `created_at` / `completed_at`: TIMESTAMP

#### ② `agent_plans` (自律実行ステップ・計画)
ゴールを達成するために Planner Agent が動的に分解した実行タスク定義。
*   `id`: UUID (Primary Key)
*   `goal_id`: UUID (Foreign Key)
*   `task_title`: VARCHAR(255) (ステップの簡潔な名前。例: 「新宿エリアの競合MEO/SEOスコア抽出」)
*   `task_description`: TEXT (そのステップで実行する具体的なAI指示・コンテキスト)
*   `execution_order`: INT (実行順序。例: `1`, `2`, `3`...)
*   `status`: VARCHAR(50) (ステータス: `WAITING` (待機中), `ACTIVE` (実行中), `SUCCESS` (成功), `FAILED` (失敗), `RETRYING` (再試行中))
*   `assigned_agent`: VARCHAR(100) (割り当て専門エージェント: `ResearchAgent`, `AnalystAgent`, `WriterAgent`等)
*   `input_context` / `output_context`: JSON (前後のタスクから受け渡される構造化データ・中間アセット情報)

#### ③ `agent_execution_logs` (実行プロセス詳細ログ)
自律エージェントの「思考・意思決定・試行錯誤」のプロセスをユーザーに1秒単位で可視化するためのシステムログ。
*   `id`: UUID (Primary Key)
*   `plan_id`: UUID (Foreign Key)
*   `timestamp`: TIMESTAMP
*   `log_level`: VARCHAR(20) (区分: `INFO`, `DEBUG`, `WARN`, `ERROR`)
*   `phase`: VARCHAR(50) (ReAct状態。`THOUGHT` (思考), `ACTION` (行動・APIコール), `OBSERVATION` (観察・APIレスポンス評価), `SYSTEM` (システム通知))
*   `message`: TEXT (具体的なログテキスト、またはエージェントの心の声)

#### ④ `agent_reports` (最終完了レポート)
すべてのタスクが正常終了した後に生成される、ユーザーへ提出する統合成果物。
*   `id`: UUID (Primary Key)
*   `goal_id`: UUID (Foreign Key, Unique)
*   `summary`: TEXT (エグゼクティブサマリー。一目で分かる結論)
*   `markdown_body`: MEDIUMTEXT (詳細な分析、表、作成された設計書などのメインテキスト)
*   `sources`: JSON (参照したWebページのタイトル・URL・ファクトチェック証拠の一覧)
*   `generated_assets`: JSON (生成されたWebモックへのURL、PDF、CSV、ZIPファイル等の参照一覧)
*   `created_at`: TIMESTAMP

## 4. 対応するコア業務の実行計画テンプレート (Preset Agentic Workflows)

エージェントが自律駆動する際、対象業務に応じて Planner Agent が展開する実行計画の「プリセット定型構造」です。これにより、ユーザーはテンプレートを選ぶだけで、最適なマルチエージェントを自動召喚できます。

1.  **市場調査・競合分析パッケージ**:
    *   **ステップ1**: `ResearchAgent` がWebから該当産業の最新トレンド、プレイヤー一覧、価格相場をクローリング。
    *   **ステップ2**: `AnalystAgent` が各社の強み（USP）、弱み（Weakness）を抽出し、SWOT/3Cマトリクスを作成。
    *   **ステップ3**: `QA Agent` が、データソースのファクトチェック、URLの妥当性を検証。
    *   **ステップ4**: `WriterAgent` が、市場参入障壁と勝率の高い戦略をまとめた「競合調査スライド原稿」を出力。
2.  **MEO/SEO/AIO 総合露出最適化パッケージ**:
    *   **ステップ1**: `ResearchAgent` が、Google Map、主要AI検索、一般検索エンジンでの対象キーワード検索を実行。
    *   **ステップ2**: `AnalystAgent` が、露出を妨げている要因（キーワードの不足、店舗情報の未登録、レビュー不足）を特定。
    *   **ステップ3**: `WriterAgent` が、AI検索（AIO）に好まれる構造化引用テキスト、SEO記事の最適構成案、およびGoogleマイビジネスの改善用返信テンプレートを起草。
3.  **アプリ・Webサイト・管理画面の要件設計パッケージ**:
    *   **ステップ1**: `PlannerAgent` がユーザーの意図を汲み取り、アクター（利用ユーザー層）、CRUDに必要なデータ項目を一覧化。
    *   **ステップ2**: `AnalystAgent` が、画面構成（ダッシュボード、ログイン、CRUD、詳細ビュー）のワイヤーフレーム案とスキーマ設計をJSONで出力。
    *   **ステップ3**: `DeveloperAgent` が、設計に基づき、すぐにコピペ・稼働が可能なReactとTailwind CSSのコンポーネントコード、Supabaseのテーブル定義SQLを生成。
    *   **ステップ4**: `QA Agent` が、TypeScriptの型エラー、コンポーネントのインポート依存関係に問題がないかを自己検証・自動修復して、ビルド可能なパッケージとして出力。
