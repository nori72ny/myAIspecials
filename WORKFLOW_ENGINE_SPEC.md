# AI Workspace 「ワークフローエンジン」設計書 (Workflow Engine Specification)

本書は、AI Workspaceにおける「単発チャット」から「自律的な業務完結型タスク実行」へと進化させるためのコアコンポーネントである**「ワークフローエンジン」**の設計書です。
ユーザーから依頼された抽象的な業務目標を、AIが自動で実行可能なマイルストーンおよび個別タスクに分解し、人間と協調しながら最終成果物を完成させるプロセスを定式化します。

対象とするコア業務には、情報調査、各種分析（競合・AIO・MEO・SEO）、資料作成（提案書・レポート）、開発プロトタイプ（Webサイト・AIツール・管理画面）を含みます。

---

## 1. 業務フロー図 (Workflow Sequence Diagram)

以下は、ユーザーが業務を依頼してから、タスク分解、複数エージェントによる実行、レビュー、そして最終成果物が完了するまでの業務プロセスフローです。

```
+--------------------+        +---------------------+        +-----------------------+        +----------------------+
|  1. ユーザーの依頼 |  ───>  | 2. Planner Agent    |  ───>  | 3. タスク分解         |  ───>  | 4. AIルーター        |
|  (ゴール・要件入力) |        |    によるインテント解析 |        |    (並行/直列ステップ)  |        |    (最適LLM・エージェント) |
+--------------------+        +---------------------+        +-----------------------+        +----------------------+
                                                                                                         │
                                                                                                         ▼
+--------------------+        +---------------------+        +-----------------------+        +----------------------+
|  8. 最終成果物確定  |  <───  | 7. ユーザーレビュー |  <───  | 6. QA Agentによる検証  |  <───  | 5. エージェント実行  |
|  (保存・エクスポート)|  (承認) | (承認/修正/リトライ)|  (却下)|   (成果物/コード検証)  |        | (中間成果物の自動生成)|
+--------------------+        +---------------------+        +-----------------------+        +----------------------+
                                         │                                                               ▲
                                         └─────────── [修正・リテイク指示] ──────────────────────────────┘
```

### 各プロセスの詳細挙動
1.  **依頼受付 (Request ingestion)**:
    ユーザーがテキスト、参考ファイル、またはURLをインプットし、希望する業務（例：「競合調査して」「LP作って」など）を送信します。
2.  **タスク自動分解 (Task Decomposition)**:
    `Planner Agent` が動き、ゴール達成に必要なサブタスクリストを自動生成。各サブタスクの「依存関係（並行か直列か）」と「期待される中間アウトプットのフォーマット」を定義します。
    *   **対象業務ごとの自動分解例**:
        *   **情報調査**: キーワード選定 ➔ 複数ソース検索 ➔ ファクトチェック ➔ 要約レポート
        *   **競合分析**: 競合リストアップ ➔ 特徴・料金データ抽出 ➔ 3C/SWOTマッピング ➔ 差別化戦略起草
        *   **AIO分析 (AI検索最適化)**: 主要AI検索（Gemini, Perplexity等）での露出度調査 ➔ 推奨引用文脈分析 ➔ 対策プラン策定
        *   **MEO分析 (地図検索最適化)**: Googleマップ掲載状況調査 ➔ 競合店舗比較 ➔ キーワード/レビュー改善戦略
        *   **SEO分析**: ターゲットキーワード競合調査 ➔ 内部構成案作成 ➔ 対策記事構成案
        *   **提案書・レポート作成**: 構成骨子策定 ➔ 各章ドラフト執筆 ➔ 推敲・検証 ➔ スライド原稿化
        *   **Webサイト・AIツール・管理画面制作**: 要件定義・UI設計 ➔ Tailwind CSS/JSプロトタイプコード生成 ➔ バグ・表示チェック
3.  **状態管理 (State Initialization)**:
    ワークフローのステータスを `RUNNING` に設定し、データベース（SQLite/Postgres）にプロジェクトおよびタスクレコードを作成します。
4.  **AIエージェント自律実行 (Multi-Agent Execution)**:
    各タスクが順番（または並行）にトリガーされます。AIルーターはタスクの特性に応じて、最適なモデル（最新検索には `Perplexity`、高度な分析には `Claude`、推論には `Gemini` など）を呼び出し、成果物を順次生成・追跡します。
5.  **中間成果物保存 (Artifact Versioning)**:
    各ステップが完了するたびに、中間データ（生データ、テキスト草案、コード断片）をバージョン履歴（`Ver 1`, `Ver 2`...）付きでデータベースへ即時コミットします。
6.  **QA自動レビュー (Automated Quality Gate)**:
    `QA Agent` が、生成された中間成果物がユーザーの当初要件や法的・倫理的制約、セキュリティ基準を満たしているか、除外対象（医療・法律・投資助言等）に触れていないかを自動チェックします。
7.  **人間によるレビュー (Human-in-the-Loop)**:
    要確認ステップや大きな成果物の完了時に、ユーザーに対して「レビュー待ち」の通知を行います。ユーザーは中間プレビューを見ながら「承認して次のステップに進む」か「プロンプトで微修正・リテイクを依頼する」かを選択できます。
8.  **完了工程 (Deployment & Handover)**:
    全タスクが承認されると、プロジェクト全体のステータスが `COMPLETED` に移行。各中間成果物を結合し、Word、PDF、ZIPなどの指定フォーマットでエクスポート可能にします。

## 2. データベース設計 (Data Schema Design)

自律実行状況、中間成果物、エージェント間の協調ログ、およびバージョン情報を厳格に管理するためのテーブル定義です。

```
                  +-------------------+
                  |     projects      |
                  +-------------------+
                  | id (PK)           |
                  | name              |
                  | description       |
                  | folder_id (FK)    |
                  | status            |
                  | created_at        |
                  | updated_at        |
                  +---------+---------+
                            | 1
                            |
                            | 1..N
                  +---------v---------+
                  |    workflows      |
                  +-------------------+
                  | id (PK)           |
                  | project_id (FK)   |
                  | current_step      |
                  | total_steps       |
                  | status            |
                  | config (JSON)     |
                  +---------+---------+
                            | 1
                            |
                            | 1..N
                  +---------v---------+
                  |      tasks        |
                  +-------------------+
                  | id (PK)           |
                  | workflow_id (FK)  |
                  | title             |
                  | description       |
                  | step_number       |
                  | assigned_agent    |
                  | assigned_model    |
                  | status            |
                  | dependencies (ARR)|
                  | started_at        |
                  | completed_at      |
                  +---------+---------+
                            | 1
                            |
                            | 1..N
                  +---------v---------+
                  |    artifacts      |
                  +-------------------+
                  | id (PK)           |
                  | task_id (FK)      |
                  | version           |
                  | content_type      |
                  | raw_content       |
                  | structured_data   |
                  | file_path         |
                  | created_at        |
                  +-------------------+
```

### テーブル詳細

#### ① `projects` (プロジェクト管理)
ユーザーが開始した業務案件の全体単位を管理。
*   `id`: UUID (Primary Key)
*   `name`: VARCHAR(255) (プロジェクト名。例: 「新規SaaS 営業資料作成」)
*   `description`: TEXT (プロジェクトの詳細目標)
*   `folder_id`: VARCHAR(50) (「営業」「マーケティング」「開発」「個人」のフォルダID)
*   `status`: VARCHAR(50) (状態: `DRAFT`, `RUNNING`, `REVIEW_PENDING`, `COMPLETED`, `FAILED`)
*   `created_at` / `updated_at`: TIMESTAMP

#### ② `workflows` (ワークフロー管理)
プロジェクトに関連付けられたAIの実行プラン全体の進捗メタデータを管理。
*   `id`: UUID (Primary Key)
*   `project_id`: UUID (Foreign Key)
*   `current_step`: INT (現在実行中のステップ番号)
*   `total_steps`: INT (総ステップ数)
*   `status`: VARCHAR(50) (全体状態: `IDLE`, `PROCESSING`, `SUSPENDED`, `FINISHED`)
*   `config`: JSON (ルーティング戦略や全体プロンプト、優先モデル設定など)

#### ③ `tasks` (個別タスク・ステップ管理)
Plannerによって分解された、実行可能な個別タスクの定義と進行状況。
*   `id`: UUID (Primary Key)
*   `workflow_id`: UUID (Foreign Key)
*   `title`: VARCHAR(255) (タスクタイトル。例: 「MEO最適化用の競合マップ露出スコア算出」)
*   `description`: TEXT (エージェントへの詳細なインプット)
*   `step_number`: INT (実行順序)
*   `assigned_agent`: VARCHAR(100) (割り当てエージェント: `ResearchAgent`, `AnalystAgent`等)
*   `assigned_model`: VARCHAR(100) (割り当てモデル: `claude-3-5-sonnet`, `perplexity`等)
*   `status`: VARCHAR(50) (状態: `WAITING`, `PENDING`, `RUNNING`, `REVIEW`, `SUCCESS`, `FAILED`)
*   `dependencies`: TEXT/JSON-ARRAY (このタスクの実行前に完了が必要な他タスクのIDリスト)
*   `started_at` / `completed_at`: TIMESTAMP

#### ④ `artifacts` (中間・最終成果物管理)
各タスクが生成したアウトプットのバージョン履歴。
*   `id`: UUID (Primary Key)
*   `task_id`: UUID (Foreign Key)
*   `version`: INT (バージョン番号。例: `1`, `2`, `3`)
*   `content_type`: VARCHAR(100) (成果物の種別: `MARKDOWN`, `HTML`, `JSON`, `SLIDE_DATA`, `MAP_DATA`, `FILE_PATH`)
*   `raw_content`: MEDIUMTEXT (生成されたテキストやコードの本体)
*   `structured_data`: JSON (スライド構成やチャート用データ、マップ比較メタデータ等)
*   `file_path`: VARCHAR(512) (エクスポートされた静的ファイルへの参照パス)
*   `created_at`: TIMESTAMP

## 4. 将来的な自動実行 ＆ 外部連携設計 (Event-Driven Automation)

ワークフローエンジンをさらに進化させ、ブラウザを閉じていても、また外部システムからのシグナルをトリガーに自律稼働するための拡張設計です。

### 1. イベントトリガー型自動実行 (Webhooks & Triggers)
*   **Gmail連携**: `client-support@example.com` に届いた問い合わせをトリガーに、ワークフローが自動起動。「1.競合状況確認」「2.社内FAQからのドラフト作成」「3.返信提案書PDF作成」を自律実行し、下書きに保存する。
*   **Slack / LINE連携**: 指定チャンネルでのコマンド `/ai-run <依頼内容>` によって、バックグラウンドでプロジェクトが立ち上がり、完了時にSlackに成果物（PDF/画像）を直接ファイル返信する。
*   **Google Drive / Notion同期**: 最終成果物が承認（`COMPLETED`）された瞬間、Notionの指定データベースに新ページとしてパブリッシュされ、Google Driveの「営業成果物」フォルダにPDFが自動アップロードされる。

### 2. 定期バッチ自動実行 (Cron Schedules)
*   **市場/MEO/SEOデイリーレポート自動生成**:
    毎朝6:00に「1.Perplexityでターゲットキーワードの地図・Web検索を実行」「2.Geminiで前日比の検索順位・マップ露出度変動分析」「3.Writerで朝会用サマリー作成」というワークフローを自動実行。人間がPCを開いたときには、ダッシュボードに「最新レポート（Ver 1）」が仕上がっている状態を作り出します。

### 3. 完全自律バックグラウンドワーカー
*   実行中のステータス（`RUNNING`）の各タスクをメッセージキュー（Redis / Cloud Pub/Subなど）に投入。
*   サーバーサイドのバックグラウンドワーカーがキューを監視し、非同期でモデルプロバイダーをポーリング・リトライ処理。
*   HMRやブラウザ接続切れに影響されず、大規模な「30ステップからなる長期分析タスク」なども完全に裏側で自律完結。
